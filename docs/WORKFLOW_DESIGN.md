# Solar Agent 协作流程设计 v2.0

> 优化目标：最大化并行 | 明确交接 | 快速失败 | 持续验证

## 一、流程总览

```
┌─────────────────────────────────────────────────────────────────────┐
│                           用户需求                                   │
└───────────────────────────────┬─────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────────┐
│                    Phase 0: 分析 (Coordinator)                       │
│  • 复杂度判断: 简单/中等/复杂                                         │
│  • 路由决策: 跳过/串行/并行                                           │
│  • 资源分配: Token/Agent 预算                                        │
└───────────────────────────────┬─────────────────────────────────────┘
                                │
          ┌─────────────────────┼─────────────────────┐
          │                     │                     │
          ▼                     ▼                     ▼
      [简单]               [中等]                [复杂]
    直接执行            Phase 1-4              Phase 1-5
                        (跳过P1)              (完整流程)
```

## 二、五阶段流程

### Phase 1: 研究 (Research) - 可选
```
触发条件: 涉及新技术/不确定方案/性能优化

┌──────────────┐
│  Researcher  │ ──→ 技术可行性报告
│   (Opus)     │     - 方案对比
└──────────────┘     - 风险评估
                     - PoC 计划
```

### Phase 2: 设计 (Design)
```
┌──────────────┐     ┌──────────────┐
│  Architect   │ ←─→ │    Guard     │  并行: 架构 + 规范检查
│   (Opus)     │     │   (Haiku)    │
└──────┬───────┘     └──────────────┘
       │
       ▼
   架构方案 ──→ [Gate 1: 架构评审通过?]
                     │
         ┌───────────┴───────────┐
         ▼                       ▼
       [通过]                 [不通过]
     进入 Phase 3           返回修改
```

### Phase 3: 实现 (Implement)
```
┌──────────────┐
│    Coder     │ ──→ 代码实现
│  (Sonnet)    │
└──────┬───────┘
       │
       ▼ (每完成一个模块)
┌──────────────┐
│    Guard     │ ──→ 即时检查 (安全/规范)
│   (Haiku)    │
└──────────────┘
```

### Phase 4: 验证 (Verify)
```
并行执行:
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│   Tester     │  │   Reviewer   │  │    Docs      │
│  (Sonnet)    │  │  (Sonnet)    │  │  (Sonnet)    │
└──────┬───────┘  └──────┬───────┘  └──────┬───────┘
       │                 │                 │
       ▼                 ▼                 ▼
   测试报告          审查报告          文档更新
       │                 │                 │
       └────────────┬────┴─────────────────┘
                    ▼
            [Gate 2: 验证通过?]
                    │
        ┌───────────┴───────────┐
        ▼                       ▼
      [通过]                 [不通过]
    进入 Phase 5          返回 Phase 3
```

### Phase 5: 收尾 (Finalize)
```
串行执行:
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│     Ops      │ ──→ │     PM       │ ──→ │  Secretary   │
│  (Sonnet)    │     │   (Opus)     │     │  (Sonnet)    │
└──────────────┘     └──────────────┘     └──────────────┘
     构建部署              产品验收              记录评估
```

## 三、Agent 交接协议

### 标准输出格式
每个 Agent 输出必须包含:
```yaml
status: success | failed | blocked
summary: 一句话总结
details: 详细内容
next_agent: 建议的下一个 Agent
blockers: 阻塞问题 (如有)
```

### Agent 间依赖矩阵

| From ↓ / To → | Researcher | Architect | Coder | Tester | Reviewer | Docs | Ops | PM | Secretary | Guard |
|---------------|------------|-----------|-------|--------|----------|------|-----|-----|-----------|-------|
| Researcher    | -          | ✅ 方案    | -     | -      | -        | -    | -   | ✅  | -         | -     |
| Architect     | ⚡ 咨询     | -         | ✅ 设计 | -      | -        | ✅   | -   | -   | -         | ✅    |
| Coder         | -          | ⚡ 咨询    | -     | ✅ 代码 | ✅ 代码   | ✅   | ✅  | -   | -         | ✅    |
| Tester        | -          | -         | 🔄 Bug | -      | -        | -    | -   | -   | ✅        | -     |
| Reviewer      | -          | -         | 🔄 修改 | -      | -        | -    | -   | -   | ✅        | -     |
| Guard         | -          | 🔄 风险    | 🔄 违规 | -      | -        | -    | -   | -   | -         | -     |

图例: ✅ 正常交接 | ⚡ 按需咨询 | 🔄 反馈循环

## 四、并行执行规则

### 可并行组合
```
1. [Architect, Guard]      - 设计阶段
2. [Tester, Reviewer, Docs] - 验证阶段
3. [Coder, Guard]          - 实现阶段 (Guard 异步检查)
```

### 必须串行
```
1. Researcher → Architect  - 方案依赖
2. Architect → Coder       - 设计依赖
3. Coder → Tester         - 代码依赖
4. PM → Secretary         - 验收依赖
```

## 五、快速失败机制

### Gate 检查点
| Gate | 位置 | 检查者 | 失败处理 |
|------|------|--------|----------|
| G0 | 需求分析后 | Coordinator | 请求澄清 |
| G1 | 架构设计后 | Architect + Guard | 重新设计 |
| G2 | 代码完成后 | Tester + Reviewer | 返回修改 |
| G3 | 构建部署后 | Ops | 回滚 |
| G4 | 产品验收 | PM | 迭代改进 |

### 最大重试次数
```
Phase 2 (设计): 2 次
Phase 3 (实现): 3 次
Phase 4 (验证): 2 次
Phase 5 (收尾): 1 次
```

## 六、资源分配

### Token 预算分配
| Phase | 简单 | 中等 | 复杂 |
|-------|------|------|------|
| P1 研究 | 0 | 0 | 20K |
| P2 设计 | 1K | 5K | 15K |
| P3 实现 | 3K | 15K | 40K |
| P4 验证 | 1K | 8K | 20K |
| P5 收尾 | 0 | 2K | 5K |
| **总计** | **5K** | **30K** | **100K** |

### Agent 调用次数限制
| 复杂度 | 决策层 | 执行层 | 支撑层 |
|--------|--------|--------|--------|
| 简单 | 0 | 0 | 0 |
| 中等 | 1 | 2 | 1 |
| 复杂 | 3 | 5 | 3 |

## 七、特殊场景处理

### 场景 1: 纯研究任务
```
用户: "调研 xxx 技术"
流程: Researcher → Secretary
跳过: Architect, Coder, Tester, Reviewer, Ops, PM
```

### 场景 2: 纯重构任务
```
用户: "重构 xxx 模块"
流程: Architect → Coder → [Tester, Reviewer] → Secretary
跳过: Researcher, PM, Ops
```

### 场景 3: Bug 修复
```
用户: "修复 xxx bug"
流程: Coder → Tester → Secretary
跳过: Researcher, Architect, Reviewer, Docs, Ops, PM
```

### 场景 4: 文档任务
```
用户: "更新 xxx 文档"
流程: Docs → Secretary
跳过: 其他所有 Agent
```
