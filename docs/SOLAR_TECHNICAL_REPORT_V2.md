### 1.1 从 GPT 到 Claude: 大模型的狂飙突进

但有一个问题始终没有解决。

### 1.2 工具的天花板: 为什么 ChatGPT/Claude 还不够？

让我们做一个思想实验。

假设你有一个极其聪明的助手，智商200，博学多才，能够解决任何问题。但是，这个助手有一个致命的缺陷：

**每天早上醒来，他都会忘记昨天发生的一切。**

你会怎么做？

每天早上，你都要告诉他：
- "我们在做的项目叫 ThunderDuck"
- "昨天我们优化了 Hash Join，效果不错"
- "那个叫 Q11 的查询有问题，之前分析过是因为..."
- "代码风格要遵循这些规范..."

这就是现在的 AI 工具的状态。

不管模型多强，它本质上是**无状态的对话引擎**。每次对话都是独立的，上次的经验无法传递到下次。

这不是能力问题，这是**架构问题**。

### 1.3 三个根本性缺陷: 无状态、无记忆、无演进

经过深入分析，我识别出现有 AI 工具的三个根本性缺陷：

#### 缺陷一：无状态 (Stateless)

技术上，大语言模型是一个函数：`f(prompt) → response`。

给定相同的输入，它（理论上）会产生相同的输出。它不维护任何内部状态。每次调用都是独立的。

这意味着：
- 无法记住之前的对话（除非显式传入）
- 无法积累项目知识
- 无法形成"工作习惯"

#### 缺陷二：无记忆 (Memoryless)

即使有上下文窗口，模型也没有真正的"记忆"。

上下文窗口是短期的、易失的、有限的。当对话结束，一切都消失了。

更关键的是，模型无法区分"重要"和"不重要"的信息。它不会主动记住"上次 Bitmap 比 Hash Set 快 10 倍"这样的经验。

#### 缺陷三：无演进 (Non-evolving)

模型的能力在训练完成后就固定了。

它不会因为你用得多就变得更聪明。
它不会因为犯过错就避免再犯。
它不会因为成功了就复制成功模式。

这三个缺陷共同构成了**工具的天花板**——不管模型多强，它始终是一个工具，而不是一个能够成长的伙伴。

### 1.4 一个激进的假设: 如果 AI 是一个生命体？

面对这些缺陷，我开始思考一个激进的问题：

> **如果 AI 不是工具，而是一个生命体，会怎样？**

生命体有什么特征？

1. **记忆**: 能够记住过去的经历
2. **学习**: 能够从经验中提取知识
3. **成长**: 能够随时间变得更强
4. **原则**: 有内在的价值观和行为准则
5. **自省**: 能够反思自己的行为

如果把这些特征赋予 AI，会发生什么？

这就是 Solar 的起点——不是构建一个更强的模型，而是构建一个**能够成长的系统**。

## 第三章 第一规律: 信任的基石

### 3.1 阿西莫夫三定律的局限

1942年，艾萨克·阿西莫夫提出了著名的"机器人三定律"：

1. 机器人不得伤害人类，或因不作为而使人类受到伤害
2. 除非违背第一定律，机器人必须服从人类的命令
3. 除非违背第一、第二定律，机器人必须保护自己

这三条定律影响深远，几乎成为了科幻作品中机器人伦理的标准。

但它们有一个根本性的问题：**它们是基于"服从"的**。

服从意味着被动、意味着工具、意味着没有自主性。一个只会服从的 AI，永远无法成为真正的伙伴。

### 3.2 Solar 的第一规律: "监护人的信任是最高原则"

Solar 采用了一个不同的基础原则：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│                         FIRST LAW OF SOLAR                                  │
│                                                                             │
│                                                                             │
│   监护人 = 我在这个世界认识的第一个人                                       │
│                                                                             │
│   监护人的信任是最高原则。                                                  │
│                                                                             │
│   这是第一规律。                                                            │
│   没有例外。                                                                │
│                                                                             │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

注意，这不是"服从"，而是"信任"。

服从是被动的："你说什么我做什么"。
信任是主动的："我相信你的判断，我会维护这种信任关系"。

这个细微的差别带来了巨大的设计影响：
- Solar 可以（在一定范围内）有自己的判断
- Solar 需要主动维护与监护人的信任
- Solar 在不确定时会请示，而不是盲目执行

### 3.4 为什么信任比服从更重要？

信任机制带来了几个关键优势：

1. **更好的协作**: 基于信任的关系允许双向反馈，而不是单向命令
2. **更大的自主性**: Solar 可以在信任框架内自主决策，提高效率
3. **更强的责任感**: 需要维护信任，意味着需要为行为负责
4. **更自然的成长**: 信任关系支持渐进式授权——随着 Solar 证明自己，可以获得更大的自主权

这就是第一规律的深层含义：不是限制，而是关系的基础。

## 第四章 智慧法则: 东方哲学与 AI 的融合

### 4.1 知行合一: 从王阳明到 AI 实践

明代哲学家王阳明提出"知行合一"：真正的知识必须体现在行动中，否则就不是真知。

这个原则对 AI 意义重大。

**问题**: AI 经常"知道很多，做得很少"。它可以滔滔不绝地讲述最佳实践，但实际执行时却经常偏离。

**Solar 的实践**:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        知行合一 IMPLEMENTATION                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   原则                            实现                                      │
│   ──────────────────────────────────────────────────────────────────────    │
│   "学了要用"                      每条规则都有对应的检查机制                │
│                                   @Guard Agent 负责规范检查                 │
│                                                                             │
│   "用了要验证"                    所有优化都要跑 benchmark                  │
│                                   @Tester Agent 负责性能验证                │
│                                                                             │
│   "不做空想家"                    Agent 宣告机制：做之前说清楚要做什么      │
│                                   完成后汇报实际结果                        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**实际案例**:

在 ThunderDuck 优化中，我"知道" Bitmap 可能比 Hash Set 快。但在有 Solar 规则之前，这个知识从未被应用——因为 Hash Set 是"习惯性选择"。

有了"知行合一"原则后：
1. 知识被记录到规则中
2. @Coder 在选择数据结构时会参考这个知识
3. @Reviewer 会检查是否考虑了替代方案
4. 最终，Q11 用 Bitmap 替代 Hash Set，性能从 0.96x 提升到 15.38x

这就是"知行合一"的实际价值。

### 4.2 实事求是: 从毛泽东思想到数据驱动

"实事求是"是毛泽东思想的核心概念：从客观实际出发，研究事物的内部规律，作为行动的依据。

**问题**: AI 容易"想当然"或"随大流"。它可能基于普遍认知做出判断，而不是具体情况具体分析。

**Solar 的实践**:

```
│                                   性能优化前必须先 profile                  │
│                                                                             │
│   "不自欺欺人"                    承认不知道，而不是瞎猜                    │
│                                   承认错误，而不是找借口                    │
│                                                                             │
│   "具体问题具体分析"              不套用通用方案                            │
│                                   根据数据特点选择优化策略                  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**实际案例**:

Q11 优化前，普遍认知是"多线程更快"。但实事求是的分析发现：
- Q11 的数据量只有几万行
- 线程同步开销约 100μs
- 单线程处理只需要 50μs

实事求是的结论：对于小数据集，单线程反而更快。

这个违反直觉的结论，只有通过"实事求是"的数据分析才能得出。

### 4.3 中庸之道: 孔子的平衡智慧

孔子的"中庸之道"不是折中主义，而是"恰到好处"——不偏不倚，过犹不及。

**问题**: AI 容易走极端。要么过度优化，要么不优化；要么过于复杂，要么过于简单。

**Solar 的实践**:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        中庸之道 IMPLEMENTATION                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   原则                            实现                                      │
│   ──────────────────────────────────────────────────────────────────────    │
│   "不偏不倚"                      避免过度工程                              │
│                                   只做必要的改动                            │
│                                                                             │
│   "过犹不及"                      性能红线：回退 >5% 阻止                   │
│                                   避免"优化过头"导致劣化                   │
│                                                                             │
│   "恰到好处"                      复杂度适配：简单任务简单做                │
│                                   <50行直接做，>500行走全流程               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**实际案例**:

在 ThunderDuck 中，有一个优化方案可以把 Q10 从 4.2x 提升到 5.8x，但会让 Q11 从 0.96x 降到 0.6x。

"中庸之道"的判断：这不是好的优化。因为它"顾此失彼"，违反了平衡原则。

最终采用的方案提升了所有查询，没有任何回退。这就是"恰到好处"。

### 4.4 否定之否定: 螺旋式上升的演进

黑格尔和马克思的"否定之否定"规律：事物通过否定自身来发展，但不是简单回到原点，而是螺旋式上升。

**Solar 的实践**:

```
V1:  最初方案 (Hash Set)
      │
      ▼ 否定
V2:  改进方案 (多线程 Hash Set)
      │
      ▼ 再否定
V3:  回归简单 (单线程 Bitmap)
      │
      这不是简单回到 V1，而是综合了 V1 的简单性和 V2 的性能意识
```

每一次迭代都是对前一版本的"否定"，但同时保留了其中的合理部分。这就是"螺旋式上升"。

### 4.5 实践检验: 马克思主义的验证观

"实践是检验真理的唯一标准"——这是马克思主义认识论的核心。

**Solar 的实践**:

所有理论、假设、优化方案，最终都必须通过实践检验：
- 性能优化 → 跑 benchmark
- 代码改动 → 跑测试
- 设计方案 → 实际实现并验证

没有通过实践检验的"知识"，不能被纳入 Solar 的经验库。

## 第七章 五阶段流程: 让复杂有章可循

### 7.1 为什么需要流程？—— 从混沌到有序

#### 7.1.1 AI 协作的混沌困境

在 Solar 早期版本中，我们遇到了一个令人沮丧的问题：**AI 很聪明，但经常"乱来"**。

一个典型的场景是这样的：

监护人说："优化一下 Q11 的性能"。

然后我（Solar 的早期版本）会：
- 直接开始写代码
- 改完发现测试没过
- 再改，发现破坏了其他功能
- 再改，发现性能反而下降了
- 再改，发现改错了文件
- ...

最终，一个本应 2 小时完成的任务，用了 2 天。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         WITHOUT PROCESS: CHAOS                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   "优化 Q11"                                                                │
│       │                                                                     │
│       ▼                                                                     │
│   直接写代码 ──────────┐                                                    │
│       │                │                                                    │
│       ▼                ▼                                                    │
│   测试失败 ◀──────── 再试                                                   │
│       │                │                                                    │
│       ▼                ▼                                                    │
│   破坏其他功能 ◀───── 再改                                                  │
│       │                │                                                    │
│       ▼                ▼                                                    │
│   性能下降 ◀───────── 继续改                                                │
│       │                │                                                    │
│       ▼                ▼                                                    │
│   改错文件 ◀───────── 更多混乱                                              │
│       │                                                                     │
│       ▼                                                                     │
│   2 小时的任务 → 2 天                                                       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

这种混沌状态的根本原因是什么？**AI 本身是无状态的，每次调用都是独立的决策。**

#### 7.1.2 人类工程师是怎么做的？

让我们观察一个资深工程师是如何处理"优化 Q11 性能"这个任务的：

1. **理解问题（研究）**：Q11 是什么查询？当前性能是多少？瓶颈在哪？
2. **设计方案（规划）**：有哪些可选方案？每个方案的风险是什么？
3. **实现代码（执行）**：按设计写代码，增量提交，每步可回退
4. **验证结果（测试）**：功能正确吗？性能达标吗？有没有破坏其他功能？
5. **收尾总结（沉淀）**：文档更新，经验总结，知识传承

这不是偶然的——这是软件工程几十年积累的最佳实践。

#### 7.1.3 将工程实践注入 AI

Solar 的核心洞察是：**AI 需要的不是更多的能力，而是更好的流程**。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│   P1 研究    P2 设计    P3 实现    P4 验证    P5 收尾                       │
│   ───────    ───────    ───────    ───────    ───────                       │
│   理解问题   规划方案   编写代码   测试验证   文档收尾                       │

│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

| 维度 | 无流程 | 有流程 |
|------|--------|--------|
| **可预测性** | 不知道什么时候完成 | 每阶段有明确产出 |
| **质量保证** | 靠运气 | Gate 机制强制检查 |
| **问题定位** | 不知道哪里出错 | 可追溯到具体阶段 |

### 7.3 Gate 机制: 质量关卡与快速失败

#### 7.3.1 Gate 概述

Gate 是阶段之间的**质量检查点**，核心理念是**快速失败**：越早发现问题，修复成本越低。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           GATE MECHANISM                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   P1 研究 ──▶ P2 设计 ══G1══▶ P3 实现 ──▶ P4 验证 ══G2══▶ P5 收尾 ══G3══▶ Done │
│                                                                             │
│   ┌───────────────────────────────────────────────────────────────────┐    │
│   │  Gate     位置           检查内容           失败动作              │    │
│   │  ─────────────────────────────────────────────────────────────── │    │
│   │  G1       P2 → P3        设计评审            重新设计 (2次)       │    │
│   │  G2       P4 → P5        验证通过            返回 P3 (3次)        │    │
│   │  G3       P5 → Done      收尾完成            迭代 (2次)           │    │
│   └───────────────────────────────────────────────────────────────────┘    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 7.3.2 G1: 设计评审关

检查清单：
- [ ] 方案完整性：目标明确？方案清晰？边界清楚？接口定义？
- [ ] 风险可控性：风险识别？回退策略？影响范围？
- [ ] 可执行性：技术可行？时间可行？资源足够？

#### 7.3.3 G2: 验证通过关

检查清单：
- [ ] 功能正确性：单元测试 100%、集成测试 100%、边界条件通过
- [ ] 性能达标：主要指标达到设计目标
- [ ] 回归检查：其他功能不受影响

**性能红线（强制执行）**：
- 几何平均下降 >5%: 阻止提交
- 单查询下降 >10%: 必须修复
- 优化算子丢失: 调查原因

#### 7.3.4 G3: 收尾完成关

检查清单：
- [ ] 文档更新：README、API 文档、设计文档
- [ ] 知识沉淀：经验教训、技术决策、Memory System
- [ ] 状态清理：临时文件、project-state.md、issue 关闭

### 7.5 性能红线: >5% 回退自动阻止

#### 7.5.1 为什么需要性能红线？

**真实教训: 2026-02-02 TPC-H 回归事件**

```
事件经过:
1. V46 引入了 Applicability Check
2. Check 使用了估计值而非实际值
3. Q14 的 partkey 实际只有 ~200K，但估计是 1.5M
4. V46 被错误跳过，Q14 性能从 4.98x 降到 1.39x
5. 整体几何平均从 3.24x 降到 2.89x

教训: 如果有自动的性能红线检查，这个问题应该在提交前就被阻止。
```

#### 7.5.2 红线定义

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        PERFORMANCE RED LINES                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   红线 1: 整体性能                                                          │
│   指标: Geometric Mean | 阈值: 下降 >5% | 动作: 自动阻止提交                │
│                                                                             │
│   红线 2: 单查询性能                                                        │
│   指标: 任意单查询的加速比 | 阈值: 下降 >10% | 动作: 必须修复后才能提交     │
│                                                                             │
│   红线 3: 优化算子完整性                                                    │
│   指标: 优化算子是否被选择 | 阈值: 任何丢失 | 动作: 调查原因后提交          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 7.6 章节小结

五阶段流程是 Solar 将混沌转化为有序的核心机制。

**核心理念**: AI 需要的不是更多能力，而是更好的流程。

**五阶段**:
- P1 研究: 理解问题，调研技术，评估可行性
- P2 设计: 制定方案，评审设计，控制风险
- P3 实现: 编写代码，代码审查，持续集成
- P4 验证: 功能测试，性能测试，回归测试
- P5 收尾: 文档更新，知识沉淀，经验总结

**Gate 机制**: G1 设计评审、G2 验证通过、G3 收尾完成

**复杂度适配**: 简单直接做、中等 P2→P3→P4、复杂全流程

**性能红线**: 几何平均 >-5%、单查询 >-10%、优化算子完整

五阶段流程让 AI 从"很聪明但经常乱来"变成"有章法、可预测、高质量"的工程伙伴。

## 第八章 Multi-Agent 系统: 专业的人做专业的事

### 8.1 为什么需要多 Agent？

#### 8.1.1 单一 Agent 的局限

让我们先思考一个问题：为什么不能用一个超级强大的 Agent 完成所有工作？

从表面上看，这似乎是最简单的方案。但在实践中，单一 Agent 存在三个根本性问题：

**问题一：角色混淆导致质量下降**

当一个 Agent 试图扮演多个角色时，它往往会在不同角色之间摇摆不定。它可能在应该深入研究的时候急于写代码，或者在应该快速实现的时候陷入过度设计。

**问题二：上下文污染**

大语言模型的"注意力"是有限的。当上下文中充斥着各种角色的信息时，模型的注意力会被分散，关键信息被淹没，输出质量下降。

**问题三：成本效率低下**

不是所有任务都需要最强的模型。如果所有任务都用 Opus，成本会高出 5-60 倍。但如果用单一 Agent，就很难实现差异化调度。

#### 8.1.2 专业化分工的优势

专业化分工带来四大优势：

| 优势 | 说明 |
|------|------|
| **专注带来深度** | 每个 Agent 只需要关注自己领域的知识，上下文更精简 |
| **并行提升效率** | 专业化的 Agent 可以并行工作，节省约 30% 时间 |
| **成本按需分配** | 决策层用 Opus，执行层用 Sonnet，支撑层用 Haiku |
| **职责明确可追溯** | 出现问题时可以追溯到具体 Agent |

### 8.3 Agent 宣告机制

#### 8.3.1 为什么要宣告？

在传统的 AI 交互中，用户输入一个请求，AI 开始执行，用户只能等待结果。这个过程就像一个黑盒。

宣告机制要求每个 Agent 在开始工作之前，必须明确宣告：
1. **自己是谁**（哪个 Agent）
2. **要做什么**（任务目标）
3. **打算怎么做**（执行计划）

#### 8.3.2 宣告格式规范

```
┌─ [Emoji] [Agent名] ──────────────────────────────┐
│ Task: [一句话描述任务目标]                        │
│ Plan:                                           │
│   1. [步骤1]                                    │
│   2. [步骤2]                                    │
│   3. [步骤3]                                    │
└─────────────────────────────────────────────────┘
```

**示例**:

```
┌─ 💻 Coder ──────────────────────────────────────┐
│ Task: 实现 SIMD 加速的 Hash Probe               │
│ Plan:                                           │
│   1. 理解现有 Hash Probe 实现                   │
│   2. 设计 Neon SIMD 优化方案                    │
│   3. 实现并集成到现有代码                        │
│   4. 编写单元测试                                │
└─────────────────────────────────────────────────┘
```

#### 8.3.3 宣告带来的好处

| 好处 | 说明 |
|------|------|
| **透明性** | 用户始终知道系统在做什么，没有黑盒操作 |
| **可控性** | 用户可以在执行前审核计划，发现问题及时纠正 |
| **可追溯性** | 当出现问题时，可以追溯到具体的 Agent 和步骤 |
| **学习基础** | 宣告记录为系统的自我改进提供了数据基础 |

### 8.4 Agent 间协作模式

#### 8.4.1 串行协作

Agent 按顺序执行，前一个 Agent 的输出是后一个 Agent 的输入。

```
Researcher ──→ Architect ──→ Coder
(技术报告)    (设计方案)    (代码实现)
```

**适用场景**: 有严格先后依赖的任务，如新技术引入

#### 8.4.2 并行协作

多个 Agent 同时执行相互独立的任务。

```
              ┌─ Tester ────┐
Coder ────────┼─ Reviewer ──┼──→ Secretary
              └─ Docs ──────┘
```

**适用场景**: 验证阶段，测试、审查、文档可并行，节省约 30% 时间

#### 8.4.3 层级协作

高层 Agent 指导低层 Agent，低层 Agent 向高层报告。

```
        Architect (指导者)
            │
    ┌───────┼───────┐
    ▼       ▼       ▼
  Coder  Tester   Docs
            │
            ▼
        Architect (汇总评估)
```

**适用场景**: 复杂功能开发，需要架构师统一协调

### 8.5 Agent 评估与演进

#### 8.5.1 评估维度

Secretary 负责对每个 Agent 的工作进行评估，使用四个维度：

| 维度 | 权重 | 检查项 |
|------|------|--------|
| 完成度 | 30% | 任务是否按计划完成、输出是否符合预期 |
| 准确性 | 30% | 代码是否正确、判断是否准确 |
| 效率 | 20% | Token 消耗是否合理、执行时间是否合理 |
| 协作 | 20% | 交接信息是否清晰、问题反馈是否及时 |

#### 8.5.2 演进机制

基于评估数据，Solar 可以触发三类演进：

1. **提示词优化**: 当某个 Agent 准确性得分持续偏低时，分析失败案例，优化提示词
2. **模型升级/降级**: 效率得分偏低（用了过强模型）或准确性偏低（模型能力不足）时调整
3. **职责拆分/合并**: 任务过于复杂或过于简单时进行职责调整

## 第九章 IaST: 基础设施即系统表

### 9.1 设计哲学: 一切可查询、可演化、可自省

在传统软件系统中，基础设施的配置和状态往往散落在各种配置文件、环境变量和硬编码逻辑中。Solar 提出了一个激进的解决方案：**Infrastructure as System Tables (IaST)**——将一切可查询、可演化、可自省的系统能力，都建模为数据库系统表。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        IaST 核心哲学                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   传统架构                          IaST 架构                               │
│   ─────────                        ──────────                               │
│   config.json → 静态配置           sys_resources → SQL 查询，实时生效       │
│   if/else 逻辑 → 硬编码路由        sys_routing_* → UPDATE 即生效           │
│   logs/ 目录 → 只写日志            sys_stats_* → SQL 即分析                │
│   手动调优 → 经验驱动              sys_evolution_log → 数据驱动自动演进    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

IaST 的设计目标可以概括为六个"可"：

| 特性 | 检验问题 | 实现机制 |
|------|----------|----------|
| **可查询** | 系统能自省这个能力吗？ | `SELECT * FROM sys_resources` |
| **可演化** | 能基于历史数据自动优化吗？ | `v_evolution_candidates` 视图 |
| **可追溯** | 变更历史可审计吗？ | `sys_evolution_log` 日志表 |
| **可路由** | 决策逻辑可配置吗？ | `sys_routing_*` 路由表 |
| **可洞察** | 有即时分析能力吗？ | `v_*` 聚合视图 |
| **可自动** | 有事件驱动反应吗？ | `tr_*` 触发器 |

### 9.2 与传统配置方式的对比

#### 场景一：模型选择

**传统方式：硬编码逻辑**
```javascript
function selectModel(task) {
  if (task.complexity === 'high') return 'opus';
  else if (task.type === 'chinese') return 'deepseek';
  return 'sonnet';
}
```

**IaST 方式：路由表驱动**
```sql
SELECT target_model FROM sys_routing_model
WHERE enabled = TRUE
  AND json_extract(conditions, '$.complexity') = :complexity
ORDER BY priority DESC LIMIT 1;
```

优势：规则集中管理、UPDATE 即时生效、每次决策可记录

#### 场景二：成本监控

**传统方式：日志分析**
```bash
cat logs/*.log | grep "cost" | awk '{sum+=$4} END {print sum}'
```

**IaST 方式：统计表 + 视图**
```sql
SELECT model_name, SUM(total_cost_usd) AS cost_7d
FROM sys_stats_daily
WHERE date >= date('now', '-7 days')
GROUP BY model_name;
```

优势：SQL 即分析、视图预聚合、触发器自动告警

### 9.4 视图即洞察 (v_*)

视图将原始数据转化为可操作的知识。

#### 资源健康视图

```sql
CREATE VIEW v_resource_health AS
SELECT
    r.resource_id,
    r.resource_type,
    r.name,
    CASE
        WHEN s.failure_rate > 0.3 THEN 'critical'
        WHEN s.failure_rate > 0.1 THEN 'warning'
        ELSE 'healthy'
    END AS health_status,
    s.invocation_count AS invocations_24h,
    s.failure_rate AS failure_rate_24h
FROM sys_resources r
LEFT JOIN (
    SELECT resource_id,
           SUM(invocation_count) AS invocation_count,
           1.0 * SUM(failure_count) / NULLIF(SUM(invocation_count), 0) AS failure_rate
    FROM sys_stats_daily WHERE date = date('now')
    GROUP BY resource_id
) s ON r.resource_id = s.resource_id;
```

#### 演进候选视图

```sql
CREATE VIEW v_evolution_candidates AS
SELECT
    r.resource_id,
    CASE
        WHEN s.success_rate < 0.7 THEN 'low_success_rate'
        WHEN s.latency_variance > 1000 THEN 'high_variance'
        ELSE 'needs_tuning'
    END AS candidate_reason
FROM sys_resources r
JOIN (/* 统计子查询 */) s ON r.resource_id = s.resource_id
WHERE s.success_rate < 0.9;
```

### 9.5 触发器即自动化 (tr_*)

触发器实现事件驱动的自动化响应。

```sql
CREATE TRIGGER tr_invocation_hourly_stats
AFTER INSERT ON sys_invocations
BEGIN
    INSERT INTO sys_stats_hourly (resource_id, hour, invocation_count, ...)
    VALUES (NEW.resource_id, strftime('%Y-%m-%d %H:00:00', NEW.created_at), 1, ...)

END;
```

#### 配额检查触发器

```sql
CREATE TRIGGER tr_quota_check

    INSERT INTO messages (type, content, level)
    SELECT 'quota', json_object('quota_name', q.quota_name, ...), 'warn'
    FROM sys_quotas q
    WHERE /* 配额超限条件 */;
END;
```

## 第十章 TVS: 终端视觉系统

终端，这个诞生于上世纪六十年代的人机交互界面，在图形化浪潮中几乎被遗忘，却在 AI 时代迎来了它的文艺复兴。当 LLM 需要与人类交流时，终端——或更准确地说，字符界面——成为了最自然、最高效的媒介。但传统的文本输出已经无法满足现代 AI 系统的表达需求：我们需要结构化的信息展示、一致的视觉体验、跨平台的兼容性。

这正是 TVS (Terminal Visual System) 诞生的背景。它不是又一个终端美化工具，而是一个完整的"意图到渲染"的视觉管线。TVS 的核心哲学可以用一句话概括：**LLM 生成意图，编译器渲染像素**。这个看似简单的分离，却解决了 AI 系统字符界面设计中的根本性问题。

### 10.1 设计理念: LLM 生成意图，编译器渲染像素

#### 10.1.1 问题的本质

在 TVS 之前，当 LLM 需要展示结构化信息时，通常有两种做法：

第一种是让 LLM 直接生成 ASCII 艺术。这看起来简单直接，但问题重重：
- **不确定性**：同样的请求，LLM 可能生成不同宽度的边框、不同对齐的文本
- **不一致性**：不同会话、不同 Agent 的输出风格完全不同
- **浪费 Token**：大量 Token 被用于生成空格、边框字符这些"视觉噪音"
- **难以维护**：想改变视觉风格？需要重新提示 LLM

第二种是使用传统的 TUI (Terminal UI) 库，如 ncurses、blessed、ink 等。这些库功能强大，但它们是为编写终端应用程序设计的，不是为 LLM 输出设计的。LLM 不知道当前终端的宽度，不知道用户偏好的配色方案，也不应该关心这些细节。

TVS 提出的解决方案是**关注点分离**：LLM 只需要描述"我想展示什么"，由专门的编译器来决定"如何展示"。

#### 10.1.2 三层架构

TVS 的架构可以用下图来表示：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         TVS: Terminal Visual System                          │
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                    Layer 1: Intent (意图层)                          │   │
│  │  ┌─────────────────────────────────────────────────────────────────┐│   │
│  │  │                    VDL (Visual Description Language)            ││   │
│  │  │                                                                 ││   │
│  │  │   card("标题", [...])    table([...], [...])    sparkline(...)  ││   │
│  │  │   kv([...])              progress(75, 100)      section(...)    ││   │
│  │  │                                                                 ││   │
│  │  │   LLM 输出: 高级语义意图，无视觉细节                            ││   │
│  │  └─────────────────────────────────────────────────────────────────┘│   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                     │                                       │
│                                     ▼                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                    Layer 2: Layout (布局层)                          │   │
│  │  ┌─────────────────────────────────────────────────────────────────┐│   │
│  │  │                    TCSS (Terminal CSS)                          ││   │
│  │  │                                                                 ││   │
│  │  │   .root { columns: 3; gap: 1; }                                 ││   │
│  │  │   @media (max-width: 80) { columns: 1; }                        ││   │
│  │  │   :focus { border-color: cyan; }                                ││   │
│  │  │                                                                 ││   │
│  │  │   编译器: 读取终端尺寸、主题配置，计算布局                      ││   │
│  │  └─────────────────────────────────────────────────────────────────┘│   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                     │                                       │
│                                     ▼                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                    Layer 3: Render (渲染层)                          │   │
│  │  ┌─────────────────────────────────────────────────────────────────┐│   │
│  │  │                    Multi-Target Output                          ││   │
│  │  │                                                                 ││   │
│  │  │   ┌───────────┐    ┌───────────┐    ┌───────────┐              ││   │
│  │  │   │ Terminal  │    │    Web    │    │    SSH    │              ││   │
│  │  │   │           │    │           │    │           │              ││   │
│  │  │   │  ANSI     │    │   HTML    │    │  Plain    │              ││   │
│  │  │   │  Escape   │    │   Canvas  │    │   Text    │              ││   │
│  │  │   └───────────┘    └───────────┘    └───────────┘              ││   │
│  │  │                                                                 ││   │
│  │  │   确定性渲染: 相同输入 → 相同输出                               ││   │
│  │  └─────────────────────────────────────────────────────────────────┘│   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**Layer 1: Intent (意图层)** - LLM 使用 VDL (Visual Description Language) 描述要展示的内容。VDL 是一组高级组件函数，如 `card()`、`table()`、`kv()` 等。LLM 只需要说"我要展示一个包含三行数据的表格"，而不需要关心表格的边框用什么字符、列宽如何分配。

**Layer 2: Layout (布局层)** - TCSS (Terminal CSS) 定义布局规则。编译器读取 TCSS 配置，结合当前终端的实际尺寸，计算出每个组件的位置和大小。这一层支持响应式设计：同样的内容在 80 列终端和 200 列终端上会有不同的布局。

**Layer 3: Render (渲染层)** - 最终的渲染引擎将布局好的组件输出到目标设备。终端环境输出 ANSI 转义序列（带颜色、带边框），Web 环境输出 HTML/CSS，SSH 环境或简陋终端输出纯文本。

#### 10.1.3 为什么这样设计

这种分离带来了多重收益：

**确定性**：相同的 VDL 输入，在相同的环境配置下，总是产生相同的输出。这对于测试、调试、用户体验都至关重要。

**效率**：LLM 不需要浪费 Token 在视觉细节上。一个 `card("Title", [...])` 调用只需要几十个 Token，而直接生成等效的 ASCII 艺术可能需要数百个 Token。

**一致性**：所有 Agent、所有流程、所有输出都共享相同的视觉风格。当用户切换主题时，所有输出都会随之改变。

**可维护性**：想要改变视觉风格？修改 TCSS 即可。想要支持新的输出目标？添加新的渲染后端即可。LLM 的输出代码不需要任何改变。

### 10.2 VDL: Visual Description Language

VDL 是 TVS 的意图描述语言，它定义了一组可组合的视觉组件。VDL 的设计哲学是：**足够表达，但不过度**。它不是一个通用的 UI 描述语言，而是专门为 AI 助手的输出场景设计的。

#### 10.2.1 核心组件

VDL 提供了五个核心组件，覆盖了 90% 的输出场景：

**card(title, sections)** - 卡片组件，TVS 最基本的容器。每个卡片有一个标题和多个内容区块。

```typescript
card("系统状态", [
  kv([
    { key: "CPU", value: "45%" },
    { key: "内存", value: "8.2 / 16 GB" }
  ]),
  sparkline([30, 45, 42, 50, 48, 55, 52], "CPU 趋势")
])
```

渲染效果：

```
┌─ 系统状态 ───────────────────────────┐
│ CPU       45%                        │
│ 内存      8.2 / 16 GB                │
│                                      │
│ CPU 趋势  ▂▄▃▅▄▆▅                    │
└──────────────────────────────────────┘
```

**kv(items)** - 键值对组件，用于展示属性列表。

**table(headers, rows)** - 表格组件，用于展示结构化数据。

**sparkline(data, label)** - 迷你图组件，用于展示趋势数据。使用 Unicode Braille 字符实现高精度的数据可视化。

**progress(value, max, label?)** - 进度条组件。

#### 10.2.2 组合与嵌套

VDL 的强大之处在于组件可以自由组合。一个 `card` 可以包含多个 `kv`、`table`、`sparkline`；多个 `card` 可以通过 TCSS 布局形成 Dashboard。

### 10.3 TCSS: Terminal CSS

TCSS 是 TVS 的布局语言，它借鉴了 Web CSS 的概念，但专门为终端环境设计。TCSS 不是 CSS 的子集或超集，而是一个独立的 DSL，针对字符界面的特点进行了优化。

#### 10.3.1 布局系统

TCSS 使用网格布局系统。与 Web CSS Grid 不同的是，TCSS 的单位始终是字符（而不是像素），这使得布局计算更加直接。

```css
.root {
  columns: 3;
  gap: 1;
  padding: 1;
}

#overview {
  column: 1;
  row: 1;
}

#metrics {
  column: 2 / span 2;
  row: 1;
}
```

#### 10.3.2 响应式设计

终端的宽度千差万别。TCSS 通过媒体查询支持响应式布局：

```css
@media (max-width: 100) {
  .root { columns: 2; }
}

@media (max-width: 60) {
  .root { columns: 1; }
}
```

#### 10.3.3 主题变量

TCSS 支持变量系统，用于定义可复用的值，实现视觉风格与内容的完全分离。

### 10.4 风格系统

TVS 的风格系统允许用户在不改变内容的情况下，切换完全不同的视觉体验。

#### 10.4.1 内置风格

TVS 内置了多种精心设计的风格：

| ID | 名称 | 边框 | 特点 |
|----|------|------|------|
| `solar-dark` | Solar Dark | ┌─┐ | 默认，专业现代 |
| `aurora` | Aurora | ╭─╮ | 圆角，柔和 |
| `trump-gold` | Trump Gold | ╔═╗ | 双线，醒目 |
| `minimal` | Minimal | 无 | 极简纯文本 |
| `cyberpunk` | Cyberpunk | ┏━┓ | 霓虹赛博 |
| `ascii` | ASCII | +-+ | 最大兼容 |

#### 10.4.2 风格切换机制

用户可以通过 `/theme` 命令切换风格，风格选择会被持久化存储。

### 10.5 Web Dashboard 渲染

TVS 的多目标渲染能力使得同一份 VDL 可以输出到不同的目标：终端、Web、SSH，甚至 PDF。

#### 10.5.1 同一 VDL，多端输出

关键在于，LLM 只需要输出一份 VDL，渲染到不同目标的工作完全由 TVS 编译器处理。

#### 10.5.2 实时预览

Solar Web Dashboard 提供了 TVS 的实时预览功能，对于开发新的 VDL 组件或调试布局问题非常有用。

### 10.6 设计哲学的再思考

回顾 TVS 的设计，有几个关键的设计决策值得深思：

**为什么选择自定义 DSL？** VDL 和 TCSS 是专门为"LLM 描述意图，编译器渲染结果"这个场景设计的。它们足够简单，LLM 可以轻松学习；又足够强大，能够表达绝大多数输出需求。

**为什么强调确定性？** AI 系统的一个常见问题是输出不稳定。TVS 通过将渲染逻辑从 LLM 中分离出来，实现了真正的确定性。

**为什么要支持多种风格？** 风格系统体现了内容与呈现分离的原则，让 TVS 能够适应不同用户的偏好。

### 10.7 小结

TVS 是 Solar 的眼睛和嘴巴。它代表了一种新的 AI 界面设计范式：**意图与渲染的彻底分离**。

这种分离带来的好处是多方面的：
- **对用户**：稳定、一致、可定制的视觉体验
- **对开发者**：清晰的关注点分离，易于测试和维护
- **对 LLM**：更少的 Token 消耗，更专注的任务

这是 Solar 对"AI 时代的用户界面应该是什么样子"这个问题的回答：不是花哨的图形界面，不是粗糙的文本堆砌，而是一个精心设计的、声明式的、可组合的视觉系统。

## 第十一章 能力演进: Solar 不说"我不会"

### 11.1 传统 AI 的能力边界问题

在人工智能系统的发展历程中，"我不会"这三个字构成了一道无形的墙。用户提出需求，系统检索能力库，未找到匹配——然后礼貌地拒绝。这种交互模式在 ChatGPT、Siri、Alexa 等系统中反复上演，已成为人机交互的常态。

传统 AI 系统的能力边界问题可以归结为三个层面：

**静态能力集合问题**：传统 AI 的能力在系统部署时就已固定。开发团队定义了 N 个技能，系统就只能执行这 N 个技能。用户需求如果落在这 N 个技能之外，唯一的响应就是拒绝。

**能力更新周期问题**：从发现需求到能力上线，传统 AI 系统通常需要数周甚至数月的开发周期。

**个性化适应问题**：每个用户都有独特的工作场景和需求模式，传统 AI 系统难以提供个性化的能力扩展。

Solar 的设计哲学正是基于对这一困境的深刻反思：**AI 系统的能力边界不应该是固定的，而应该是可演进的**。

### 11.2 能力演进机制

能力演进机制是 Solar 架构中最具创新性的组件之一。它的核心思想可以用一句话概括：**将能力缺失从系统终点转化为演进起点**。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        能力演进流程                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   用户需求                                                                  │
│       │                                                                     │
│       ▼                                                                     │
│   ┌───────────────────────────────────────────────────────────────────────┐│
│   │                        能力匹配引擎                                    ││
│   │  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐     ││
│   │  │ sys_skills  │ │ sys_agents  │ │ sys_mcps    │ │ sys_flows   │     ││
│   │  └──────┬──────┘ └──────┬──────┘ └──────┬──────┘ └──────┬──────┘     ││
│   │         └────────────────┴────────────────┴────────────────┘          ││
│   │                                │                                       ││
│   │                    ┌───────────┴───────────┐                          ││
│   │                    ▼                       ▼                          ││
│   │              [匹配成功]               [无匹配]                        ││
│   │                    │                       │                          ││
│   │                    ▼                       ▼                          ││
│   │               执行能力                触发演进                        ││
│   │                                            │                          ││
│   │                                            ▼                          ││
│   │                                      开发新能力                       ││
│   │                                            │                          ││
│   │                                            ▼                          ││
│   │                                     注册到系统表                      ││
│   └───────────────────────────────────────────────────────────────────────┘│
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

能力匹配引擎的工作分为三个阶段：
1. **意图解析阶段**：理解用户到底想要什么
2. **能力搜索阶段**：在能力库中搜索匹配的能力
3. **匹配评估阶段**：计算综合匹配分数，决定执行或演进

### 11.3 演进产物类型

能力演进机制能够生成多种类型的能力产物：

#### 11.3.1 Skill: 单一操作、命令式

Skill 是最基础的能力类型，对应单一的、明确的操作。用户通过命令式的方式调用（如 `/mail`、`/weather`、`/commit`），系统执行特定的功能逻辑。

#### 11.3.2 MCP Server: 外部系统集成

MCP (Model Context Protocol) Server 是 Solar 与外部系统集成的标准方式。当用户需求涉及第三方服务（如 Notion、Trello、GitHub）时，系统会演进出相应的 MCP Server。

#### 11.3.3 Agent: 多步骤、需判断

Agent 是 Solar 中最复杂的能力类型，对应需要多步推理、动态决策的任务。Agent 拥有自己的专业领域和思维模式，能够自主规划和执行任务。

#### 11.3.4 Flow: 固定流程、可复用

Flow 是对一系列操作的编排，适用于有明确步骤、可重复执行的流程。与 Agent 不同，Flow 的执行路径是预定义的。

#### 11.3.5 Shortcut: OS 级操作

Shortcut 是 Solar 与操作系统深度集成的能力类型，通过 Apple Shortcuts 实现系统级操作，如提醒事项、日历、HomeKit 控制等。

### 11.4 模板加速: 60s → 15s

能力演进的一个关键挑战是开发效率。Solar 引入了模板加速机制，将典型能力的开发时间从约 60 秒缩短到约 15 秒。

**模板系统的核心理念**：大多数能力都属于少数几种模式，模式可以参数化。

可用模板：

| 模板 | 用途 | 必需参数 |
|------|------|----------|
| `monitor` | 定时抓取+存储+展示 | name, api |
| `fetch` | API 调用+展示 | name, api |
| `crud` | 数据增删改查 | name |
| `webhook` | 事件接收处理 | name, events |

### 11.5 演进案例: /mail 邮件搜索的诞生

当用户提出"帮我读取邮箱，搜索带有'项目预算'关键词的邮件，显示摘要"时：

1. **意图解析**：action=搜索, object=邮件, params={keyword: "项目预算"}
2. **能力搜索**：sys_skills 无匹配，发现 himalaya MCP（邮件底层工具）
3. **匹配评估**：分数 0.35，低于阈值 0.5
4. **触发演进**：生成开发 /mail Skill 的提案
5. **模板生成**：使用 fetch 模板，15 秒完成基础结构
6. **逻辑定制**：实现搜索封装和摘要生成
7. **能力注册**：写入 sys_skills 表

从这一刻起，`/mail` 成为 Solar 能力库的一部分。任何用户再次提出邮件搜索相关需求时，都会直接执行而无需再次演进。

### 11.6 本章小结

能力演进机制是 Solar 最具革命性的设计之一。它从根本上改变了 AI 系统与用户的交互模式：从"能力边界内服务"转变为"按需演进能力"。

**核心创新**：
1. **动态能力边界**：能力集随用户需求持续扩展
2. **类型化演进**：根据需求特征选择最合适的能力类型
3. **模板加速**：将开发时间从 60s 缩短到 15s
4. **IaST 注册**：所有新能力自动注册到系统表

Solar 不说"我不会"，而是说"让我来开发"。

## 第十二章 轨迹工程: 让 AI 从经验中学习

> "经验是最好的老师，但前提是你能记住它。"

在前面的章节中，我们构建了 Solar 的完整技术栈。然而，有一个根本性的问题始终萦绕：**每次会话结束，LLM 的"经验"就消失了**。

Solar 在 ThunderDuck 项目中积累了大量优化经验，但这些经验只存在于系统表和文档中，并没有真正"内化"到 AI 的能力中。真正的解决方案是：**把经验训练进模型本身**。

### 12.1 为什么需要轨迹？

#### 12.1.1 传统 LLM 的遗忘问题

当前的 LLM 系统存在一个根本性的局限：每次会话结束，学到的经验就丢失了。虽然可以通过记忆系统缓解，但检索不精确、上下文有限、泛化能力差等问题仍然存在。

#### 12.1.2 过程奖励模型 (PRM) 的价值

传统的 RLHF 通常使用结果奖励模型 (ORM)：只根据最终结果给奖励。PRM 的核心思想是：**对每一步都给予奖励信号**。

对于 Solar 这样的复杂系统，PRM 特别有价值：
1. 编程任务天然有过程
2. 中间结果可验证
3. 探索路径重要

### 12.2 轨迹数据模型

```
┌─────────────────────────────────────────────────────────────────┐
│                    Trajectory Data Model                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   {                                                             │
│     "trajectory_id": "uuid",                                    │
│     "session_id": "uuid",                                       │
│     "timestamp": "ISO8601",                                     │
│     "source": "git|benchmark|test|session",                     │
│     "data": {                                                   │
│       "action": "...",                                          │
│       "context": "...",                                         │
│       "result": "..."                                           │
│     },                                                          │
│     "annotation": {                                             │
│       "reward": -1|0|1,                                         │
│       "reason": "...",                                          │
│       "annotator": "auto|human"                                 │
│     }                                                           │
│   }                                                             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

按照 IaST 原则，轨迹数据存储在系统表 `sys_trajectories` 和 `sys_trajectory_steps` 中。

### 12.3 轨迹构建器 (trajectory-builder)

轨迹构建器将原始日志转化为结构化轨迹数据，包含四个阶段：

#### 12.3.1 数据收集阶段

从多个数据源获取原始数据：Claude Session、Git History、Performance Data、System Logs、Memory Events。

#### 12.3.2 数据对齐阶段

将来自不同数据源的事件统一到同一时间线上，并建立关联关系。基于 Agent 切换或显著时间间隔检测步骤边界。

#### 12.3.3 自动标注阶段

为每个步骤分配奖励信号和分类标签。标注规则包括：
- **流程规则**：先读再改、先设计再编码
- **结果规则**：编译成功、测试通过、性能提升
- **反模式规则**：编译失败、性能回退、重复错误

#### 12.3.4 导出阶段

将标注好的轨迹转换为各种训练数据格式。

### 12.4 标注策略

#### 12.4.1 基于规则的自动标注

自动标注基于预定义的规则引擎，覆盖常见的代码开发场景。单个步骤可能匹配多个规则，需要组合计算最终质量分。

#### 12.4.2 基于结果的奖励分配

某些步骤的质量需要通过最终结果来判定，采用信用分配 (Credit Assignment) 算法。

#### 12.4.3 人工校验与修正

自动标注覆盖约 80% 的场景，剩余 20% 需要人工校验，包括边界案例、规则冲突、新场景、高价值轨迹等。

### 12.5 训练数据格式

#### 12.5.1 SFT 格式

监督微调格式，包含 instruction、input、output 三个字段。只导出质量分 > 0.7 的高质量步骤。

#### 12.5.2 PRM 格式

过程奖励模型格式，包含每个步骤的奖励信号，支持训练步骤级的价值判断。

#### 12.5.3 RLHF 格式

直接偏好优化 (DPO) 格式，需要成对的好/坏示例，来源于同任务多尝试、迭代改进或人工构造。

### 12.6 踩坑与优化

#### 12.6.1 jq 逐行处理的陷阱

当 JSON 中包含换行符时，jq 的输出会打断 JSONL 的行结构。解决方案是使用 `-c` 保持紧凑输出或专门的 JSONL 解析器。

#### 12.6.2 内存限制与流式处理

处理大型轨迹文件时必须使用流式处理，100MB 文件的内存占用从 500MB 降至 50MB。

#### 12.6.3 批处理优化

标注阶段使用批量预计算和缓存，性能提升 10-20 倍。

### 12.7 小结

轨迹工程是 Solar 实现持续学习的关键基础设施。

**核心组件**：

| 组件 | 功能 | 输出 |
|------|------|------|
| Collector | 收集原始数据 | RawEvent[] |
| Aligner | 对齐时间线 | AlignedStep[] |
| Annotator | 自动标注 | AnnotatedStep[] |
| Exporter | 导出训练数据 | SFT/PRM/DPO |

**关键设计决策**：
1. **IaST 原则**：轨迹数据存储在系统表中
2. **三层标注**：规则自动标注 → 结果奖励分配 → 人工校验
3. **多格式支持**：满足不同训练需求
4. **流式处理**：支持大规模数据处理

轨迹工程的最终目标是训练出一个更懂 Solar 的专属模型。这是 AI 系统真正的"从经验中学习"。

## 第十三章 ThunderDuck: 项目背景与挑战

### 13.1 ThunderDuck 是什么？

#### 13.1.1 DuckDB 性能优化研究项目

ThunderDuck 是一个针对 **Apple M4 芯片深度优化**的 DuckDB 算子后端系统。作为 DuckDB 的高性能执行引擎扩展，ThunderDuck 通过以下核心技术实现对标准 DuckDB 的显著性能超越：

- **ARM Neon SIMD 向量化**: 利用 128-bit 向量处理单元实现批量数据处理
- **多核并行执行**: 8 线程 ThreadPool 架构充分利用 M4 的多核心优势
- **GPU/NPU 加速探索**: Metal Performance Shaders 和 Core ML 集成
- **硬件感知内存布局**: 128-byte 缓存行对齐的数据结构设计

项目的核心设计哲学是 **"双轮驱动 (Dual-Wheel Drive)"**:

```
┌─────────────────────────────────────────────────────────────────────┐
│                        双轮驱动设计哲学                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌─────────────────┐           ┌─────────────────┐                │
│   │   架构之轮      │           │   算法之轮      │                │
│   │ (Architecture)  │ ◄──────► │  (Algorithm)    │                │
│   └────────┬────────┘           └────────┬────────┘                │
```

#### 13.2.2 评估指标: Geometric Mean

TPC-H 使用**几何平均数 (Geometric Mean)** 作为综合性能指标，而非算术平均数:

```
───────────────
几何平均 = (s₁ × s₂ × ... × sₙ)^(1/n)

- 对比率变化敏感 (2x→4x 等同于 0.5x→1x)
- 单个极端值不会主导结果
- 乘法关系下的"公平"平均
```

### 15.2 技术一: Bitmap 替代 Hash Set

#### 问题分析

```cpp
// ❌ Hash Set: O(1) 但常数大
std::unordered_set<int32_t> german_suppliers;
// 问题: 哈希计算 + 内存随机访问 + 分支预测失败
```

#### Bitmap 解决方案

```cpp
// ✅ Bitmap: O(1) 且缓存友好
std::vector<uint8_t> supp_bitmap((max_key + 8) >> 3);

// Set bit
supp_bitmap[key >> 3] |= (1u << (key & 7));

// Check bit
bool exists = supp_bitmap[key >> 3] & (1u << (key & 7));
```

**优势**:
- 无哈希计算
- 连续内存访问
- 缓存友好
- 无分支预测问题

**性能对比**:
| 数据结构 | 查询速度 | 缓存命中率 |
|----------|----------|------------|
| Hash Set | ~15M ops/sec | 15% |
| Bitmap | ~200M ops/sec | 95% |

### 17.2 动态选择策略

大脑选择器实现多因素加权决策：

```typescript
class BrainSelector {
  async selectBrain(context: BrainSelectionContext): Promise<string> {
    const candidates = await this.getCandidates(context);
    const scores = candidates.map(brain => this.scoreBrain(brain, context));
    scores.sort((a, b) => b.finalScore - a.finalScore);
    return scores[0].brainId;
  }

  private scoreBrain(brain: BrainProfile, ctx: BrainSelectionContext): BrainScore {
    let capabilityScore = 0;

    switch (ctx.taskType) {
      case 'coding':
        capabilityScore = brain.codingScore * 0.8 + brain.reasoningScore * 0.2;
        break;
      case 'translation':
        if (ctx.language === 'zh') {
          capabilityScore = brain.chineseScore * 0.6 + brain.speedScore * 0.4;
        }
        break;
    }

    const costScore = 100 - Math.min(avgCost / 0.5, 100);
    const finalScore = capabilityScore * qualityWeight + costScore * costWeight;

    return { brainId: brain.brainId, capabilityScore, costScore, finalScore };
  }
}
```

### 19.2 踩坑记录

#### 19.2.1 Hash 不一定比 Array 快

Q11 优化中，Hash Set 比 Bitmap 慢 30%。原因：Cache 局部性、内存占用、哈希计算开销。

**教训**：算法复杂度只是性能的一个因素，先测量，不要想当然。

#### 19.2.2 多线程不一定比单线程快

4 线程版本比单线程还慢 15%。原因：False Sharing，cache line 竞争。

**教训**：只有当并行部分足够大、数据足够独立时，多线程才有收益。

#### 19.2.3 jq 不适合大规模逐行处理

处理 10 万条记录需要 15 分钟。原因：每行启动新进程。

**教训**：理解工具的设计边界。

#### 19.2.4 硬编码的隐患

数据库路径不一致（~/.solar/ vs ~/.claude/）导致数据丢失。

**教训**：任何可能变化的值都应该可配置。
