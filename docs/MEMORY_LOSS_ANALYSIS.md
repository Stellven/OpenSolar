## 根本原因分析

### 原因1: 多窗口并行无统一持久化

```
┌─────────────────────────────────────────────────────────────┐
│           多窗口并行工作的数据流                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  窗口1 (小区神经网)          窗口2 (Capsule)                │
│    │                            │                          │
│    ▼                            ▼                          │
│  .jsonl                      .jsonl + docs/*.md ✓          │
│  (142MB)                     (保存成功)                     │
│    │                                                        │
│    └─ ❌ 会话关闭后未持久化                                 │
│                                                             │
│  窗口3 (反思)                窗口4 (Ontology)               │
│    │                            │                          │
│    ▼                            ▼                          │
│  .jsonl                      .jsonl + core/ontology/ ✓     │
│  (可能存在)                   (代码已保存)                  │
│    │                                                        │
│    └─ ❌ 未触发 @Secretary 持久化                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**问题**:
- 会话内容只存在 `.jsonl` 文件中（临时）
- 如果没有显式写入文档，内容会随会话关闭而"丢失"
- `.jsonl` 文件难以检索和恢复

### 原因2: @Secretary 未被自动触发

**现有机制**:
- 用户说"好"/"确认"/"OK" → 触发 @Secretary 持久化
- 但多窗口并行时，可能某些窗口没有明确的"确认"指令

**缺失**:
- 无自动会话检查点机制
- 无周期性持久化（如每30分钟）
- 无窗口关闭前的强制保存

### 原因3: 无会话间数据同步

```
窗口1的讨论内容 → 只在窗口1的 .jsonl 中
窗口2无法访问窗口1的内容
窗口间无共享记忆存储
```

### 原因4: 语义记忆未充分利用

**发现**:
```sql
-- 检查语义记忆表
SELECT COUNT(*) FROM evo_memory_semantic WHERE namespace LIKE '%neural%';
-- 结果: 0

SELECT COUNT(*) FROM evo_memory_semantic WHERE namespace LIKE '%distributed%';
-- 结果: 0
```

**问题**: 讨论内容未被写入 `evo_memory_semantic` 表

## 改进方案

### 短期方案（立即实施）

#### 1. 自动会话检查点

**实现**: Hook + 定时任务

```typescript
// hooks/session-checkpoint.sh
// 每30分钟自动触发
const AUTO_CHECKPOINT_INTERVAL = 30 * 60 * 1000;

setInterval(() => {
  if (hasActiveSession()) {
    triggerSecretary("自动检查点");
  }
}, AUTO_CHECKPOINT_INTERVAL);
```

#### 2. 窗口关闭前保存

**实现**: SessionEnd Hook

```bash
# hooks/session-end.sh
#!/bin/bash
# 会话结束前强制保存

bun ~/.claude/agents/secretary.ts save-session
```

#### 3. 关键讨论实时写入语义记忆

**触发词**: "这很重要"/"记住这个"/"保存到记忆"

```typescript
if (userMessage.includes("记住") || userMessage.includes("重要")) {
  await saveToSemanticMemory({
    namespace: "conversations",
    content: extractKeyPoints(conversation),
    timestamp: now()
  });
}
```

## 经验教训

### 对监护人的建议

1. **多窗口工作时**:
   - 定期说"保存"/"记住这个"
   - 每30分钟手动触发 `/save`
   - 重要讨论结束后确认"好的"触发持久化

2. **窗口关闭前**:
   - 检查是否有未保存的重要内容
   - 执行 `/save` 或明确说"保存当前进度"

3. **长期项目**:
   - 使用 `.solar/project-state.md` 追踪状态
   - 定期 `git commit` 重要文档
   - 利用 @Secretary 汇总会议纪要

### 对Solar的改进

1. **不依赖用户主动保存**
   - 系统应主动检测重要内容
   - 自动持久化机制

2. **多窗口场景优化**
   - 窗口间数据同步
   - 统一的会话管理

3. **记忆系统增强**
   - 语义记忆自动填充
   - 从会话中提取结构化知识
