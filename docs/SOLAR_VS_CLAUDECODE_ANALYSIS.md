## 第一章: 系统定位 — 操作系统 vs 执行原语

### 1.1 Claude Code 的定位

Claude Code 是 Anthropic 提供的 **执行原语层**:

- 提供基础工具调用 (Read/Write/Bash/Edit)
- 提供 LLM 推理 (claude-sonnet-4-5 / opus-4-6)
- 每次会话独立, 无持久状态
- 无内置路由, 无个性化, 无知识积累

这相当于裸机 + BIOS: 强大但无状态, 无个性, 无历史。

### 1.2 Solar 的定位

Solar 是运行在 Claude Code 之上的 **AI Native OS**:

```
┌──────────────────────────────────────────────────────────────┐
│                    SOLAR STACK                                │
├──────────────────────────────────────────────────────────────┤
│  L4: 应用层    │ 94 Skills + 多智能体工作流                  │
│  L3: 服务层    │ Brain Router + Cortex + 记忆系统            │
│  L2: 治理层    │ 铁律系统 + CLAUDE.md + 75 Hooks             │
│  L1: 状态层    │ 344 DB Tables + STATE.md + 持久化           │
│  L0: 原语层    │ Claude Code (工具调用 + LLM)                │
└──────────────────────────────────────────────────────────────┘
```

### 1.3 核心差异矩阵

| 维度 | Claude Code | Solar |
|------|------------|-------|
| 状态持久化 | 无 | STATE.md + 344 张表 |
| 个性化 | 无 | 铁律 + D&D KNOBS |
| 知识积累 | 无 | 90 artifacts, 482 sources |
| 模型选择 | 固定 claude | 12 模型动态路由 |
| 错误学习 | 无 | 铁律 = 失败编译产物 |
| 上下文注入 | 无 | 每消息 12 hooks |
| 多智能体 | 无 | Solar Farm 架构 |
| 成本优化 | 固定 $0.015/1K | $0.0001 - $0.015 动态 |

```
│  rules/01-three-core-laws.md ~3,000 bytes   三大铁律详解        │
│  rules/state-persistence.md  ~5,000 bytes   状态持久化          │
│  rules/solar-farm.md         ~3,000 bytes   阳光牧场规则        │
│  rules/cortex-first.md       ~3,000 bytes   Cortex 优先         │
│  rules/multi-expert-analysis ~2,000 bytes   多专家会审          │
│  rules/call-niuma-with-*.md  ~3,000 bytes   人格注入规则        │
│  其他 rules/*.md              ~6,000 bytes   领域规则            │
├─────────────────────────────────────────────────────────────────┤
│  总计: ~30,000 bytes (每次会话固定注入)                         │
└─────────────────────────────────────────────────────────────────┘
```

这相当于每次会话昊哥都在对话中贴了 30KB 的训练材料。

### 2.2 Hooks 系统 (75 个)

```
┌───────────────────────────────────────────────────────────────┐
│                     HOOK 触发统计                              │
├───────────────────────────────────────────────────────────────┤
│  触发时机              数量    代表 Hook                        │
│  ─────────────────────────────────────────────────────────   │
│  UserPromptSubmit      12     上下文预加载、Cortex 提醒         │
│  SessionStart           4     STATE.md 加载、环境检查          │
│  SessionEnd             7     状态自动保存、git checkpoint      │
│  PreToolUse            ~25    经验提醒、安全检查               │
│  PostToolUse           ~27    记忆记录、绩效追踪               │
├───────────────────────────────────────────────────────────────┤
│  每条消息触发 12 个 Hook = 用户看到消息前，系统已自动准备      │
└───────────────────────────────────────────────────────────────┘
```

每次用户发消息, 以下 12 个 UserPromptSubmit hooks 已经触发:
- 数据资产账本检查
- Cortex 相关查询提示
- Tantivy 搜索引擎状态
- 铁律关键词检测
- 任务分配决策辅助
- 上下文预加载 (已有能力列表)
- ...

### 2.3 铁律系统 (Iron Laws)

铁律不是指南, 是**失败事件编译成的防错代码**:

```
失败事件                    →    编译为铁律
─────────────────────────────────────────────────────────────────
人格丢失 (2026-02-07)       →    personality-retention.md
没查 Cortex 重复造轮子       →    cortex-first.md
调牛马不带人格 (2026-02-08) →    call-niuma-with-personality.md
状态跨会话丢失 (2026-02-09) →    state-persistence.md
数据计算没持久化             →    guardian-data-first.md
分析只用单专家               →    multi-expert-analysis.md
直接执行没规划               →    plan-before-execute.md
...                          →    ...
```

每条铁律都带有 `来源:` 字段, 可追溯到具体失败时间。这是 Solar 区别于 Claude Code 的核心竞争力之一: **从错误中学习并固化**。

### 5.2 STATE.md 状态持久化

```markdown
# STATE.md 五段式结构

## Mission
一句话目标 (可验收)

## Constraints
不可破坏的约束条件

## Current Plan (Top-5)
具体执行计划

## Decisions (Why)
[日期] 选择 X 而非 Y: 原因...

## Progress
- Done: ...
- In-Progress: ...
- Blocked: ...

## Next Actions (Exact)
- [ ] 具体命令
- [ ] 具体文件修改
```

每次 compact 前必须更新, 新会话第一步必须读取。
解决了 Claude Code "每次会话重置" 的根本问题。

### 6.2 自我变更日志 (sys_self_changelog)

所有对 Solar 自身的修改都必须记录:
```sql
INSERT INTO sys_self_changelog
(change_type, target_file, old_value, new_value, reason, session_id)
VALUES ('rule', 'rules/xxx.md', '...', '...', '为什么改', 'abc12345');
```

Solar 可以追踪自己的演化轨迹。Claude Code 无此能力。

### 6.3 专家团队治理

涉及 Solar 自身的修改必须经过专家团队审核:
- 稳健派 (gemini-2.5-pro): 技术架构审查
- 探索派 (gemini-3-pro): 权衡决策
- 审判官 (deepseek-r1): 逻辑规则审核

单兵作战被明确禁止。

### 6.4 多专家会审机制

分析任务必须并行调用 2-3 个专家:
```
❌ 错误: 调用单一专家分析
✅ 正确: 审判官 + 探索派 + 创想家 并行分析, 综合结论
```

1,015 条 collab_performance 记录为路由决策提供数据支撑。
