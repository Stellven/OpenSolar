## 第五章 Day 5-6 (1/31-2/1): 突破 — 极致优化与能力演进

### 引言：临界点前夜

1月30日深夜，监护人盯着屏幕上的数字：3.24x。

这是 ThunderDuck 在 TPC-H 基准测试上的几何平均加速比。比四天前的 2.77x 提升了 17%，已经相当不错。但有一个数字依然刺眼：Q11 的 0.96x。

"三天了，" 监护人说，"该解决 Q11 了。"

### 5.1 V93 算子的诞生：Q11 从 0.96x 到 15.38x

#### 5.1.1 问题分析

@Researcher 花了整整一个上午进行性能分析。使用 perf 和 VTune 对 V58 算子进行 profiling 后，发现了四个关键瓶颈：

```
┌─────────────────────────────────────────────────────────────────────┐
│                Q11 PERFORMANCE BOTTLENECKS                           │
├─────────────────────────────────────────────────────────────────────┤
│  Bottleneck              Impact    Root Cause                       │
│  ─────────────────────────────────────────────────────────────────  │
│  1. Hash Set Lookup      35%       Cache miss rate > 40%            │
│  2. Multi-threading      25%       Sync overhead > computation      │
│  3. Multiple Passes      22%       3 passes over same data          │
│  4. Hash Aggregation     18%       Hash computation overhead        │
└─────────────────────────────────────────────────────────────────────┘
```

**关键洞察：** Q11 的 partkey 是 1 到 200,000 的连续整数，数据量只有约 800K 行。我们一直用"大数据"的思维去优化一个"中等数据"的查询。

#### 5.1.2 解决方案

分析结果汇报给 @Architect 后，我们进行了颠覆性的设计：

1. **Bitmap 替代 Hash Set** — O(1) 且缓存友好，直接寻址无分支
2. **Dense Array 替代 Hash Map** — sums[partkey] += value，零哈希开销
3. **单线程执行** — 800K 行数据，避免所有同步开销
4. **单遍扫描** — 合并 threshold 计算和聚合

#### 5.1.3 验证结果

@Tester 运行完整基准测试，结果震撼：

```
┌─────────────────────────────────────────────────────────────────────┐
│                Q11 BENCHMARK RESULTS                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Version          DuckDB (ms)    ThunderDuck (ms)    Speedup        │
│  ─────────────────────────────────────────────────────────────────  │
│  V58 (Before)        384              400              0.96x        │
│  V93 (After)         384               25             15.38x        │
│  ─────────────────────────────────────────────────────────────────  │
│                                                                     │
│  Improvement: 400ms → 25ms = 16x faster                             │
│  vs DuckDB:   0.96x → 15.38x = +1502%                               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**从 0.96x 到 15.38x，提升幅度达到 1502%。** 这不是渐进式优化，这是范式转变。

- 遗漏边界条件
- 忽略性能影响
- 风格不一致

引入宣告机制后：

```

```

这种强制性的"先计划后执行"模式，将一次通过率从 60% 提升到 85%。

## 第十一章 技术深度：Q11 优化的完整复盘

### 11.1 问题发现

Q11 加速比只有 0.96x，比 DuckDB 还慢 4%。

### 11.2 根因分析 (5 Whys)

```
Why 1: Q11 慢         → Hash Set 查找慢
Why 2: 查找慢          → 缓存不友好 + 分支预测失败
Why 3: 缓存不友好      → Hash Set 内存布局不连续
Why 4: 用 Hash Set     → 习惯性选择，未分析数据特点
Why 5: 未分析数据      → 缺乏系统性的性能分析流程

根因: 工作流程缺陷
```

### 11.3 解决方案

**关键洞察:** partkey 是 1-200K 的连续整数，数据量 ~800K 行。

**三个优化:**
1. Bitmap 替代 Hash Set (缓存命中率 15%→95%)
2. Dense Array 替代 Hash Map (零哈希开销)
3. 单线程执行 (避免同步开销)

### 11.4 验证结果

| Version | DuckDB (ms) | ThunderDuck (ms) | Speedup |
|---------|-------------|------------------|---------|
| V58 | 400 | 384 | 0.96x |
| V93 | 400 | 26 | **15.38x** |

**提升: +1502%**
