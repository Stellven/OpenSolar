## 原因一: 训练持久化效应

### 机制说明

Claude Code 没有记忆。每次会话结束, 一切清零。

Solar 通过 CLAUDE.md + rules/ 实现了**伪持久化训练**:

```
昊哥的纠正 → 昊哥写成规则文件 → Solar 下次会话读取 → 行为改变
                                                      ↑
                                              这不是 Claude 学会了
                                              这是规则文件里写了
```

### 量化规模

```
每次会话注入的"训练材料":

┌──────────────────────────────────────────────────────────────┐
│  CLAUDE.md                         ~3,000 bytes              │
│  rules/00-core-laws.md             ~2,000 bytes  (核心铁律)  │
│  rules/01-three-core-laws.md       ~3,000 bytes  (三铁律)    │
│  rules/state-persistence.md        ~5,000 bytes  (状态持久)  │
│  rules/solar-farm.md               ~3,000 bytes  (多智能体)  │
│  rules/cortex-first.md             ~3,000 bytes  (知识优先)  │
│  rules/multi-expert-analysis.md    ~2,000 bytes  (多专家)    │
│  rules/call-niuma-with-personality ~3,000 bytes  (人格注入)  │
│  rules/master-brain-persona.md     ~4,000 bytes  (主脑人格)  │
│  rules/tvs-rendering.md            ~2,000 bytes  (渲染规则)  │
│  其他 rules/*.md                    ~6,000 bytes              │
├──────────────────────────────────────────────────────────────┤
│  总计: ~36,000 bytes ≈ 9,000 tokens (每次会话固定消耗)       │
└──────────────────────────────────────────────────────────────┘
```

### 对比效果

```
Claude Code (无训练注入):
  用户: "帮我分析这个技术方案"
  Claude: [通用分析, 无个性化]

Solar (30,000 bytes 训练注入):
  用户: "帮我分析这个技术方案"
  Solar: [自动调用 Cortex 查询] + [自动多专家会审] +
         [战略家+治理官双签输出] + [TVS 格式渲染]
```

昊哥不需要每次都说"先查 Cortex 再分析"。规则文件里已经写了, 每次自动执行。

## 原因二: 每消息 12 个前置 Hook

### 机制说明

Claude Code 看到用户消息时, 脑子是空的。

Solar 看到用户消息前, 12 个 UserPromptSubmit hooks 已经触发:

```
┌─────────────────────────────────────────────────────────────────┐
│             用户消息到达后的 12 个自动动作                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1. 数据资产账本检查 → "你有哪些可用数据"                      │
│   2. Cortex 关键词提醒 → "先查知识库"                           │
│   3. Tantivy 搜索引擎状态检查                                   │
│   4. 铁律关键词检测 → 触发相应规则提醒                          │
│   5. 任务分配决策辅助 → "这个任务谁来干"                        │
│   6. 已有能力列表 (94 Skills) 预加载                            │
│   7. 当前路由模式检查                                           │
│   8. STATE.md 状态摘要注入                                      │
│   9. 成本模式检查 (economy/balanced/glm_only)                   │
│  10. 安全规则检查                                               │
│  11. 上下文利用率监控                                           │
│  12. 人格状态检查 (是否漂移)                                    │
│                                                                 │
│  效果: Solar 处理消息时, 已经知道"有什么工具"+"该怎么做"        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 为什么感觉"快"

Claude Code: 思考 → 工具调用 → 发现缺信息 → 再调用 → 发现缺工具 → 再调用...

Solar: 12 个 hooks 已预加载关键上下文 → 直接执行 → 结果

这就是为什么 Solar 感觉"一步到位", 而 Claude Code 感觉"绕弯子"。

## 原因三: 铁律是失败事件的编译产物

### 铁律 ≠ 建议

Claude Code 里的最佳实践是建议。Solar 的铁律是**防错代码**:

```
┌──────────────────────────────────────────────────────────────────┐
│                   铁律溯源表 (部分)                              │
├──────────────────┬──────────────────┬───────────────────────────┤
│  铁律名称        │  触发时间        │  原始失败事件              │
├──────────────────┼──────────────────┼───────────────────────────┤
│ 先查Cortex       │ 2026-02-13       │ 凭主观经验设计, 重复造轮子 │
│ 调牛马带人格     │ 2026-02-08       │ 4专家不带人格, 输出不一致  │
│ 调牛马带人格(v2) │ 2026-02-17       │ 手写过时 Big Five 格式     │
│ 状态持久化       │ 2026-02-09       │ compact 后失忆             │
│ 数据资产优先     │ 2026-02-04       │ 3400条数据只用了31条       │
│ 多专家会审       │ 2026-02-07       │ 分析只用单专家             │
│ 先规划后执行     │ 2026-02-07       │ 直接执行PDF, 上下文炸裂    │
│ 承诺即执行       │ 2026-02-06       │ 说了OK但没执行索引         │
│ 人格防挤出       │ 2026-02-07       │ 长对话后人格丢失            │
└──────────────────┴──────────────────┴───────────────────────────┘
```

每一条铁律背后都有一个具体的、代价高昂的失败案例。

### 累积效应

铁律不是一次性的。每次昊哥纠正 → 新铁律 → 下次会话自动防错。

这是一个**单向棘轮**: 只能变好, 不会退步。

Claude Code 没有这个棘轮。每次会话的错误都不会被"编译"。

## 原因四: 多模型专业化路由

### Claude Code 的局限

Claude Code 始终使用同一个模型处理所有任务:
- 写文档 → Claude
- 深度推理 → Claude
- 快速查询 → Claude (贵!)
- 代码生成 → Claude

### Solar 的路由智慧

```
┌──────────────────────────────────────────────────────────────────┐
│                  SOLAR 任务路由决策                               │
├────────────────────┬─────────────────────┬───────────────────────┤
│  任务类型          │  路由至              │  原因                 │
├────────────────────┼─────────────────────┼───────────────────────┤
│ 深度逻辑推理       │ deepseek-r1 (审判官) │ judge KNOBS 最强      │
│ 架构设计审查       │ gemini-2.5 (稳健派)  │ 一致性高, verifier    │
│ 创意方案探索       │ gemini-3 (探索派)    │ explorer KNOBS 最强   │
│ 中文创意写作       │ deepseek-v3 (创想家) │ 中文最好              │
│ 简单查询/格式化    │ glm-4-flash (小快手) │ 最便宜 $0.0001/1K     │
│ 长文档处理         │ gemini-flash (闪电侠)│ 1M 上下文最大         │
│ 战略决策分析       │ glm-5 (智囊)         │ architect KNOBS       │
│ 批量代码生成       │ glm-4-plus (建设者)  │ 成本低、配合度高      │
└────────────────────┴─────────────────────┴───────────────────────┘

成本对比:
• Claude Code: 固定 $0.003-$0.015/1K tokens
• Solar: $0.0001-$0.015 动态, 简单任务省 99%
```

### 专业化的质量效果

同一个分析任务:
- Claude Code: 通用分析, 无专业化
- Solar: 审判官做深度推理 + 探索派做方案探索 + 创想家做实现思路, 三路并行综合

这就是为什么 Solar 的分析感觉"更深更全"。

## 原因五: 状态连续性 (STATE.md)

### 问题: 每次会话的 Claude Code

```
会话 1: 昊哥说"我在做X系统, 要注意Y约束"
         Claude: 理解, 执行...
         会话结束

会话 2: 昊哥: "继续"
         Claude: "继续什么?" (完全忘了)
```

### Solar 的解法

```
会话 1: Solar 在 STATE.md 写入:
         Mission: 做 X 系统
         Constraints: Y 约束
         Decisions: [日期] 选择方案 A 而非 B, 原因...
         Progress: Done: 步骤1, In-Progress: 步骤2

会话 2: Solar 第一步读 STATE.md:
         "Mission: 做 X 系统, 约束 Y, 上次做到步骤2"
         自动接续
```

### 更深层: Decisions 追踪

STATE.md 的 Decisions 段落记录了所有决策的**理由**:

```markdown
## Decisions (Why)
- [2026-02-15] 选择 Tantivy 而非 FTS5:
  原因: Tantivy 已有15,513文档索引, 重建FTS5违反guardian-data-first铁律
- [2026-02-17] KNOBS 格式而非 Big Five:
  原因: Big Five 是过时格式, 被监护人纠正两次, 统一为 D&D KNOBS
```

Claude Code 没有决策历史。每次都可能做出相同的错误决策。

## 原因六: Cortex 知识库的累积效应

### 机制

每次 /insight 深度研究会写入 Cortex:

```
一次研究 → 90 artifacts 之一 → 下次相关任务自动查询
```

### 实际效果

```
第1次: 昊哥问关于 AI Agent 记忆系统
       → Solar 调用 insight-agent-v2 研究
       → 写入 Cortex: A-MEM, MemSurvey, Mem0 三篇论文分析

第3次: 昊哥设计 Solar 记忆架构
       → Solar 查询 Cortex: "memory"
       → 自动召回之前三篇论文分析
       → 基于证据设计, 不是凭空想象
```

Claude Code 没有这个积累。同一主题每次都要重新研究。

### 量化

```
Cortex 数据:
┌────────────────────────────────────────────────┐
│  artifacts:  90 个 (深度洞察产出)               │
│  sources:   482 个 (参考资料)                   │
│  entities:  128 个 (知识图谱实体)               │
│  relations: 1,343 个 (实体间关系)               │
└────────────────────────────────────────────────┘
这些都是可查询的、结构化的知识资产。
每次设计/开发前, unified-query.ts 自动检索。
```

## 原因七: collab_performance 数据驱动路由

### 1,015 条实际执行记录

```
每次牛马执行任务后记录:
┌──────────────────────────────────────────────────────────┐
│  model_id          │ 哪个模型                            │
│  task_type         │ 什么类型任务                        │
│  quality_score     │ 质量评分 (牛马互评)                 │
│  context_util      │ 上下文利用率                        │
│  latency_ms        │ 延迟                                │
│  cost_usd          │ 成本                                │
│  success           │ 是否成功                            │
└──────────────────────────────────────────────────────────┘
```

### 数据驱动路由优化

```sql
-- 可以查询: 哪个模型在代码生成任务上质量最高?
SELECT model_id, AVG(quality_score) as avg_quality,
       AVG(cost_usd) as avg_cost
FROM collab_performance
WHERE task_type = 'code_generation'
GROUP BY model_id
ORDER BY avg_quality DESC;
```

Claude Code: 每次都用同一个模型, 没有数据积累。
Solar: 1,015 条历史 → 路由决策有数据依据。

把以上七个原因叠加:

```
┌─────────────────────────────────────────────────────────────────┐
│                    "感觉强" 的来源分解                           │
├──────────────────────────┬──────────────────────────────────────┤
│  原因                    │  感知效果                            │
├──────────────────────────┼──────────────────────────────────────┤
│  30K bytes 训练注入      │ "不需要解释偏好, 默认就对"           │
│  12 个前置 Hooks         │ "很快就找到方向, 不绕弯子"           │
│  铁律防错系统            │ "同样的错误不再犯"                   │
│  12 模型专业化           │ "分析更深, 质量更好"                 │
│  STATE.md 状态连续       │ "记得之前的工作, 不需要反复交代"     │
│  Cortex 知识积累         │ "懂得越来越多, 越用越顺"             │
│  collab 数据路由         │ "任务分配越来越准"                   │
└──────────────────────────┴──────────────────────────────────────┘
```

### 最关键的洞察

**Solar 不是一个更聪明的 Claude。**
**Solar 是一个被昊哥训练了的 Claude。**

```
普通用户 + Claude Code:
  用户 = 空白画布上每次涂色
  Claude = 每次都是新画布

昊哥 + Solar:
  昊哥 = 30,000 bytes 的个人化训练数据
  Solar = 每次会话加载这 30,000 bytes
  效果 = 每次都是定制化 Claude
```

这个差距会随时间**单调递增**:
- 昊哥每纠正一次 → 新铁律 → 差距扩大
- 每次 /insight 研究 → Cortex 增长 → 差距扩大
- collab_performance 积累 → 路由更优 → 差距扩大

## 类比: 新员工 vs 老员工

```
Claude Code = 新员工 (每天第一天上班)
• 能力强, 但不知道公司规矩
• 不知道昊哥的偏好
• 不知道哪些坑踩过了
• 不知道上次做到哪里了

Solar = 老员工 (工作了一年)
• 同等能力, 但知道所有规矩
• 知道昊哥喜欢什么不喜欢什么
• 知道哪些路不能走 (踩过坑了)
• 知道项目全貌和历史决策
```

老员工"感觉强"不是因为更聪明。而是因为**积累了上下文**。

Solar 通过技术手段模拟了这种"积累"。
