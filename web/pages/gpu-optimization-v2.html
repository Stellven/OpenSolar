🎯 方案一: Two-Pass Radix Partitioning + Shared Histogram
两阶段基数分区 + 共享直方图 预期 2-3x
SIGMOD 2025 / PMC 2024
论文核心发现: GPU Join 性能瓶颈 75% 来自随机访问。通过两阶段分区 + 共享直方图，大幅减少内存访问冲突。

第一阶段: 粗分区

将数据分成 1024 个分区，每个分区大小适配 L2 Cache

第二阶段: 细分区

每个分区再分成 <512 tuples，完全放入 Shared Memory

共享直方图

每个 Thread Block 维护一个共享直方图，通过原子操作更新，比 per-thread 直方图节省 N 倍内存

VT (Values per Thread)

每个线程同时加载多个 tuple 到寄存器，提高指令级并行

// Metal Shader: Two-Pass Radix Partition kernel void radix_partition_pass1( device uint32_t* keys [[buffer(0)]], device uint32_t* histogram [[buffer(1)]], threadgroup uint32_t* shared_hist [[threadgroup(0)]], uint tid [[thread_position_in_grid]], uint lid [[thread_position_in_threadgroup]] ) { // 初始化共享直方图 (只需一次) if (lid < 1024) shared_hist[lid] = 0; threadgroup_barrier(mem_flags::mem_threadgroup); // 每个线程处理 VT=4 个元素 for (int i = 0; i < 4; i++) { uint32_t key = keys[tid * 4 + i]; uint32_t partition = (key >> 22) & 0x3FF; // 10-bit radix atomic_fetch_add_explicit(&shared_hist[partition], 1, memory_order_relaxed); } threadgroup_barrier(mem_flags::mem_threadgroup); // 合并到全局直方图... }

💡 关键洞察

传统方案: 每线程私有直方图 → 需要 threads × 1024 × 4B = 巨大内存
新方案: 共享直方图 + 原子操作 → 只需 1024 × 4B = 4KB / ThreadGroup

🎯 方案二: M3 Dynamic Shader Core Memory
利用 M3 动态寄存器分配 自动提升
Apple WWDC 2023
M3 (Apple Family 9) 引入动态着色器核心内存，寄存器按需分配/释放，显著提高线程占用率。

动态寄存器分配

旧: 静态分配最大寄存器 → 占用率低
新: 按需分配 → 自动提高占用率

灵活片上内存

Register/Threadgroup/Tile/Stack 共享统一缓存池，动态分配

使用 FP16

half 类型: 更少寄存器 + 更快运算 + 更高占用率

避免 Threadgroup 缓存

M3 上直接读 Device Buffer 可能比复制到 Threadgroup 更快

// M3 优化: 使用 half 类型 + 动态寄存器 kernel void bitmap_antijoin_m3( device half* customer_keys [[buffer(0)]], // FP16 节省寄存器 device uint64_t* bitmap [[buffer(1)]], device uint32_t* output [[buffer(2)]], device atomic_uint* counter [[buffer(3)]], ushort tid [[thread_position_in_threadgroup]], // ushort 足够 uint gid [[thread_position_in_grid]] ) { // M3: 直接从 Device Buffer 读取，利用 Flexible On-Chip Memory uint32_t key = (uint32_t)customer_keys[gid]; uint64_t mask = bitmap[key >> 6]; // SIMD 批量检查 (32 threads 并行) bool has_order = (mask >> (key & 63)) & 1; if (!has_order) { uint idx = atomic_fetch_add_explicit(counter, 1, memory_order_relaxed); output[idx] = key; } }

特性 M1/M2 M3
寄存器分配 静态 (程序全程占用最大值) 动态 (按需分配/释放)
Threadgroup 内存 推荐手动缓存 可能直接读更快

🎯 方案三: GFTR - 减少随机访问
GFTR (GPU-Friendly Table Reordering) 2.3x 加速
arXiv 2312.00720
论文发现: 75% 的 GPU Join 运行时间花在随机内存访问上。GFTR 通过表重排序预处理，将随机访问转为顺序访问。

问题根源

Hash Join Probe 阶段: hash(key) → 随机位置 → Cache Miss

GFTR 思路

预先按 hash 值排序 Probe 表，使访问 Build 表变得更局部化

Sort vs Hash 权衡

排序成本 O(n log n) vs 随机访问惩罚，数据量大时排序更划算

适用场景

Build 表较大、Probe 次数多、随机访问严重时收益最大

// Anti-Join with GFTR: 预排序 + 顺序访问 // Step 1: 对 Probe 表按 bitmap 位置排序 // 使 bitmap 访问从随机变为顺序扫描 sort_by_key(customer_keys, [](uint32_t k) { return k >> 6; }); // Step 2: 顺序扫描 bitmap (利用缓存预取) kernel void antijoin_sequential( device uint32_t* sorted_keys [[buffer(0)]], device uint64_t* bitmap [[buffer(1)]], ... ) { // 连续的 gid 访问连续的 bitmap 块 // GPU L2 Cache 命中率大幅提升 uint64_t chunk = bitmap[sorted_keys[gid] >> 6]; // Coalesced memory access! }

💡 为什么对 Anti-Join 有效

我们的 Q22 场景:
• 150K customers 检查 bitmap (19KB)
• 原方案: customer_id 随机分布 → bitmap 随机访问
• GFTR: 按 customer_id/64 排序 → bitmap 顺序扫描
• 19KB bitmap 完全放入 M3 L2 Cache (4MB) → 近乎 100% 命中
