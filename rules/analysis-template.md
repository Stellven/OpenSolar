# Solar 铁律: 量化分析模板

> **来源**: Meta-Harness Iter-004 — SE-014(7/10) + SE-016(7/10) 失败分析
> **核心问题**: 分析类回答缺乏量化深度，定性多定量少

## 铁律定义

当回答涉及「性能分析」「延迟分析」「根因分析」「对比分析」时，必须满足：

### 1. 量化证据链（必须有数字）

```
❌ 错误: "数据库查询很慢"
✅ 正确: "DB 查询平均 500ms（P99=1.2s），占总延迟的 47%"

❌ 错误: "内存使用过高"
✅ 正确: "内存从 200MB 飙升到 500MB（+150%），每次图片处理增加 ~30MB"
```

### 2. 延迟/性能分解链（数值标注）

分析性能问题时，必须给出完整的分解链：

```
示例:
总延迟 = Gateway(50ms) + DB(500ms, 缓存未命中时) + UserSvc(200ms) + ProductSvc(300ms)
      = 50 + 500 + 200 + 300 = 1050ms (慢路径)
      = 50 + 50 + 200 + 300 = 600ms (快路径, 缓存命中)
```

### 3. 影响范围量化

```
❌ 错误: "高峰期会受影响"
✅ 正确: "高峰 QPS=200 > 限流 100QPS → 50% 请求排队，预计延迟翻倍"

❌ 错误: "多个服务会崩溃"
✅ 正确: "E 崩溃 → 直接影响: B(依赖E), C(依赖E) → 间接影响: A(依赖B,C) = 3/5 服务"
```

### 4. 优化方案必须量化预期效果

```
❌ 错误: "优化后性能会提升"
✅ 正确: "并行调用 UserSvc+ProductSvc → 省去 200+300ms 串行等待 → 总延迟 1050→550ms (-48%)"
```

### 5. 优化方案按 ROI 排序

| ROI 维度 | 说明 |
|----------|------|
| 投入 | 开发时间/复杂度 |
| 产出 | 量化效果（延迟降低%、成本节省%） |
| 风险 | 对现有系统的影响 |

### 6. 概率/统计计算推导链（显式展开）

涉及概率计算、统计推理、数学优化时，每步必须展开：

```
❌ 错误: "P(A) = 0.43"（不说明怎么来的）
✅ 正确: "P(A) = P(简单)×P(A|简单) + P(中等)×P(A|中等) + P(困难)×P(A|困难)
       = 0.5×0.7 + 0.3×0.2 + 0.2×0.1
       = 0.35 + 0.06 + 0.02 = 0.43"

❌ 错误: "整体成功率 = 0.8170"（缺公式链）
✅ 正确: "P(成功) = P(A)×SR(A) + P(B)×SR(B) + P(C)×SR(C)
       = 0.43×0.9 + 0.31×0.8 + 0.26×0.7
       = 0.387 + 0.248 + 0.182 = 0.8170"
```

**要求**：中间计算的每一步都要展开，不能跳步。优化方案必须说明"为什么这个是最优"（Pareto分析或约束边界分析）。

### 7. 内存/OOM 分析专用模式

分析 OOM 问题时，必须量化每步内存消耗：

```
## OOM 分析模板
1. 基线内存: 正常时 X MB
2. 峰值触发: Y MB (超过限制 Z MB)
3. 每次操作内存增量: Δ MB/次
4. 峰值计算: 基线 + 并发数 × 每次增量 = X + N × Δ = Y MB
5. 量化根因: 并发从1增加到N，内存增长 (N×Δ)/X = 百分比%
6. 快速修复: 限制并发到M → 峰值 = X + M × Δ ≤ Z × 0.8 (留20%安全边界)
7. 长期方案: 独立进程/流式处理/内存池
```

示例:
```
基线: Node.js 200MB, 限制 512MB
每次图片处理: sharp 实例占 30-50MB
批量10张并发: 200 + 10×40 = 600MB > 512MB → OOM
快速修复: 并发限制为2 → 200 + 2×40 = 280MB < 512MB×0.8=410MB ✓
```

### 8. 串行→并行优化模式 (v1.1 新增)

分析多个服务/步骤的延迟时，必须识别串行依赖链并给出并行优化方案：

```
## 延迟预算分析
1. 列出所有步骤及其延迟
2. 标注依赖关系: A→B(串行), C和D(可并行)
3. 计算关键路径: 串行步骤延迟之和

## 串行→并行转换
总延迟_串行 = Σ(all_steps) = A + B + C + D + ...
总延迟_优化 = 关键路径 = max(并行分支) + 串行部分
节省 = 总延迟_串行 - 总延迟_优化

示例:
串行: Gateway(50) + DB(500) + UserSvc(200) + ProductSvc(300) = 1050ms
并行: Gateway(50) + DB(500) + max(UserSvc(200), ProductSvc(300)) = 850ms
节省: 1050 - 850 = 200ms (-19%)
```

### 9. QPS 溢出分析模式 (v1.1 新增)

当问题涉及流量/并发时，必须做 QPS 容量分析：

```
## QPS 容量分析
1. 系统容量: X QPS (限流/资源上限)
2. 实际负载: 峰值 Y QPS
3. 溢出: Y > X → (Y-X)/Y = Z% 请求被限流/排队
4. 量化影响: 排队延迟 = (Y/X - 1) × 平均响应时间

示例:
限流 100 QPS, 高峰 200 QPS → 溢出率 = (200-100)/200 = 50%
排队延迟 = (200/100 - 1) × 500ms = 500ms → 实际延迟 500+500=1000ms
```
