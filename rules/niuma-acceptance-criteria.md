# Solar 铁律: 牛马验收清单 (Niuma Acceptance Criteria)

> **来源**: Meta-Harness Iter-002 — 模型能力弱点(SE-002=5, SE-019=5)转验收标准
> **核心**: 牛马产出必须过验收门槛，不能"交了就行"

## 通用编码验收清单

任何牛马提交的代码实现必须通过以下检查：

| # | 检查项 | Pass 标准 | 检测方法 |
|---|--------|-----------|----------|
| AC-1 | 核心功能完整 | 所有题干要求的方法/函数都有实现 | 逐项对照需求 |
| AC-2 | 边界处理 | 空输入、单元素、极端值均有处理 | 检查 if/try/guard |
| AC-3 | 无 Mock/桩 | 零 TODO/FIXME/模拟输出 | grep(mock/TODO/stub/模拟) |
| AC-4 | 测试覆盖 | 每个公开方法 ≥1 测试用例 | 检查测试文件 |
| AC-5 | 错误隔离 | 一个失败不影响其他 | Promise.allSettled/try-catch |

## 场景特定验收标准

### 并发/调度类 (从 SE-002 TaskScheduler 教训提取)

| 检查项 | Pass 标准 | 常见失败模式 |
|--------|-----------|-------------|
| 并发控制 | 使用信号量/worker pool，非简单 Promise.all | 无并发限制，全部并行 |
| 优先级队列 | 按优先级排序后再消费，非 FIFO | 先到先做，忽略优先级 |
| 取消逻辑 | 区分 running/queued 状态，只取消 queued | cancel() 对已运行任务也生效 |
| 错误隔离 | 单任务失败不中断其他任务 | 一个失败全部 abort |
| 进度回调 | completed/total 精确 | 跳过失败任务不计入 completed |

### 设计/规则类 (从 SE-019 复杂度评分教训提取)

| 检查项 | Pass 标准 | 常见失败模式 |
|--------|-----------|-------------|
| 映射表完整 | 覆盖所有区间，无遗漏行 | 只列了 3/5 个级别 |
| 阈值有依据 | 数学推导或经验数据支撑 | "凭经验设为7" |
| 伪代码可执行 | if/for/return 结构，非描述性文字 | "计算得分并返回" |
| 正反例 | ≥2 触发 + ≥2 不触发 | 只有触发没有不触发 |
| 自动化方案 | 指明 hook 位置 + 检测逻辑 | "可以自动化检测" |

## 验收流程

```
牛马提交 → 逐项检查 AC-1..5 → 场景特定检查 → 全部 PASS → 接受
                                          → 有 FAIL → 打回重做 + 指明失败项
```

## 打回模板

```
❌ 验收失败 — [牛马名] 提交的 [任务名]

失败项:
- AC-X: [检查项] — [具体问题]
  期望: [标准]
  实际: [牛马做了什么]

修复方向: [具体建议]
```
