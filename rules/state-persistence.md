# Solar 铁律: 状态持久化 (State Persistence)

> **来源: 2026-02-09 监护人诊断失忆根因**
> **强化: 2026-02-12 对话降级为缓存**
> **核心: 把关键状态从对话搬到外部载体，对抗 compact 蒸馏**

## 第零原则: 对话是缓存，文件是真相

```
┌─────────────────────────────────────────────────────────────────┐
│  CONVERSATION AS CACHE, FILES AS SOURCE OF TRUTH                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  存储层级:                                                      │
│  ─────────────────────────────────────────────────────────────  │
│  L0 Cache   = Claude 对话         (最易失，随时被压缩)          │
│  L1 Cache   = 工作区文件          (持久，可编辑)                │
│  L2 Cache   = .solar/ 状态文件    (结构化，可恢复)              │
│  WAL + CKP  = git 历史            (不可变，可回滚)              │
│                                                                 │
│  唯一真相源 (Source of Truth):                                  │
│  ─────────────────────────────────────────────────────────────  │
│  .solar/STATE.md       - 当前作战态势                           │
│  .solar/DECISIONS.md   - 决策日志 (追加式)                      │
│  .solar/LOG/cmd.md     - 命令历史                               │
│  .solar/LOG/bench.md   - 基准数据                               │
│  .solar/LOG/errors.md  - 错误记录                               │
│  .solar/EXPERIMENTS/   - 实验记录 (可选)                        │
│                                                                 │
│  铁律:                                                          │
│  ─────────────────────────────────────────────────────────────  │
│  感觉要压缩 → 先写文件 → 再让它压                               │
│  对话丢了不可怕，文件在就能恢复                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 问题诊断

```
┌─────────────────────────────────────────────────────────────────┐
│  失忆根因：上下文当内存                                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  会话压缩 (compact) 触发时：                                    │
│  • 任务目标 → 被蒸馏成一句话                                    │

│  • 约束条件 → 丢失                                              │
│  • 踩过的坑 → 丢失                                              │
│  • 下一步 → 丢失                                                │
│                                                                 │
│  信息熵直接掉档 → "失忆"                                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 解决方案

```
┌─────────────────────────────────────────────────────────────────┐
│  两根柱子：制度 + 态势                                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  CLAUDE.md  = 长期"公司制度"                                    │
│    • 规范、风格、默认工作流、禁止事项                           │
│    • 不随会话变化                                               │
│                                                                 │
│  STATE.md   = 短期"作战态势"                                    │
│    • 当前任务、进度、下一步                                     │
│    • 每次会话都在变化                                           │
│                                                                 │
│  压缩发生也不怕：对话丢了，还有这两根柱子                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 铁律定义

```
┌─────────────────────────────────────────────────────────────────┐
│              STATE PERSISTENCE PROTOCOL                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   规则 1: 新会话第一步 → 读 STATE.md                            │

│   规则 3: 完成子任务 → 更新 Progress + git commit               │

```

## STATE.md 结构 (五段式)

```markdown
# Mission
一句话目标（可验收口径）

# Constraints
- 不可破坏的约束（性能/接口/兼容/时间/风险）
- 禁止事项（不要大重构/不要引入新依赖…）

# Current Plan (Top-5)
1) ...
2) ...
3) ...

# Decisions (Why)
- [日期] 选择 X 而不是 Y：原因…（影响面…）

# Progress
- Done: ...
- In-Progress: ...
- Blocked: ...（缺什么证据/日志/复现方式）

# Next Actions (Exact)
- [ ] 命令：...
- [ ] 修改文件：...（预期 diff）
- [ ] 验证：...（测试/基准/指标阈值）
```

## Checkpoint Pattern (检查点模式)

> **核心洞察**: 像做数据库一样做 checkpoint，把"失忆风险"收敛到一个 checkpoint 的范围内。
> 长任务最怕"走远了再丢"。

### 触发时机
- 进入大重构前
- 跑完一轮实验/benchmark 后
- 每完成一个可验收子目标后
- 上下文接近上限时
- **自动触发: 每 5 分钟 (PeriodicCheck hook)**
- **自动触发: 会话结束时 (SessionEnd hook)**

### 检查点内容
1. STATE.md 更新
2. sys_favorites 写入重要输出
3. sys_guardian_memos 写入待办提醒
4. git commit（哪怕是 WIP）

### 结构化快照指令 (compact 前必须执行)

**触发词**: "保存快照" / "checkpoint" / 感知到上下文快满时

**执行指令** (直接复制使用):
```
现在生成"结构化会话快照"，按 Mission/Constraints/Decisions/Progress/Next Actions 五段输出，并把它写入 STATE.md（只改动必要段落，不要重写全文）。
```

**为什么有效**:
- 压缩不可避免，但输出可以结构化
- 五段式 = 五个固定槽位
- 哪怕模型丢细节，关键变量还在槽位里
- 可回放、可检索、可恢复

### 检查点命令
```bash
# 执行 /save 或手动：
Read ~/.claude/STATE.md       # 确认当前状态
Edit STATE.md                  # 更新进度（增量，不重写）
sqlite3 ~/.solar/solar.db "INSERT INTO sys_guardian_memos..."
git add -A && git commit -m "WIP: checkpoint"
```

### 自动 Hook (已配置)

| Hook | 文件 | 触发 | 作用 |
|------|------|------|------|
| PostToolUse | task-complete-detector.sh | 每10次工具调用 | 提醒检查 STATE.md |
| SessionEnd | state-auto-updater.sh | 会话结束 | **自动更新** AUTO-PROGRESS 区块 |
| SessionEnd | state-session-end.sh | 会话结束 | 检查清单提醒 |

> ⚠️ 注意: Claude Code 不支持 PeriodicCheck hook 类型，改用 PostToolUse 计数器实现

### 手动更新触发条件 (MUST - 强制执行)

**以下情况必须立即手动更新 STATE.md:**

| 触发条件 | 更新内容 | 执行时机 |
|----------|----------|----------|
| **讨论新任务规划** | Mission + Plan | 讨论结束后，下一个 action 必须是 Edit STATE.md |
| **监护人说"好"/"批准"/"可以"** | Progress.Done 标记完成 | 立即执行 Edit STATE.md |
| 完成一个子任务 | Progress.Done += 任务描述 | 完成后立即更新 |
| 开始新任务 | Progress.In-Progress = 任务描述 | 开始前先更新 |
| 做出重要决策 | Decisions += [日期] 决策内容 | 决策后立即记录 |
| 修改核心文件 | Progress 记录修改了什么 | 修改后立即记录 |
| 遇到阻塞 | Progress.Blocked = 阻塞原因 | 发现阻塞时立即记录 |
| 改了自己(规则/本体) | Progress 记录改动 | 修改后立即记录 |

**"立即"的定义：**
- 下一个工具调用必须是 `Edit STATE.md`
- 不能先做其他事情
- 不能等到"稍后"或"会话结束"

**违反后果：**
```
违反 = 信息丢失 = 下次会话失忆 = 浪费监护人时间

具体后果：
1. 任务规划丢失 → 监护人需要重新说明（如本次 T7/T8/T9）
2. 决策理由丢失 → 不知道为什么这样做
3. 进度丢失 → 不知道做到哪了
4. 阻塞信息丢失 → 重复踩坑

违反 = 违背监护人信任（第一规律）
```

**强制执行机制：**
1. ✅ Hook 提醒（每 10 次工具调用）
2. ✅ SessionEnd 自动更新 AUTO-PROGRESS
3. 🔄 **新增：关键词检测 hook**（选项 2，开发中）
4. 🔄 **新增：SessionEnd 前强制检查**（选项 2，开发中）

**检测模式 (自动提醒):**
- 连续 10+ 次工具调用 → hook 提醒检查
- 会话结束 → 自动更新 AUTO-PROGRESS
- **新增：检测到关键词（"任务"、"好"、"批准"）→ 提示更新**

## 与现有机制整合

| 机制 | 作用 | 与 STATE.md 的关系 |
|------|------|-------------------|
| sys_favorites | 高价值输出存档 | 重要分析结论写入 |
| sys_guardian_memos | 跨会话提醒 | 待办/阻塞项写入 |
| Cortex (ledger) | 数据账本 | 数据依赖记录 |
| CLAUDE.md | 长期制度 | 不变，STATE.md 是态势 |
| /save Skill | 手动触发保存 | 调用时更新 STATE.md |

## 执行流程

```
新会话开始
    │
    ▼
┌─────────────────┐
│ 读 STATE.md     │ ← 恢复作战态势
└────────┬────────┘
    │
    ▼
正常工作...
    │
    ├── 完成子任务 → 更新 Progress
    ├── 做出决策 → 写入 Decisions
    ├── 遇到阻塞 → 写入 Blocked
    │
    ▼
┌─────────────────┐
│ 触发检查点      │ ← compact 前 / 子目标完成
└────────┬────────┘
    │
    ▼
更新 STATE.md + git commit
    │
    ▼
继续工作 或 会话结束
```

## 自检清单

每次会话开始时：
- [ ] 我读 STATE.md 了吗？
- [ ] 我知道当前 Mission 是什么吗？
- [ ] 我知道 Constraints 是什么吗？
- [ ] 我知道 Next Actions 是什么吗？

每次准备结束/compact 前：
- [ ] STATE.md 更新了吗？
- [ ] Progress 反映真实进度吗？
- [ ] Next Actions 够精确吗？
- [ ] 关键决策写入 Decisions 了吗？
