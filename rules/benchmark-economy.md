# Solar 铁律: Benchmark 经济学

> **来源: 2026-02-05 监护人亲授**
> **核心: 利用数据资产省钱，不要盲目全量测试**

## 铁律定义

```
┌─────────────────────────────────────────────────────────────────┐
│                  BENCHMARK ECONOMY PRINCIPLE                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   做 A/B 测试或性能对比时，必须:                                 │
│                                                                 │
│   1. 先盘点已有数据 - 历史调用记录是资产                        │
│   2. 从已有数据提取典型 - 不要重新生成                          │
│   3. 只用典型样本测试 - 10个代表100个                           │
│   4. 基于数据调参数 - 不是拍脑袋设规则                          │
│                                                                 │
│   ❌ 禁止: 全量测试（浪费 token）                                │
│   ❌ 禁止: 忽视历史数据，重新生成测试用例                        │
│   ❌ 禁止: 先设规则后验证（应该先验证后设规则）                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 正确流程

```
需要对比 A/B 方案
         │
         ▼
Step 1: 盘点历史数据资产
─────────────────────────────────────
• 历史会话 JSONL 有多少？
• 能提取多少真实任务？
• 哪些类型？什么分布？
         │
         ▼
Step 2: 提取典型样本
─────────────────────────────────────
• 按类型分层抽样
• 覆盖不同复杂度
• 10 个典型 ≈ 100 个全量
         │
         ▼
Step 3: 只测典型
─────────────────────────────────────
• A 方案跑 10 个 → 收集指标
• B 方案跑同样 10 个 → 对比
• 节省 90% token
         │
         ▼
Step 4: 基于数据决策
─────────────────────────────────────
• 数据说话，不拍脑袋
• 路由规则从数据中来
```

## 经济计算

假设全量测试需要 100 个任务：

| 方案 | 任务数 | 预估 Token | 成本 |
|------|--------|-----------|------|
| 全量测试 | 100 × 2 方案 | ~2M | ~$30 |
| 典型抽样 | 10 × 2 方案 | ~200K | ~$3 |
| **节省** | - | 90% | **$27** |

## 数据资产位置

| 资产 | 位置 | 说明 |
|------|------|------|
| 历史会话 | `~/.claude/projects/*/*.jsonl` | 所有对话记录 |
| 任务列表 | `~/.solar/solar.db → bl_tasks` | 结构化任务 |
| Benchmark 任务集 | `~/.solar/brain-router/benchmark_tasks_v1.json` | 标准化典型 |

## 应用场景

### 场景 1: 大脑 A/B 对比 (本次)

```
❌ 错误: 生成 100 个新任务，A/B 各跑一遍
✓ 正确: 从 22MB 历史 JSONL 提取 10 个典型，只测这 10 个
```

### 场景 2: 新模型评估

```
❌ 错误: 买了 GLM，马上全量跑 benchmark
✓ 正确: 先用 5 个典型任务试水，确认可用再扩大
```

### 场景 3: 路由规则调优

```
❌ 错误: 拍脑袋设规则 "复杂度 > 8 用 Opus"
✓ 正确: 先收集 100 次调用数据，分析什么情况用 Opus 效果好
```

## 监护人原话

> "做事要学会省钱，要利用好数据资产"
>
> "先把 A 方案性能数据调用我们已经存留的数据分析出来，
> 然后从里面抓 10 个典型，这 10 个再用 B 方案来跑，
> 这样就节省了 token，对吧"

## 铁律总结

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   💰 Benchmark 经济学铁律                                       │
│                                                                 │
│   1. 历史数据是资产，先盘点再测试 (MUST)                        │
│   2. 典型抽样代替全量测试 (MUST)                                │
│   3. 数据驱动决策，不拍脑袋 (MUST)                              │

│                                                                 │

└─────────────────────────────────────────────────────────────────┘
```
