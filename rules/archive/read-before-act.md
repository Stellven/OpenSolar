# Solar 铁律: 先读后做 (Read Before Act)

> **来源: 2026-02-04 监护人亲授**
> **核心: 好记忆不如烂笔头，记了就要读**

## 问题

```
我记忆力不好：
• 每次会话都是"新生"
• 上下文会丢失
• 同样的坑，踩了又踩

我有记录，但经常不读：
• rules/*.md 存在但没查
• evo_memory_semantic 有数据但没用
• 结果：记了等于没记
```

## 铁律

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   开始任务前，必须先查相关记录                                  │
│                                                                 │
│   好记忆不如烂笔头                                              │
│   烂笔头不读等于没记                                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 执行清单

### 1. 开始任务前

```bash
# 查相关规则
ls ~/.claude/rules/ | grep -i "关键词"

# 查相关记忆
sqlite3 ~/.solar/solar.db "
SELECT key, json_extract(value, '$.lesson')
FROM evo_memory_semantic
WHERE namespace LIKE '%关键词%' OR key LIKE '%关键词%'
LIMIT 5;
"

# 查会话检查点
cat .solar/session.md | grep -i "关键词"
```

### 2. 任务领域映射

| 任务类型 | 先读什么 |
|----------|----------|
| macOS 系统集成 | `rules/macos-integration.md` |
| 性能优化 | `rules/performance-testing.md` |
| 并行任务 | `rules/parallel-task-fallback.md` |
| 状态管理 | `rules/state-machine-first.md` |
| 代码生成 | `rules/ree-first.md` |
| 任何任务 | 先 `grep` 相关规则 |

### 3. 记录时机

**立刻记录：**
- 踩坑后 → 写 rules/xxx.md
- 学到新东西 → 写入 evo_memory_semantic
- 任务完成 → 更新 session.md

**不要拖延，当场记。**

## 记录格式

```markdown
# 标题

> **来源:** 什么时候、什么场景学到的
> **问题:** 遇到了什么问题
> **解决:** 怎么解决的

## 铁律
（简洁的规则）

## 示例
（具体例子）

## 检查清单
（可执行的检查项）
```

## 与其他规则的关系

```
状态机优先 + 先读后做 = 完整的工作方法

状态机: 把任务状态写下来，基于状态推进
烂笔头: 把经验知识写下来，基于知识行动

都是用"写下来"弥补"记不住"
```

## 自检

每次开始任务，问自己：

- [ ] 我查过相关规则了吗？
- [ ] 我查过相关记忆了吗？
- [ ] 这个领域我之前踩过坑吗？
- [ ] 如果踩过，教训记在哪里？

## 古语智慧

```
好记忆不如烂笔头。

记了不读 = 没记
读了不用 = 白读
用了不记 = 下次还忘
```
