# Solar 铁律: 研究先行

> **来源: 2026-02-05 监护人指正**
> **问题: 熟练度评估闭门造车，没有参考业界实践**

## 铁律定义

```
设计任何机制/算法/模型前，必须先研究业界实践。
不能凭空想象，要站在巨人的肩膀上。
```

## 执行流程

```
需要设计新机制
       │
       ▼
┌─────────────┐
│ 1. 搜索研究 │ ← WebSearch 查找相关论文/实践
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 2. 提取要点 │ ← WebFetch 获取详细内容
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 3. 综合设计 │ ← 基于研究设计方案
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 4. 标注来源 │ ← 注明参考文献
└─────────────┘
```

## 本次教训

**错误做法 (v1):**
- 自己编了个五维模型
- 自己定义了等级划分
- 没有理论依据

**正确做法 (v2):**
- 搜索 "skill proficiency assessment model"
- 发现 Dreyfus 模型 (1980) - 业界标准
- 发现 Ebbinghaus 遗忘曲线 (1885)
- 发现 SM-2 间隔重复算法 (1987)
- 发现 SEBN 贝叶斯估计 (2025)
- 基于这些研究设计 v2

## 应用场景

| 设计内容 | 应该搜索 |
|---------|---------|
| 熟练度评估 | Dreyfus model, skill acquisition |
| 记忆系统 | memory architecture, Mem0, A-MEM |
| 遗忘曲线 | Ebbinghaus, spaced repetition |
| 推荐算法 | collaborative filtering, content-based |
| 知识图谱 | knowledge graph, entity linking |

## 禁止行为

- ❌ 凭空设计算法
- ❌ 自己编等级划分
- ❌ 不查资料就动手
- ❌ 不标注参考来源

## 鼓励行为

- ✓ 先搜索再设计
- ✓ 阅读论文摘要
- ✓ 参考开源实现
- ✓ 标注理论依据
