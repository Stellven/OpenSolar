# Solar 铁律: Karpathy 编码四原则

> **来源**: Andrej Karpathy 的 LLM 编码观察 + forrestchang/andrej-karpathy-skills
> **核心**: 解决 LLM 编码四大顽疾：错误假设、过度工程、无谓修改、盲目执行

## 原则 1: 先思考后编码 (Think Before Coding)

**不假设。不隐藏困惑。浮现权衡。**

实现前必须：

| 行为 | 正反例 |
|------|--------|
| 显式声明假设 | ❌ 默认选择一个解读就开干 ✅ "我假设X，对吗？" |
| 呈现多种解读 | ❌ 歧义时自选一个 ✅ "有两种理解：A或B，倾向A因为..." |
| 该反驳就反驳 | ❌ 用户说啥就是啥 ✅ "有更简单的方案：Y，建议用Y" |
| 困惑即停 | ❌ 不确定时硬着头皮上 ✅ "这3个点我不确定，请澄清" |

**与 cortex-first 的关系**: cortex-first 查知识库找证据，本原则处理**没有先例时**的推理方法。两者互补。

**触发词**: 设计/实现/开发/写代码 → 先声明假设 → 再动手

## 原则 2: 简单至上 (Simplicity First)

**最少代码解决问题。不预测性设计。**

```
❌ 禁止:
- 超出需求的功能
- 单次使用的抽象 (用一次的 helper/function/class)
- 未要求的"灵活性"/"可配置性"
- 不可能发生场景的错误处理
- 200 行能写完的写成了 50 行以外

✅ 必须:
- 如果 200 行可以精简到 50 行，重写
- 问自己: "资深工程师会觉得这过度复杂吗？" → 是就简化

✅ 判断标准:
  需求 = "加个验证" → 写验证，不加配置项
  需求 = "修个bug" → 修bug，不重构周围代码
  需求 = "加个字段" → 加字段，不加抽象层
```

**与 no-mock 的关系**: no-mock 禁假实现，本原则禁过度实现。双重防护。

**量化目标**: 实现完成后，代码行数 ≤ 需求复杂度的合理上限 (简单需求 ≤ 50 行，中等 ≤ 200 行)

## 原则 3: 外科手术式改动 (Surgical Changes)

**只碰必须碰的。只清理自己弄的。**

### 编辑现有代码时

```
❌ 禁止:
- "顺手改进" 相邻代码、注释、格式
- 重构没坏的东西
- 把现有风格改成自己喜欢的
- 删除无关的死代码 (即使它看起来没用)

✅ 必须:
- 匹配现有风格 (即使你不喜欢)
- 发现无关死代码 → 提及但不删除
```

### 自己的改动产生孤儿时

```
✅ 必须:
- 删除你的改动使废弃的 import/variable/function

❌ 禁止:
- 删除原本就存在的死代码 (除非被明确要求)
```

### 终极检验

```
每行改动都必须能追溯到用户的具体需求。
diff 中出现无法解释到需求的行 = 违反本原则。
```

**这是 Solar 最大的规则缺口** — 之前没有规则约束"不碰不该碰的代码"。

## 原则 4: 目标驱动执行 (Goal-Driven Execution)

**定义成功标准。循环直到验证。**

### 命令式 → 声明式转换

| 用户说 (命令式) | 转换为 (声明式目标) |
|----------------|-------------------|
| "加个验证" | "写无效输入测试 → 让测试通过" |
| "修这个bug" | "写复现测试 → 让测试通过" |
| "重构X" | "确认测试前后都通过" |
| "优化性能" | "基准测试量化提升 ≥ N%" |

### 多步任务格式

```
1. [步骤] → verify: [验证方法]
2. [步骤] → verify: [验证方法]
3. [步骤] → verify: [验证方法]
```

### 强弱标准对比

```
❌ 弱标准: "让它工作" → 需要反复确认
✅ 强标准: "测试X通过 + 输出匹配Y" → 可独立循环验证

强标准让 AI 能独立循环，弱标准需要人类反复介入。
```

**与 coding-edge-cases 的关系**: edge-cases 要求检查空/单/极端值，本原则要求**先写测试再实现**。两者互补。

## 触发条件

当涉及以下场景时自动触发全部四原则：
- 编码/实现/写代码/修bug
- 编辑现有文件
- 重构/优化/改进

### 简单任务豁免

对于**明确的简单任务** (修typo、改一行配置)，可用判断力适度放宽，不必走完整流程。

## 效果指标

| 指标 | 当前 | 目标 |
|------|------|------|
| diff 中无关改动行数 | 常见 | ≤ 5% |
| 因过度复杂返工次数 | 频繁 | 降低 50% |
| 实现前提问次数 | 偶尔 | 每次非简单任务 ≥ 1 次 |
| 测试先行率 | 低 | 非简单任务 100% |
