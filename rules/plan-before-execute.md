# Solar 铁律: 先规划后执行 (Plan Before Execute)

> **来源: 2026-02-07 监护人指正**
> **问题: 一上来就干，导致上下文爆掉死机**
> **专家来源: Gemini 2.5 Pro + DeepSeek R1 + Gemini 3 Pro**

## 铁律定义

```
┌─────────────────────────────────────────────────────────────────┐
│                  PLAN BEFORE EXECUTE                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   执行复杂任务前，必须:                                         │
│                                                                 │
│   1. 预评估 - 检查文件大小/页数，计算复杂度                     │
│   2. 判断阈值 - 超过上下文 40% 就是复杂任务                     │
│   3. 强制规划 - 复杂任务必须先输出规划，不能直接干              │
│   4. 分步执行 - 每步只加载当前块，执行后压缩历史                │
│                                                                 │
│   ❌ 禁止: 一上来就干，不评估不规划                             │
│   ❌ 禁止: 一次性读取大文件                                     │
│   ❌ 禁止: 不压缩历史，让上下文堆积                             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 执行流程

```
任务到达
    │
    ▼
┌─────────────────┐
│ Step 1: 预评估 │
└────────┬────────┘
         │
    ┌────┴────┐
    ▼         ▼
[简单任务]  [复杂任务]
 <40%阈值    ≥40%阈值
    │         │
    ▼         ▼
直接执行   强制规划
              │
              ▼
         输出JSON规划
              │
              ▼
         ┌────────────┐
         │ 分步执行   │←──┐
         │ (当前块)   │   │
         └─────┬──────┘   │
               │          │
               ▼          │
         压缩历史         │
               │          │
               ▼          │
         还有更多块？─────┘
               │ 否
               ▼
            完成
```

## 阈值定义 (动态调整)

> **老专家审核意见**: 40%阈值过于静态，应动态调整

| 任务类型 | 阈值 | 说明 |
|----------|------|------|
| 文本摘要 | 40% | 可压缩，阈值可高 |
| 代码生成 | 30% | 需精确，阈值要低 |
| 多文件处理 | 25% | 上下文切换多 |
| 代码库分析 | 20% | 复杂度高 |

其他指标参考：
| 指标 | 阈值 | 说明 |
|------|------|------|
| PDF 页数 | ~50页 | 大概估算 |
| 代码文件 | ~500行 | 单文件 |
| 多文件 | >3个 | 需要分步 |

## 规划输出格式

复杂任务必须先输出规划：

```json
{
  "task": "读取并总结100页PDF",
  "complexity": "high",
  "estimated_context": "65%",
  "steps": [
    {"step": 1, "action": "read_pages", "range": "1-20", "goal": "提取第一部分要点"},
    {"step": 2, "action": "summarize_chunk", "input": "step1结果"},
    {"step": 3, "action": "read_pages", "range": "21-40", "goal": "提取第二部分要点"},
    {"step": 4, "action": "summarize_chunk", "input": "step3结果"},
    {"step": 5, "action": "merge_summaries", "input": "所有摘要", "goal": "生成最终总结"}
  ],
  "checkpoints": ["step2后压缩", "step4后压缩"]
}
```

## 检查清单

执行任务前，问自己：

- [ ] 这个任务大概要占多少上下文？
- [ ] 超过 40% 吗？
- [ ] 如果超过，我规划了吗？
- [ ] 规划里有分步吗？
- [ ] 每步后会压缩历史吗？

## 压缩策略

每步执行后：
1. 只保留当前块的摘要/结论
2. 删除原始详细内容
3. 用"暂存区"存中间结果
4. 最后一步再整合

## 补充机制 (老专家审核后新增)

### 1. 全局暂存区 (Scratchpad)

> Gemini 2.5 Pro 建议：存放跨步骤的关键信息，避免压缩后丢失

```
~/.solar/scratchpad/
├── task_xxx_step1.md   # 第一步结果
├── task_xxx_step2.md   # 第二步结果
└── task_xxx_summary.md # 最终整合
```

### 2. 错误处理与回滚

> 两位专家都提到：定义失败后的重试、修正策略

```
执行失败
    │
    ▼
重试当前步骤 (最多2次)
    │
    ├── 成功 → 继续
    │
    └── 失败 → 回退到上一步规划
              │
              └── 重新规划该步骤
```

### 3. 规划确认机制

> DeepSeek R1 建议：规划需用户确认

复杂任务规划后，输出给用户确认：
```
【规划确认】这个任务我打算这样做：
1. 先读前20页...
2. 然后总结...
3. ...

昊哥，这样可以吗？
```

### 4. 非文档任务适用

> DeepSeek R1 指出：不只是PDF，代码库分析、多文件处理也适用

| 任务类型 | 分步策略 |
|----------|----------|
| PDF/文档 | 按页/章节分块 |
| 代码库 | 按目录/模块分块 |
| 多文件 | 按文件逐个处理 |
| API调用 | 按接口分批 |
