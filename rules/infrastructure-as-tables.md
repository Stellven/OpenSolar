# Solar 铁律: 基础设施即系统表

> Infrastructure as System Tables (IaST)

## 核心理念

**一切可查询、可演化、可自省的系统能力，都应该建模为系统表。**

```
┌─────────────────────────────────────────────────────────────┐
│                    传统方式 vs IaST                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ❌ 传统: 硬编码配置                                        │
│   ┌──────────────────────────────────────────────────────┐ │
│   │  config.json → 静态配置                               │ │
│   │  if/else     → 硬编码路由                             │ │
│   │  logs/       → 只写日志                               │ │
│   └──────────────────────────────────────────────────────┘ │
│                          │                                  │
│                          ▼                                  │
│   ✅ IaST: 系统表驱动                                       │
│   ┌──────────────────────────────────────────────────────┐ │
│   │  sys_resources    → 资源可查询                        │ │
│   │  sys_routing_*    → 路由可配置                        │ │
│   │  sys_stats_*      → 统计可分析                        │ │
│   │  sys_evolution_*  → 演进可追溯                        │ │
│   │  v_*              → 视图即洞察                        │ │
│   └──────────────────────────────────────────────────────┘ │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 五层实现模式

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 5: High-Level Interface (高级接口)                    │
│  ┌─────────────────────────────────────────────────────────┐│
│  │  SystemCatalog.selectModel({ complexity: 'high' })      ││
│  │  SystemCatalog.getHealthReport()                        ││
│  └─────────────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────────────┤
│  Layer 4: TypeScript API (程序接口)                         │
│  ┌─────────────────────────────────────────────────────────┐│
│  │  MetadataManager.registerResource()                     ││
│  │  MetadataManager.recordInvocation()                     ││
│  └─────────────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────────────┤
│  Layer 3: Views (视图即洞察)                                │
│  ┌─────────────────────────────────────────────────────────┐│
│  │  v_resource_health, v_quota_status, v_evolution_*       ││
│  └─────────────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────────────┤
│  Layer 2: Triggers (自动化)                                 │
│  ┌─────────────────────────────────────────────────────────┐│
│  │  tr_invocation_stats, tr_quota_check, tr_evolution_*    ││
│  └─────────────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────────────┤
│  Layer 1: Tables (数据结构)                                 │
│  ┌─────────────────────────────────────────────────────────┐│
│  │  sys_resources, sys_agents, sys_skills, sys_models...   ││
│  └─────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────┘
```

## 实施检查清单

设计任何 Solar 基础设施时，必须回答：

| 检查项 | 问题 | 示例 |
|--------|------|------|
| **可查询** | 系统能自省这个能力吗？ | `SELECT * FROM sys_agents` |
| **可演化** | 能基于历史数据自动优化吗？ | `v_evolution_candidates` |
| **可追溯** | 变更历史可审计吗？ | `sys_evolution_log` |
| **可路由** | 决策逻辑可配置吗？ | `sys_routing_model` |
| **有视图** | 有即时洞察吗？ | `v_resource_health` |
| **有触发器** | 有自动化反应吗？ | `tr_quota_check` |

## 应用场景

### 1. Agent Memory System

```sql
-- 不是: memory.json
-- 而是:
sys_memories         -- 记忆存储
sys_memory_links     -- 动态链接
sys_memory_graph     -- 图结构
v_memory_relevance   -- 相关性视图
tr_memory_evolution  -- 演化触发器
```

### 2. Skill Registry

```sql
-- 不是: skills/*.md 散落
-- 而是:
sys_skills           -- 技能注册
sys_skill_stats      -- 使用统计
v_skill_ranking      -- 排行视图
tr_skill_preference  -- 偏好学习
```

### 3. Cost Management

```sql
-- 不是: if (cost > threshold) switch_model()
-- 而是:
sys_quotas           -- 配额配置
sys_stats_daily      -- 日统计
v_quota_status       -- 状态视图
v_recommended_model  -- 自动推荐
tr_quota_check       -- 超限告警
```

## 收益

| 维度 | 传统方式 | IaST 方式 |
|------|---------|----------|
| 可观测性 | 日志挖掘 | SQL 查询 |
| 可配置性 | 重新部署 | UPDATE 语句 |
| 可演化 | 手动调优 | 自动演进 |
| 可审计 | 无 | 完整历史 |
| 可扩展 | 代码修改 | 添加行 |

## 铁律总结

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   🔥 IaST 铁律                                              │
│                                                             │
│   1. 能力 = 表      (sys_*)                                 │
│   2. 洞察 = 视图    (v_*)                                   │
│   3. 自动 = 触发器  (tr_*)                                  │
│   4. 决策 = 路由表  (sys_routing_*)                         │
│   5. 历史 = 日志表  (sys_*_log)                             │
│                                                             │
│   一切皆可查询，一切皆可演化。                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
