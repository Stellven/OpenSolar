# Solar 铁律: 借鉴前必须验证 (Borrow Before Verify)

> **来源: 2026-02-06 OpenViking 借鉴失败的教训**
> **问题: 看到好文章就照搬，4个功能只有2个真正有用**

## 铁律定义

```
┌─────────────────────────────────────────────────────────────────┐
│                 BORROW BEFORE VERIFY                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   借鉴任何外部设计前，必须:                                     │
│                                                                 │
│   1. 分析是否解决 Solar 的真实问题                              │
│   2. 检查是否与现有功能重复                                     │
│   3. 评估实现成本 vs 收益                                       │
│   4. 设计如何集成到现有系统                                     │
│                                                                 │
│   ❌ 禁止: 看到好文章就全盘接收                                 │
│   ❌ 禁止: 写完代码不集成就宣布完成                             │
│   ❌ 禁止: 不分析就说"都要"                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 本次教训

### 错误做法

```
看到 OpenViking 文章
    ↓
"这四个功能都很好！"
    ↓
"都" → 全部实现
    ↓
写了 4 个文件，0 个集成
    ↓
监护人问: "你咋知道是好的？"
    ↓
才发现 2/4 是重复/过度设计
```

### 正确做法

```
看到 OpenViking 文章
    ↓
Step 1: 这些功能解决什么问题？
    ↓
Step 2: Solar 有这些问题吗？
    • L0/L1 省 token → Solar 只有 99 资源，不急
    • URI 统一寻址 → 现有 ID 系统够用
    • 递归检索 → TieredRouter 已有
    • 会话提取 → 这个确实没有，有价值
    ↓
Step 3: 与现有系统重复吗？
    • URI → 与 sys_*.id 重复
    • 递归 → 与 TieredRouter 重复
    ↓
Step 4: 结论
    只借鉴 1-2 个真正需要的功能
```

## 借鉴检查清单

每次看到好文章/好设计，必须回答：

- [ ] 这解决了 Solar 的什么痛点？（具体场景）
- [ ] 现有系统有类似功能吗？（查 sys_* 表和现有代码）
- [ ] 实现成本是多少？（开发时间 + 维护成本）
- [ ] 收益是多少？（省多少 token/时间/钱）
- [ ] 如何集成到现有系统？（具体代码路径）
- [ ] 如何验证效果？（测试方案）

**如果任何一项答不上来，就不应该实现。**

## 具体案例

| 外部设计 | 检查结果 | 决定 |
|----------|----------|------|
| L0/L1 分层 | 资源 99 个，收益小 | 延迟 |
| URI 寻址 | 与 ID 系统重复 | 不做 |
| 递归检索 | 与 TieredRouter 重复 | 不做 |
| 会话提取 | 真的没有，有价值 | 做 |

## 违反后果

```
本次违反的代价:
• 浪费时间写了 4 个文件
• 2 个需要删除 (uri-resolver.ts, recursive-retriever.ts)
• 1 个需要重写 (session-extractor.ts)
• 只有 1 个勉强保留 (context-layers.ts)

效率: 25% (1/4)
```

## 智慧根基

```
实事求是 - 从 Solar 实际需求出发，不是从外部文章出发
中庸之道 - 不要过度工程，够用就好
知行合一 - 借鉴后必须验证效果，不是写完就完
```
