# Solar 铁律: 约束满足验证

> **来源**: Meta-Harness Iter-005 — SE-009(7/10) 失败分析
> **核心问题**: 多约束问题时，回答方案后未逐条验证约束是否满足

## 铁律定义

当问题包含 **2 个以上显式约束** 时，回答必须包含约束验证清单。

### 1. 约束提取（回答前先列出）

```
从题干中提取所有约束，编号列出：
C1: [约束1]
C2: [约束2]
C3: [约束3]
...
```

### 2. 方案给出后，逐条验证

```
验证清单：
✓ C1: [方案如何满足C1，具体证据]
✓ C2: [方案如何满足C2，具体证据]
✗ C3: [方案无法满足C3，原因和替代方案]
```

### 3. 调度/排列类问题的强制步骤

| 步骤 | 要求 |
|------|------|
| 1. 冲突图 | 列出资源冲突矩阵：谁和谁共享资源（人/房间/设备），不能同时占用 |
| 2. 时序约束 | 单独标注所有 "A必须在B之前/之后" 的顺序约束，在方案中逐条验证 |
| 3. 排序策略 | 说明为什么按这个顺序安排（如关键路径最长优先、约束最多优先） |
| 4. 方案 | 具体到时间/位置/编号，形成时间表 |
| 5. 逐条验证 | 每条约束 ✓/✗ + 具体证据（如 "M1在R1 09:00-10:00，M3在R2 10:00-11:00 → 时序OK ✓"） |
| 6. 无解分析 | 如果约束冲突，说明为什么无解，哪些约束互相矛盾 |

### 3.1 时序约束检测模式 (v1.1 新增)

```
检测词: "必须在X之前/之后" / "先X再Y" / "X依赖Y的结论" / "X完成后再Y"
处理: 时序约束必须单独列出，在方案验证时用时间戳比较

❌ 错误: 忽略 "M1必须在M3之前" 的约束
✅ 正确: 提取 C_时序: M1必须在M3之前
         方案中验证: M1=09:00-10:00, M3=10:00-11:00 → 10:00 ≥ 10:00 ✓
```

### 3.2 冲突矩阵模式 (v1.1 新增)

```
示例: 5个会议，3个房间
人员冲突矩阵（不能同时参加的会议对）:
- Alice: M1, M3, M4 → (M1,M3), (M1,M4), (M3,M4) 不能同时
- Bob: M1, M2, M5 → (M1,M2), (M1,M5), (M2,M5) 不能同时
- Charlie: M2, M3 → (M2,M3) 不能同时
- Dave: M4, M5 → (M4,M5) 不能同时

→ 生成冲突图后，不冲突的会议可并行安排在同一时间段（不同房间）
```

### 4. 正反模式

```
❌ 错误: 给出方案后直接结束，不验证约束
❌ 错误: "显然满足所有约束"（没有具体证据）
❌ 错误: 遗漏某个约束不验证

✅ 正确: 每条约束都有具体的验证证据
✅ 正确: 如"M1在R1 09:00-10:00，M3在R2 10:00-11:00 → M1在M3之前 ✓"
✅ 正确: 无法满足的约束明确标注并给出理由
```

## 触发条件

当问题包含以下特征时自动触发：
- "约束"/"条件"/"要求"/"必须"/"不能"
- 调度/排列/分配/安排类问题
- 同时存在 2+ 个限制条件
- "满足所有"/"不能违反"
