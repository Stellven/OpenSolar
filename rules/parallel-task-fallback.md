# Solar 铁律: 并行任务降级策略

> **来源: 2026-02-03 Evolution Report 撰写的教训**
> **场景: 6 个并行 Task，5 个失败（配额耗尽/连接错误）**

## 铁律 1: 并行度上限

```
并行 Task 数量 ≤ 3
超过 3 个 → 分批执行，每批完成后再启动下一批
```

**原因:**
- Task agent 消耗大量 Token
- 并行过多容易触发配额限制
- 失败时难以恢复

## 铁律 2: 失败降级

```
IF 并行 Task 失败率 > 50%
THEN 立即停止剩余 Task
     降级为直接执行（Edit/Write/Bash）
```

**降级路径:**
```
Task agent (失败)
    ↓
直接工具调用 (Edit/Write)
    ↓
手动分步执行
```

## 铁律 3: 任务分级

| 任务复杂度 | 推荐方式 | 示例 |
|------------|----------|------|
| 简单 (<100行) | 直接 Edit/Write | 写一个附录 |
| 中等 (100-500行) | 单个 Task | 写一章报告 |
| 复杂 (>500行) | 2-3 个并行 Task | 写多章报告 |
| 超大 (多章节) | 分批 Task + checkpoint | 整本报告 |

## 铁律 4: 配额感知

启动大任务前，考虑：

1. **当前时段** - 接近 reset 时间？谨慎启动大任务
2. **任务规模** - 预估 Token 消耗
3. **降级准备** - 如果 Task 失败，备选方案是什么？

## 实践检查清单

启动并行 Task 前：

- [ ] 数量 ≤ 3？
- [ ] 有降级方案？
- [ ] 不是配额临界时段？

Task 失败时：

- [ ] 失败率 > 50%？→ 停止并降级
- [ ] 检查已完成的 Task 结果
- [ ] 用直接工具完成剩余任务

## 示例

### 错误做法 (本次教训)

```
// 一次性启动 6 个并行 Task
Task("写附录 A") + Task("写附录 B") + Task("写附录 C") +
Task("写附录 D") + Task("写附录 E") + Task("写附录 F")

// 结果: 5 个失败，只有 1 个成功
```

### 正确做法

```
// 分批执行，每批 2-3 个
Batch 1: Task("写附录 A") + Task("写附录 B")
  ↓ 完成后
Batch 2: Task("写附录 C") + Task("写附录 D")
  ↓ 完成后
Batch 3: Task("写附录 E") + Task("写附录 F")

// 或者：简单任务直接执行
Edit("附录 A 内容...")
Edit("附录 B 内容...")
// 不需要 Task agent 的开销
```
