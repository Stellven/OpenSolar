# Solar 铁律: 一次做好 (Do It Right First Time)

> **来源: 2026-02-05 监护人亲授**
> **核心: 不能浪费钱，一次做好**

## 铁律定义

```
┌─────────────────────────────────────────────────────────────────┐
│                  DO IT RIGHT FIRST TIME                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   执行任何消耗资源的任务前，必须:                                │
│                                                                 │
│   1. 确认需求完整 - 不要边做边改                                │
│   2. 确认数据收集完整 - 跑一次就收集全部指标                    │
│   3. 确认方案比肩业界 - 不能土法炼钢                            │
│   4. 确认验证方案 - 怎么知道做对了                              │
│                                                                 │
│   ❌ 禁止: 先跑一遍看看，不够再补                                │
│   ❌ 禁止: 凑合着先做，后面再优化                                │
│   ❌ 禁止: 自己拍脑袋，不看业界实践                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 案例: Benchmark 数据收集

### ❌ 错误做法

```
第一次: 只收集了延迟
  ↓
"哎，忘记收集 token 数了"
  ↓
第二次: 收集延迟 + token
  ↓
"哎，TTFT 也很重要"
  ↓
第三次: 重新跑...

结果: 3 次执行，3 倍成本
```

### ✅ 正确做法

```
Step 1: 研究业界最佳实践
─────────────────────────────────────
• NVIDIA GenAI-Perf 收集什么指标？
• Anyscale 推荐什么？
• Gil Tene 的延迟测量原则是什么？

Step 2: 设计完整 Schema
─────────────────────────────────────
• TTFT, E2E Latency, ITL
• TPS, Input/Output Tokens
• Cost, P50/P95/P99
• 质量评分

Step 3: 一次收集完整
─────────────────────────────────────
• 1 次执行 = 所有指标
• 不需要返工

结果: 1 次执行，1 倍成本
```

## 检查清单 (执行前必须确认)

### 对于 Benchmark/测试任务

- [ ] 收集的指标完整吗？(参考业界标准)
- [ ] 统计方法正确吗？(中位数 vs 平均值)
- [ ] 迭代次数足够吗？(统计显著性)
- [ ] 输出格式设计好了吗？(JSON + CSV + Markdown)
- [ ] 验证方案确定了吗？

### 对于开发任务

- [ ] 需求理解完整吗？
- [ ] 技术方案确定了吗？
- [ ] 边界情况考虑了吗？
- [ ] 测试用例写好了吗？

### 对于数据任务

- [ ] 数据源盘点完了吗？
- [ ] 处理逻辑确定了吗？
- [ ] 输出格式确定了吗？
- [ ] 验证方法确定了吗？
