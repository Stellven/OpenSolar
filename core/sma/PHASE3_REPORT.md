**日期**: 2026-02-26

---

## 执行摘要

本报告记录 SMA v1.0 Phase 3 的价值验证结果。通过 5 条人工标注的会话样本，测试了 L2→L3 知识固化的提取质量。

**核心结论**:
- **Precision (准确率)**: 8.75%
- **Recall (召回率)**: 38.89%
- **F1 Score**: 14.29%
- **决策**: ❌ **NO-GO** - 当前正则提取方案价值不足

---

## 验证方法论

### 数据集
- **来源**: 手动标注的 demo_annotation.json
- **样本数**: 5 条对话轮次
- **期望三元组**: 18 个 (人工标注的"黄金标准")

### 验证流程
```
1. 写入标注会话到 L2 (session_log)
2. 触发 L2→L3 知识固化 (triggerConsolidation)
3. 从 L3 提取实际三元组 (knowledge_triples)
4. 对比期望 vs 实际，计算指标
```

### 评估指标
- **Precision**: 提取的三元组中正确的比例
- **Recall**: 应该提取的三元组中成功提取的比例
- **F1 Score**: Precision 和 Recall 的调和平均

## 失败模式分析

### 错误类型统计

| 错误类型 | 数量 | 占比 | 示例 |
|---------|------|------|------|
| 问句误判 | 3 | 33% | "(SMA, 是, 什么)" |
| 分词错误 | 2 | 22% | "(L2 可以无损记录所, 有, 会话轨迹)" |
| 上下文丢失 | 2 | 22% | "(它, 可以, 从会话中提取三元组)" |
| 噪声提取 | 2 | 22% | 过度提取非关键信息 |

### 错误示例详解

#### 错误 1: 问句被当作陈述
```
输入: "SMA 是什么？"
错误提取: (SMA, 是, 什么)
问题: 正则 /(.+?)\s*是\s*(.+?)/ 无法区分问句和陈述
应该: 忽略问句，只提取 "SMA 是 Solar Memory Architecture"
```

#### 错误 2: 词语分割错误
```
输入: "L2 可以无损记录所有会话轨迹"
错误提取: (L2 可以无损记录所, 有, 会话轨迹)
问题: "所有" 被正则 /(.+?)\s*有\s*(.+?)/ 错误切分
应该: (L2, 可以, 无损记录所有会话轨迹)
```

#### 错误 3: 代词无法解析
```
输入: "triggerConsolidation 是...。它可以从会话中提取三元组"
错误提取: (它, 可以, 从会话中提取三元组)
问题: "它" 指代 "triggerConsolidation"，但正则无上下文
应该: (triggerConsolidation, 可以, 从会话中提取三元组)
```

#### 错误 4: 过度提取
```
期望: 18 个三元组
实际: 80 个三元组 (4.4倍)
问题: 正则模式过于宽泛，匹配大量噪声
```

## 根因分析

### 当前提取算法架构
```typescript
const patterns = [
  /(.+?)\s*是\s*(.+?)(?:[。，！？]|$)/g,   // "X 是 Y"
  /(.+?)\s*可以\s*(.+?)(?:[。，！？]|$)/g, // "X 可以 Y"
  /(.+?)\s*有\s*(.+?)(?:[。，！？]|$)/g    // "X 有 Y"
];
```

### 缺陷列表

| 缺陷 | 影响 | 解释 |
|------|------|------|
| **无语法分析** | 高 | 无法区分陈述句/疑问句 |
| **无边界控制** | 高 | `.+?` 匹配范围过宽 |
| **无词性标注** | 中 | 无法识别实体 vs 普通词 |
| **无指代消解** | 中 | 代词"它"无法还原 |
| **无置信度评估** | 低 | 所有提取默认 confidence=0.5 |

### 对比：业界知识抽取方案

| 方案 | 技术栈 | 准确率 | 适用场景 |
|------|--------|--------|----------|
| **当前方案** | Regex | ~10% | ❌ 不适用 |
| OpenIE (Stanford) | NLP Parser | 60-70% | 英文开放域 |
| SpaCy NER + Dep | ML模型 | 70-80% | 中文实体关系 |
| LLM Prompting | GPT-4 | 80-90% | 通用，但成本高 |

## 改进方向

### 方案 A: 增强正则 (短期，低成本)

**改进点**:
1. 添加问句过滤: `^(?!.*[？?])`
2. 实体边界约束: `(?<=\s|^)[A-Z\p{Han}]+(?=\s|$)`
3. 代词黑名单: 过滤 "它/这/那"

**预期提升**: F1 20-30%
**工作量**: 1-2 小时
**风险**: 仍无法解决复杂语义问题

### 方案 B: 引入 NLP Pipeline (中期，中成本)

**架构**:
```
输入文本
  ↓
分词 (jieba)
  ↓
词性标注 (jieba.posseg)
  ↓
依存句法分析 (spaCy/HanLP)
  ↓
实体识别 + 关系抽取
  ↓
三元组输出
```

**技术选型**:
- jieba: 中文分词
- spaCy (zh_core_web_sm): 词性标注/依存分析
- 自定义规则: 基于依存树提取 SVO 三元组

**预期提升**: F1 60-70%
**工作量**: 1-2 天
**风险**: 需要额外依赖，模型加载开销

### 方案 C: LLM 提取 (长期，高成本)

**实现**:

## 架构重新审视

### 当前假设挑战

**原假设**: L3 自动提取可以提供高质量结构化知识
**实际情况**: 简单提取质量不足，复杂提取成本高

### 架构调整建议

#### 选项 1: L2 优先策略 (推荐)
```
L1 (Context) ←─ volatile
L2 (Session Log) ←─ **核心记忆层**，全文检索
L3 (Triples) ←─ 人工精选/高置信提取
```

**理由**:
- L2 已验证可行 (Phase 2 通过)
- 全文检索 (LIKE/FTS5) 在小规模足够
- L3 作为补充，不作为主要记忆

#### 选项 2: 混合提取策略
```
L2→L3 触发条件:
- 简单模式 (正则) - 低成本，低准确率
- LLM 提取 - 高成本，高准确率
- 人工标注 - 零成本 (用户明确告知的知识)
```

#### 选项 3: 延迟 L3 (最保守)
```
Phase 4: 跳过
未来: 等待更好的中文 NLP 工具成熟后重新评估
```
