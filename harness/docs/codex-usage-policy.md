# Codex 调用分级策略

> codex 额度有限，只用刀刃上。不用于常规编码。

## S 级 — 必调 (上限 4000 out-tokens)

codex 是这类任务的**最佳选择**，投入产出比最高。

### 触发场景
1. 审判官同一 Sprint 连续 FAIL >= 2 轮 → 自动以 S 级调 codex 做根因梳理
2. 架构方案 A vs B 对比决策，需要深度权衡分析
3. 复杂算法设计：并发协议、状态机、分布式一致性
4. 安全审计中发现的深层逻辑漏洞分析
5. 跨模块重构的影响范围评估 (blast radius)

### 反例 (看似 S 级但不是)
1. ~~"这个代码风格不统一"~~ → 建设者用 lint 修
2. ~~"这个函数太长需要拆分"~~ → 建设者重构
3. ~~"这个 bug 复现不了"~~ → 审判官分析，不到 2 轮不调 codex

## A 级 — 可调 (上限 2000 out-tokens)

需要 codex 的第二意见，但不紧急。

### 触发场景
1. 跨模块逻辑梳理：理解 A 模块修改对 B 模块的影响
2. 疑难 bug 根因：单轮排查未果，需要换个视角
3. 规划者写大合约前的 Done 定义 sanity check
4. 性能优化方案的 ROI 排序建议
5. 新技术选型的 trade-off 分析

### 反例 (看似 A 级但不是)
1. ~~"这个 API 怎么用"~~ → 查文档 / 知识库
2. ~~"这个测试怎么写"~~ → 建设者写
3. ~~"这个配置项什么意思"~~ → 读代码 / grep

## B 级 — 禁调

这些任务交给建设者/审判官/GLM，不浪费 codex 额度。

### 典型场景
1. CRUD 操作 / 格式化 / 命令执行
2. 单元测试生成 / 简单脚本
3. 已有知识库答案的问题
4. 代码格式化 / lint 修复
5. 文件重命名 / 目录调整
6. 配置文件修改
7. 常规代码质量评估 (审判官用 GLM-5.1 做)

### 误触发识别
1. "分析一下代码" → 先看是架构级分析(S)还是代码风格(A/B)，按实际分级
2. "优化一下" → 简单优化是 B，性能瓶颈分析是 A，架构级优化是 S
3. "写个测试" → B 级，除非是复杂的集成测试设计 (可能 A)

## 策略执行

调用前由 `call_codex()` 做 tier 校验：

```
tier=B → REJECTED_BY_POLICY (不计入预算)
tier=A → 预算检查 → 通过则执行
tier=S → 预算检查 → 通过则执行
```

所有调用记录到 `codex-bridge/ledger.jsonl`。
