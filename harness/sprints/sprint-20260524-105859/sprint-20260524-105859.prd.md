# P0 PRD — Knowledge Ingest Dispatcher 统一控制面

## 背景

知识库当前已经具备 raw 落盘、QMD 索引、ThunderOMLX 抽取、Obsidian vault、Tech Hotspot/AI Influence 等多条能力，但这些能力不是由统一控制面调度，而是由各入口脚本各自串联。

这导致三个系统性问题：

1. 入口没有统一：raw、vault、accepted、YouTube、ChatGPT、Web、GitHub、Solar 产物可能各走各的。
2. 状态没有统一：manifest/log/report/QMD 状态无法稳定对账。
3. 抽取产物没有强契约：ThunderOMLX 直接输出 Markdown，容易 validation_failed。

## 用户问题 / Problem

知识库使用者（昊哥本人 + 后续协作者）今天面对的具体痛点：

1. **无法稳定回答 "这条知识在不在库里"** — 同一篇 ChatGPT/YouTube/Solar accepted 产物可能被两条不同链路各自摄入一次，状态不可对账，重复抽取浪费 ThunderOMLX 配额。
2. **无法稳定回答 "这条知识抽取失败的原因"** — 当前各 adapter 自带 log，失败原因散落在 stderr/manifest 注释里，无法按 source_kind / failure_class 聚合归因。
3. **无法稳定重抽** — 想对某一批 source（例如最近 7 天 YouTube transcript）整体重抽时，没有统一 watermark/registry 查询面，只能手工 grep 文件。
4. **回答时无法保证 grounding** — extracted 产物有时被直接当作事实回答，缺少回源 raw/vault span 的强约束，模型会基于 "L2 加工后的二手陈述" 反向编造证据。
5. **无法在故障期主动停手** — 当 ThunderOMLX 出现连续 validation_failed（schema 漂移或模型退化）时，没有 circuit breaker，会把坏数据继续灌进 QMD extracted index。

## 用户故事 / User Stories

- **As 知识库维护者**, I want 一条统一 ingest 命令把任意 source（raw 文件、Obsidian note、accepted artifact、YouTube transcript、ChatGPT/Web/GitHub report）丢进 dispatcher, so that 我不必记忆每个 source_kind 对应哪个脚本，也不会出现入口绕过 registry。
- **As 知识库维护者**, I want 一个 registry 查询命令告诉我 "近 24h 内 source_kind=youtube_transcript 的 100 个事件中各状态分布", so that 我能在不读 log 的情况下判断这一批是否健康。
- **As 抽取系统运维**, I want validation_failed 二次失败的 source 被自动 quarantine 且不进入 QMD extracted index, so that 坏数据不会污染语义路由层。
- **As 抽取系统运维**, I want 当 rolling 25 任务中 validation_failed > 25% 或连续 5 次失败时 dispatcher 暂停 extract 队列, so that ThunderOMLX 退化不会被静默放大。
- **As 知识库使用者**, I want 任何由 extracted 触发的回答都必须给出回源 raw/vault span, so that 我能验证模型没有基于二手 paraphrase 编造证据。
- **As Planner**, I want PRD 把 9 个功能 (F1-F9) 拆成可独立交付的 DAG 节点, so that 我能并行调度 builder 而不踩交叉依赖。

## 用户目标

让 Solar 知识库进入可观测、可回滚、可重抽、可证据落地的稳态闭环。

## 核心原则

- QMD 是确定性索引面，不替换为 ThunderOMLX。
- ThunderOMLX 是 L2 语义加工 worker，不是事实源。
- raw/vault 是 grounding source，extracted 是 semantic routing material。
- registry 是状态事实源，manifest 是导出物。
- 所有入口必须走 dispatcher。
- JSON-first，Markdown 由程序渲染。

## 功能需求

### F1 — Unified Ingest Event

所有 adapter 提交统一事件：

```json
{
  "event_id": "evt_...",
  "source_kind": "obsidian_vault",
  "source_path": "/Users/lisihao/Knowledge/concepts/qmd-indexing.md",
  "source_adapter": "obsidian_adapter",
  "content_kind": "markdown",
  "declared_doc_type": "concept",
  "ingest_policy": "default",
  "extract_policy": "default"
}
```

### F2 — Registry

实现 `Knowledge/_registry/knowledge_ingest.sqlite`，提供状态查询、审计、watermark、失败归因。

### F3 — Source Adapters

一期必须支持：

- raw adapter
- obsidian vault adapter
- accepted artifact adapter
- youtube transcript adapter
- chatgpt/web/github report adapter

### F4 — State Machine

必须支持状态：

```text
NEW
SOURCE_CAPTURED
RAW_MATERIALIZED
VAULT_DISCOVERED
SPAN_MAPPED
RAW_OR_VAULT_QMD_INDEX_PENDING
RAW_OR_VAULT_QMD_INDEXED
EXTRACT_ELIGIBLE
THUNDEROMLX_EXTRACT_PENDING
THUNDEROMLX_EXTRACT_RUNNING
EXTRACT_CANDIDATE_GENERATED
EXTRACT_VALIDATION_PENDING
EXTRACTED_VALIDATED
EXTRACTED_QMD_INDEX_PENDING
EXTRACTED_QMD_INDEXED
DONE
VALIDATION_FAILED
REPAIR_PENDING
QUARANTINED
```

### F5 — JSON-First Extraction

ThunderOMLX 输出 `extracted.candidate.json`，程序校验后生成 `*.extracted.md`。

### F6 — Validation / Repair / Quarantine

最多 repair 一次，二次失败 quarantine。quarantine 不进入 QMD extracted index。

### F7 — Circuit Breaker

rolling 25 个任务中 validation_failed > 25% 或连续失败 >= 5 时暂停抽取队列。

### F8 — QMD Microbatch Watermarks

显示 raw/vault/extracted 三层水位。

### F9 — Query Grounding

extracted 只能路由，回答证据必须回源 raw/vault/source span。

## 验收标准

验收以 contract D1-D10 为准。最小样本：

- raw/chatgpt: 20
- raw/youtube: 20
- raw/solar: 20
- vault/concepts: 20
- vault/references: 20
- vault/synthesis: 20

样本通过率：

- validation pass >= 90%
- repair 后终态 failed <= 5%

## 非目标

- 不改 QMD embedding 模型。
- 不改 Obsidian 原文。
- 不做全量历史 backfill，直到样本验收通过。
- 不把 RAGFlow 或 ThunderOMLX 变成 source of truth。

## 约束 / Constraints

- **运行环境**: Mac mini M4 (macOS arm64), bash 5.3.9, Python 3.11 venv，产出禁止落 `/tmp`，必须落 `~/Knowledge/_registry/` 或 sprint artifact 目录。
- **存储事实源**: `Knowledge/_registry/knowledge_ingest.sqlite` 是唯一状态事实源；manifest/log/report 是导出物，禁止反向写回 registry。
- **现有 API 不破坏**: QMD index 接口、ThunderOMLX extract worker 调用契约、Obsidian wiki 现有 `_raw/solar-harness/...` 路径布局不得变更。
- **JSON-first**: ThunderOMLX 必须先输出 `extracted.candidate.json`（带 schema 版本号），renderer 程序生成 `*.extracted.md`；禁止直接发布模型生成的 Markdown。
- **Repair 上限**: 任意 source_id 最多 repair 一次，二次 validation_failed 直接进入 QUARANTINED 终态。
- **Circuit breaker 触发阈值**: rolling 25 任务 validation_failed > 25% 或连续 ≥ 5 次失败时必须暂停 extract 队列，恢复需要人工 `dispatcher resume`。
- **Grounding 强约束**: extracted 内容只允许作为 routing material；回答必须回源 raw/vault/source span，否则禁止出回答。
- **样本验收门禁**: 在 6 类样本 (raw/chatgpt、raw/youtube、raw/solar、vault/concepts、vault/references、vault/synthesis) 各 ≥20 条且 validation pass ≥ 90% 之前，不得启动历史 backfill。
- **能力边界**: 不引入新 embedding 模型、不替换 QMD、不修改 Obsidian 原文、不把 RAGFlow/ThunderOMLX 升级为事实源（与"非目标"互锁）。

## 风险

| 风险 | 影响 | 缓解 |
|---|---|---|
| JSON schema 太复杂 | 模型输出不稳 | 小 schema + renderer |
| registry 与 manifest 不一致 | 状态不可信 | registry 为唯一事实源 |
| QMD 长批次滞后 | 状态误判 | 微批水位 |
| Obsidian 图谱未解析 | 召回质量不足 | wikilinks/tags metadata |
| backfill 再次失控 | 浪费算力 | circuit breaker |

## Planner Handoff

Planner 必须把实现拆成独立 DAG 节点，优先级：

1. Registry schema + migration.
2. Event model + dispatcher CLI.
3. Raw/vault discovery + span sidecar.
4. JSON-first extractor + renderer + validator.
5. Circuit breaker + QMD watermarks.
6. Health/audit/report.
7. Sample migration and verification.
