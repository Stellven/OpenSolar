## 技术方案

### D1: 合约 Anatomy 升级

**contract-template-v2.md** 5 段结构:

```markdown
---
name: <sprint-name>
description: <需求摘要>
triggers: [<触发关键词>]
---

## When to Use
✓ 使用: (≥3条)
✗ 不使用: (≥3条)

## Process
1. [步骤1]
2. [步骤2]
...

## Red Flags
- ❌ 无 mock/模拟/TODO/FIXME
- ❌ 无硬编码路径/密钥
- ❌ 无超出范围的改动

## Verification Gates
- [ ] D1: (条件)
  verify:
    cmd: "grep -c '^## ' <file>"
    expected_exit: 0
    output_pattern: "[5-9]"
```

**solar-harness 修改**: `new_sprint()` 中将 `cat > contract.md` 的 heredoc 替换为读取 `contract-template-v2.md` + 填入变量。

关键: 向前兼容 — 旧 sprint 没有 `---` frontmatter 时, 审判官/协调器代码不 break。

### D2: 7 阶段 Sprint 生命周期

**phase-state-machine.sh** 核心接口:

```bash
phase_init <sid>          # 在 status.json 设 phase=spec
phase_current <sid>       # 输出当前 phase
phase_transition <sid> <from> <to>  # 阶段转换
phase_entry_gate <sid> <phase>      # 入口检查, 返回 0/1
phase_exit_gate <sid> <phase>       # 出口检查, 返回 0/1
phase_list               # 列出所有合法 phases
```

**合法转换图**:
```
spec → plan → build → test → review → ship
                              ↑          ↓
                              └── fail ──┘
```

**entry_gate 逻辑**:
- spec→plan: 合约 Done 已填写 (非 "(条件1)")
- plan→build: plan.md 存在
- build→test: handoff.md 存在
- test→review: eval.md 存在
- review→ship: eval verdict=PASS
- review→build (fail loop): eval verdict=FAIL, round < 3

**status.json 新增字段**: `"phase": "spec"`, 缺失时默认 `"legacy"`。

**bash 3.2 兼容**: 不用 `declare -A`, 用 case/esac 做状态转换验证。

**events.jsonl**: 每次成功 transition 追加 `phase_transition` 事件。

### D3: SDD-CACHE 集成

**目录结构**:
```
~/.solar/harness/cache/
├── metadata.jsonl    # 每行: {key, origin, etag, last_modified, sha256, cached_at, size}
└── bodies/           # 缓存 body 文件, 以 sha256 命名
```

**solar-cache 命令**:

| 子命令 | 功能 |
|--------|------|
| `get <url_or_query>` | 查 metadata → 命中则 revalidate → 返回 body 路径或 miss |
| `put <url_or_query> <body_file>` | 写 body + 更新 metadata |
| `validate` | 扫描所有条目, 验证 origin 是否仍 304 |
| `stats` | 输出 hit_count / miss_count / revalidation_count / total_size |

**Revalidation 逻辑**:
- HTTP URL: `curl -I -H "If-None-Match: <etag>"` → 304 = hit, 200 = miss+更新
- 本地 SQLite: `stat -f %m` (mtime) + `shasum -a 256` 对比 metadata

**约束**: 禁止 TTL, 必须 origin-validated。

**bash 3.2 兼容**: 用 `jq` 做 JSON 操作, 不用 bash 关联数组。

### D4: Verification Gates 显式化

**合约 Done 字段新格式** (YAML 块在 Markdown 中):

```markdown
- [ ] D1: 合约 5 段式
  verify:
    cmd: "grep -c '^## ' ~/.solar/harness/sprints/<sid>.contract.md"
    expected_exit: 0
    output_pattern: "[5-9]"
```

**solar-verify 命令**:
```bash
solar-verify <sid>           # 执行所有 Done 的 verify 块
solar-verify <sid> D1        # 只跑 D1
solar-verify <sid> --json    # JSON 输出
```

**流程**: 读取合约 → 提取 verify 块 → 用 `sed`+`awk` 解析 cmd/expected_exit/output_pattern → 逐个执行 → 打印 PASS/FAIL 矩阵。

**解析策略**: 用 `awk` 按 `verify:` 块提取, 不依赖 yq/yaml 解析器 (bash 3.2 兼容)。

### D5: Red Flags 内置合约

模板中默认 3 条 red flags:
1. `grep -rE 'TODO|FIXME' <实现文件>` → hit = fail
2. `grep -rE 'mock|stub|模拟|TODO.*实现' <实现文件>` → hit = fail
3. `grep -rE '/tmp/|/var/tmp/' <实现文件>` → hit = fail

审判官在 eval 流程中增加 red_flag 检查步骤。这不需要修改审判官 prompt 大架构, 只需在 `solar-verify` 中加 `--red-flags` 模式。

## 风险点

| # | 风险 | 缓解 |
|---|------|------|
| 1 | bash 3.2 兼容 — macOS `/bin/bash` 没有 `declare -A`, `readarray` | 全部用 case/esac + jq, 测试用 `bash --version` 双版本验证 |
| 2 | 合约模板解析 — YAML 块在 Markdown 中, awk 提取可能脆弱 | 限制格式: verify 块严格 4 空格缩进, cmd 必须单行; 提供解析失败回退 |
| 3 | Cache revalidation 延迟 — HTTP HEAD 可能慢 | 约束要求 < 500ms; 超时则视为 miss, 不阻塞 |
| 4 | 旧 sprint forward-compat — status.json 缺 phase 字段 | 默认 `phase=legacy`, 协调器/审判官代码不依赖 phase 必须存在 |
| 5 | export bundle 体积 — 新增模板+lib 影响不大 | 新增文件 < 50KB, 可忽略 |
| 6 | solar-verify 解析合约 verify 块的边界 — awk 可能误切 | 用 `verify:` 开始 + 下一个 `- [ ]` 或 EOF 结束的 strict delimiter |
