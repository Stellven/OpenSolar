## 1. Problem (Restated)

把 5 个长期分散在 Solar 控制面里的概念正式做成分层架构 + 执行合约：

1. **Physical Operator** = 谁来执行（pane / agent / model+sandbox 组合）
2. **Skill** = 某类任务的可复用执行配方（instructions / scripts / rules / checklist / I-O contract / failure handling / MCP capability requirements）
3. **MCP** = 外部能力与数据接入协议；**不直接等于任务逻辑**
4. **Capsule (Execution Capsule)** = `Skill + MCP bindings + Policy + Contract + Data/resources + Verification rules` 的企业级能力颗粒
5. **Agent Plan Optimizer (APO)** = 在目标 / 成本 / 风险 / 配额下选择 logical operators、skills/capsules、physical operators、MCP capability binding 的组合

要给出：A 四层 registry；B 与 APO/logical operators/evidence ledger 的关系；C 三种 skill 角色；D MCP capability layer + least privilege + policy guard + audit + lease；E operator registry 字段；F optimizer 从 task 到 physical plan 的流程；G MCP 必须经 capability registry + policy guard 的原因；H 首版 rollout=heuristic only；I 正式 PRD/contract/task_graph/handoff；J 不破坏 APO foundation。

## 2. Naming Collision Resolution

## 4. The 4-Tier Registry Architecture

```text
                    Agent Plan Optimizer (APO)
                              │
            ┌─────────────────┼───────────────────┐
            ▼                 ▼                   ▼
       Skill Registry   Execution Capsule    Operator Registry
            │            Registry (ECR)            │
            │                 │                    │
            └──────► MCP Capability Registry ◄─────┘
                     (+ Policy Guard + Lease + Audit)
                              │
                   actual MCP servers / tools
```

Layering rules:

- **Skill Registry** declares pure logic recipes (no infra binding).
- **MCP Capability Registry** declares external capabilities + policy/lease/audit; subset of the existing low-level Capability Registry (namespace `mcp.*`).
- **Execution Capsule Registry** = composition layer: references one or more skills + binds required MCP capabilities + carries Policy/Contract/Data/Verification.
- **Operator Registry** declares what each physical operator (pane/agent/model) has *installed* (skills, mounted MCPs, allowed capsules) and what is *forbidden*.
- **APO** is the consumer of all four registries; it selects (skills, capsule, operator, MCP binding) for each task.

## 5. Three Skill Roles (raw_request item C)

| Role | Definition | Example | Optimizer treatment |
|---|---|---|---|
| **skill-as-macro** | 一组高阶步骤指令；APO 用来 *拆解* 任务，本身不直接执行 | "research_then_summarize" | optimizer 把它展开为子任务，再各自选 operator |
| **skill-as-implementation** | 具体可执行配方（scripts/instructions），由 operator 直接跑 | "research.web.brief.v3" | optimizer 把它绑到 operator + MCP binding，进 physical plan |
| **skill-as-enforcer** | 检查/守门 skill（"在所有 research 输出前必须跑 PII 扫描"） | "guard.pii.scan.v1" | optimizer 把它**强制插入**到匹配触发条件的 capsule 执行前后 |

Skill schema 的 `role` 字段是 `enum`，optimizer 根据 role 走不同分支。

## 6. APO Optimizer Flow (raw_request item F)

```text
task (logical_operator + acceptance + constraints)
  │
  ▼
[1] resolve skill/capsule requirements
  │   - from task.required_skills + logical_operator.preferred_skills
  │   - look up ECR: which execution capsules cover all required skills?
  ▼
[2] candidate operators
  │   - filter Operator Registry by allowed_execution_capsules ⊇ {selected capsule}
  │   - filter by installed_skills ⊇ capsule.skills
  │   - filter by mounted_mcp ⊇ capsule.mcp_bindings && (not in forbidden_mcp)
  ▼
[3] MCP capability binding
  │   - per capsule.mcp_bindings, fetch MCP Capability entries
  │   - apply Policy Guard; reject if policy denies
  │   - issue capability lease (max_seconds, audit hooks)
  ▼
[4] cost / risk / quota scoring (heuristic, v1)
  │   - per (capsule × operator × binding): score = w1*cost + w2*risk + w3*quota_pressure
  │   - no learned model in v1; weights are config
  ▼
[5] selected physical plan
  │   - {operator_id, execution_capsule_id, mcp_binding_ids, lease_ids, evidence_hooks}
  │   - write to evidence ledger as "apo.decision" event
  ▼
graph dispatcher consumes selected plan; runtime claims lease; runs capsule
```

## 7. Why MCP Must Go Through Capability Registry + Policy Guard (item G)

- **Privilege scope drift**: bare MCP server URL ≠ capability — a `web.search` server can also expose `web.scrape` and `web.post`; without registry we can't enforce least privilege per verb.
- **Cross-tenant policy**: same MCP server might be public for skill A and forbidden for skill B; only a policy guard layer can express that conditional.
- **Audit**: evidence ledger needs `(skill_id, capsule_id, mcp_capability_id, lease_id)` tuple. Bypass MCP→registry hop and we lose the join key.
- **Lease/revocation**: long-lived MCP tokens are a liability. Registry-issued lease has TTL and can be revoked; raw token cannot.
- **Substitution**: optimizer needs to swap providers (mcp-server-web-search → mcp-server-tavily) without rewriting skills. Only possible via capability indirection.

These 5 are the formal justifications carried into the ADR.

## 8. Heuristic-First Rollout (item H)

| v1 (this sprint's contract) | v2 (future) |
|---|---|
| Heuristic weighted score (config-driven w1/w2/w3) | Learned optimizer (bandit / RL) |
| Greedy per-task selection | Plan-level joint optimization |
| No exploration | Thompson sampling + kill_criteria |
| Evidence ledger writes decision | Evidence ledger feeds back into learned model |

**v1 hard rule**: no model training, no learned weights, no bandits. Optimizer is deterministic given (registries snapshot + weights). Explicit kill-switch via `OPTIMIZER_MODE=passthrough` (uses old `logical_operator_router` fallback).
