# 基于现有 APO/AQO 方向，正式补一张 strategy sprint：把 Skill、MCP、Capsule、Physical Operator、Opti

## 1. Problem
基于现有 APO/AQO 方向，正式补一张 strategy sprint：把 Skill、MCP、Capsule、Physical Operator、Optimizer 的分层关系做成 Solar 控制面的正式架构和执行合约。核心判断：1. Physical Operator = 谁来执行；2. Skill = 某类任务的可复用执行配方（instructions/scripts/rules/checklist/input-output contract/failure handling/MCP capability requirements）；3. MCP = 外部能力与数据接入协议，不直接等于任务逻辑；4. Capsule = Skill + MCP bindings + Policy + Contract + Data/resources + Verification rules 的

## 12. Capsule Addendum

Planner must read:

- `sprint-20260524-134738.capsule-architecture-addendum.md`

This addendum is normative for the architecture pass and clarifies that:

- Capsule is not Skill
- Capsule is not MCP
- Capsule is not Physical Operator
- Capsule is the higher-order capability unit used by APO for planning/binding/governance

The final planner output must reflect:

1. Capsule Manifest as a formal schema target
2. Effect Model as a mandatory field group
3. Capsule Registry as distinct from Skill/MCP/Operator registries
4. Optimizer binding from logical operators to capsules, then to operators + MCP capability subsets
5. A four-layer IR stack: `TaskIR -> LogicalPlanIR -> CapsulePlanIR -> PhysicalPlanIR`
6. The compiler chain `Logical Operators -> Capsules -> Physical Operators -> Evidence Ledger`
7. Capsule must be modeled as a computation unit with typed I/O, effect semantics, and proof obligations
8. The architecture must introduce `Capsule Planner` and `Capsule Rewriter`
9. Capsule composition must cover type/effect/policy/proof propagation and guard insertion
