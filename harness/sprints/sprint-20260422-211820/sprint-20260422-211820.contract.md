## 需求

## Done 定义

> 规划者填写：把"做好"变成具体可检查的条件

- [ ] **D1 coordinator 启动自写 pidfile**：`coordinator.sh` 主循环开头（当前使用 `exec 2>>...` 之后、第一次 `log` 之前）必须写入 `$$ > $HARNESS_DIR/.coordinator.pid`。写入前先处理 stale pidfile：(a) 如果 pidfile 存在且 PID 活着且不是自己 → exit 0 退出（不重复启动，由 watchdog 仲裁）；(b) 如果 pidfile 存在但 PID 已死 → 覆盖写自己的 PID；(c) 不存在 → 直接写。实活验证：`kill -TERM <old_pid> && sleep 8 && cat $HARNESS_DIR/.coordinator.pid` 必须是新 PID 且 `kill -0` 可达
- [ ] **D2 pidfile 所有权同步**：退出时（trap EXIT 或 trap TERM）必须清理自己的 pidfile—— `if [[ "$(cat .coordinator.pid 2>/dev/null)" == "$$" ]]; then rm -f .coordinator.pid; fi`。不清理别人的。避免残留。实活验证：kill 后 1 秒内 pidfile 被删（然后 watchdog 拉新的再写入）
- [ ] **D3 watchdog 不 wake 终态 sprint**：`coordinator-watchdog.sh` 在 wake 逻辑里（当前每次拉起 coordinator 后会选最近 sprint wake），必须跳过状态为 `passed/failed/cancelled/superseded` 的 sprint。只 wake 状态为 `drafting/active/reviewing/approved` 的非终态 sprint。如果所有 sprint 都是终态，不发任何 wake 指令（log: `[watchdog] 无非终态 sprint, 跳过 wake`）
- [ ] **D4 watchdog wake 状态校验函数**：新增一个 bash 函数 `is_actionable_state() { case "$1" in drafting|active|reviewing|approved) return 0 ;; *) return 1 ;; esac }`。wake 流程里调用 `is_actionable_state` 过滤。写死终态白名单，避免字符串比较零散
- [ ] **D5 实活验证端到端**：建设者在 `~/.solar/harness/tests/test-coordinator-pidfile.sh` 中脚本化验证：(1) kill 当前 coordinator → sleep 8 → pidfile 不为空且对应活进程；(2) 把所有 sprint 状态临时改成 passed（备份后改，测完还原）→ kill coordinator → watchdog 拉起后 coordinator.log 必须出现 `无非终态 sprint, 跳过 wake` 而非乱 dispatch；(3) 恢复原始 sprint 状态。退出码 0 = PASS。**强约束：测试前备份 sprints/*.status.json，测试后必须还原**
- [ ] **D6 防回归**：`test-harness-resilience.sh` 新增 3 条断言——(i) `grep 'is_actionable_state' coordinator-watchdog.sh` ≥ 1 处；(ii) `grep 'superseded\|cancelled' coordinator-watchdog.sh` 在 wake 跳过逻辑里命中；(iii) `grep 'rm -f.*coordinator.pid' coordinator.sh` trap 里命中。PASS 数目标 ≥ 47（原 44 + 新 3）
