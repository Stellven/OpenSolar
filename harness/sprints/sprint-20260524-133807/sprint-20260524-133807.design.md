## 1. Problem (Restated)

控制面的 **external prerequisite** 当前只能按整张 sprint 的 `status` (或与 `phase` 字符串相等) 来放行。当上游 sprint 的关键节点已经 passed、但整单还没结束时，下游 sprint 会被无谓门禁。`sprint-20260523-agent-plan-optimizer-foundation` 已踩过这个坑。

要求：
1. prerequisite schema 升级为结构化字典：`{sprint_id, required_phase?, required_status?, required_node_id?, required_node_status?}`。
2. **workflow_guard / graph_scheduler / epic_decomposer / solar-autopilot-monitor** 对该结构必须一致解释。
3. 兼容现有字符串形式：`"sid"` 与 `"sid:status"`。
4. 上游关键节点 passed 或 phase 满足时 ready node 必须放行。
5. 回归测试覆盖 APO 这类 case、dict prerequisite、节点级放行与状态回写一致性。
6. 不允许再手改 `task_graph.json` 临时放行。

## 3. Solution

### 3.1 Schema (Normalized)

```jsonc
// 入口允许 string 或 object，标准化后内部统一是 object：
{
  "sprint_id":            "sprint-20260523-agent-plan-optimizer-foundation",  // required
  "required_status":      "passed",                  // optional, default "passed" if neither phase/node given
  "required_phase":       "planning_complete",       // optional
  "required_node_id":     "S2",                      // optional
  "required_node_status": "passed"                   // optional, default "passed" if node_id given
}
```

**字符串向后兼容：**
- `"sid"` → `{sprint_id: "sid", required_status: "passed"}`
- `"sid:phase_x"` → `{sprint_id: "sid", required_status: "phase_x"}` （**保持现有歧义解释**：required 同时匹配 status 或 phase；新结构里推荐用 `required_phase` 明示）

### 3.2 Satisfaction Rule (one source of truth)

A prerequisite is **satisfied** iff **all of the following sub-checks that are present** pass:

| 子条件 | 检查 |
|---|---|
| `required_status` | `upstream.status == required_status` |
| `required_phase`  | `upstream.phase  == required_phase`  |
| `required_node_id` (+ optional `required_node_status`, default `"passed"`) | upstream sprint 的 `task_graph.json.nodes[id==required_node_id]` 的 effective status (合并 `node_results[id].status` 与 `node.status`，passed/skipped 视为通过) `== required_node_status` |

若三个 optional 子条件全都缺失：默认要求 `required_status == "passed"` （保持现有默认）。

**字符串遗留语义**：`required_status` 字段也兼容旧 "phase 或 status" 二选一匹配 (避免破坏已经写成 `"sid:planning_complete"` 的合约)；但新形式建议用 `required_phase` 明示。

### 3.3 Shared Module

新文件 `~/.solar/harness/lib/prerequisite_resolver.py`，公开 3 个函数：

```python
def normalize_prerequisite(entry: Any) -> dict | None:
    """Return canonical dict or None if entry is empty."""

def evaluate_prerequisite(entry: Any, sprints_dir: Path) -> tuple[bool, dict]:
    """Return (ok, detail). detail 至少包含: requirement (json), sprint_id,
    required_status, required_phase, required_node_id, required_node_status,
    current_status, current_phase, current_node_status, reason"""

def iter_blocked(graph: dict, sprints_dir: Path) -> list[dict]:
    """Walk graph['prerequisites'] + graph['dependency_policy']['blocks_until'],
    dedupe by (sprint_id, required_status, required_phase, required_node_id,
    required_node_status), return list of failing detail dicts."""
```

四个消费者只保留薄壳：

```python
from prerequisite_resolver import iter_blocked, evaluate_prerequisite, normalize_prerequisite
```

`workflow_guard._blocked_external_prerequisites(path)` / `graph_scheduler.blocked_external_prerequisites(graph)` / `epic_decomposer.blocked_child_graph_prerequisites(sid)` / `autopilot-monitor.child_graph_external_prerequisite_blocks(sid)` 全部 delegate 到 `iter_blocked`（参数差异通过包装解决：path → 读 json；sid → 拼路径读 json）。

**保持现有公开函数名不变**，只换内部实现 → 其他模块不需要改 import。

### 3.4 Sprints / Status Lookup

`sprints_dir` 默认 `Path(os.environ.get("SPRINTS_DIR", Path.home()/".solar/harness/sprints"))`，与 4 个消费者现行约定一致。共享模块通过参数接收以便测试隔离（已有 `test-autopilot-prereq-dict.sh` 用 `HARNESS_DIR` 覆盖）。

### 3.5 Node-Level Status Resolution

读取 upstream 的 `task_graph.json`：
- `effective_status = (node_results.get(id) or {}).get("status") or node.get("status") or "pending"`，lower()。
- `passed` / `skipped` 视为 PASS。
- 若 task_graph 缺失 → `reason = "upstream_task_graph_missing"`，未通过。
- 若 node_id 不存在 → `reason = "upstream_node_missing:<id>"`，未通过。

### 3.6 Constraints / Invariants

- IR 仍是唯一事实源；本 sprint 只动 4 个 consumer + 1 个新文件 + 测试。
- 公开函数名/返回值结构不变（detail dict 字段只增不减），下游 telemetry/messages 不需要改。
- 不修改 schema `~/.solar/harness/schemas/task-graph.schema.json`（write_scope 已是 oneOf，prerequisites 字段无 strict schema，dict/string 都合法）；如要锁 schema，**Non-goal**，留给后续 sprint。
- 不动 epic/sync_graph_from_children 逻辑。

### 3.7 Risk & Compatibility

| Risk | Mitigation |
|---|---|
| 共享模块对字符串旧合约的解析与现有四份微差异 | 严格用现有 test fixtures (`test-autopilot-prereq-dict.sh`, `test-pm-planner-dag-workflow-guard.sh`) 跑回归；新增 fixture 覆盖 dict + node |
| `required` 字段歧义 (旧 string `"sid:planning_complete"` 想表达 phase) | normalize 时把 `required_status` 也存为 fallback；evaluate 时优先看显式 `required_phase`，否则保持旧的 "status==required or phase==required" 双匹配 |
| upstream task_graph 缺失时被错误判定为 missing → 永久 blocked | reason=`upstream_task_graph_missing` 让 autopilot 监控可识别并 page；不静默通过 |
| 多模块替换出现行为漂移 | 一次性把 4 个消费者全部切到共享模块 (S2 单节点，5 文件 write_scope)，避免半切状态 |

### 3.8 Stop Rules (Hard Gates)

- 缺少可验证 acceptance 不得标 完成。
- 缺少 verifier 决策不得进入 DONE。
- 任何回归测试 FAIL → S3 不得进入 G_VERIFY 通过。
- 手改 task_graph.json 临时放行 = 违例，自动监控应在 S5 (Critic) 中检查。
