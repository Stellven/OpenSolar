## 需求

Agent Loop 上下文工程升级 (参考《Harness Engineering 二》): 解决文章点名的 3 个致命短板:
1. **反馈非结构化** — eval.md 是长文本，建设者修复时需要 LLM 重新解析"哪里 FAIL"
2. **上下文无限膨胀** — 每个 Sprint 的 contract+plan+handoff+eval 累积，第 3 轮 failed_review 时重读所有历史，token 浪费严重
3. **KV Cache 不命中** — dispatch.md 结构不稳定，模型无法复用前序计算，每次都当新请求

目标: 让一个典型 Sprint 的总 token 消耗降低 40%（基线取最近 5 个 Sprint 的 API 调用数据）。

## Done 定义

- [ ] D1: 反馈结构化 — 在 `~/.solar/harness/personas/evaluator.md` 中新增协议: 审判官写 eval.md 后必须额外输出 `sprint-<id>.eval.json` 结构化反馈文件，schema 包含: `{"verdict":"PASS|FAIL","failed_conditions":["D3","D4"],"errors":[{"cond":"D3","severity":"high|med|low","evidence":"grep 输出","fix_hint":"..."}],"tokens_used":N}`。建设者修复时优先读 eval.json 的 `failed_conditions` 而不是解析长文本

- [ ] D2: 上下文压缩归档 — 实现 `~/.solar/harness/archive.sh`: (a) `archive <sid>` 把已 passed 的 Sprint 的 events.jsonl 压缩为 `<sid>.summary.md`（保留: 最终状态 + 关键时间戳 + 轮次数 + 失败原因若有，约 500 字） (b) 原 events.jsonl 移到 `sprints/archive/` 子目录 (c) session.sh 读 archived sprint 时自动返回 summary 而非全量 events (d) 主目录 `sprints/` 只保留进行中/最近 3 个 passed sprint 的完整文件

- [ ] D3: Dispatch KV Cache 友好化 — 重构 coordinator.sh 的 dispatch.md 生成: 把**稳定内容**（协调器指令模板、合约路径说明、步骤编号）放前面，**变化内容**（sprint-id、具体任务描述）放后面。使用分隔符 `<!-- CACHE_BOUNDARY -->` 明确标识。让 Anthropic API 的 prompt caching 能在 stable prefix 上命中

- [ ] D4: 建设者修复循环短路 — 当 `<sid>.eval.json` 存在且 verdict=FAIL 时，coordinator 发给建设者的 dispatch.md 中只注入 `failed_conditions` + 对应的 `errors` 数组，**不注入整个 eval.md 原文**。建设者只需读 JSON 就能定位问题

- [ ] D5: 退出条件扩展 — 在状态机中新增 `needs_human_review` 状态: (a) 当建设者或审判官在 handoff/eval 中写入 `@NEEDS_HUMAN: <原因>` 时，coordinator 检测到后改 status=needs_human_review (b) 派发通知到规划者 pane (c) 规划者可以继续合约或终止 (d) 不计入 3 轮失败上限

- [ ] D6: Token 基线与量化对比 — 实现 `~/.solar/harness/token-tracker.sh`: (a) 从历史 5 个 passed Sprint 抓取总 token（来自 anthropic API 响应或 log） (b) 本 Sprint 完成后跑相同逻辑 (c) 输出对比报告到 `~/.solar/reports/token-savings-<date>.md`，必须包含**真实数字**（非估算），降低 40% 以上为合格

## 原理回顾 (给建设者参考)

### 文章原文精华

**退出条件:**
> 模型拥有对"完成"的完全定义权。当模型不再调用工具时，系统就认为任务完成了。

→ 我们已经有 (pass/fail)，但缺 `needs_human`

**反馈来源:**
> 模型收到的唯一反馈是工具执行的原始结果。系统不做任何加工、筛选或评价。

→ 我们的 eval.md 是**文字评论**不是**原始信号**。改为 JSON 让建设者直接读

**对话历史无限膨胀:**
> 读一个 1000 行的文件吃掉 ~4000 token，跑一条命令的输出可能 ~2000 token，30 个文件加 20 条命令就可能逼近上下文窗口的极限。

→ 我们每 Sprint 是 1 个 contract (500 行) + 1 plan (300 行) + 1 handoff (500 行) + 1 eval (500 行) = ~10K token。3 轮 failed_review 就是 30K+。需要压缩

**KV Cache:**
> 为了避免重复的 K 向量和 V 向量的计算，在进行推理时部署模型的厂商都会使用 KV 缓存（cache）机制...提升"缓存命中"可以有效增加推理的速度和降低推理的成本

→ dispatch.md 重构让 stable prefix 稳定
