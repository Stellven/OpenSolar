## 技术方案

### 数据流

```
sprints/*.status.json
  → issue-adapter.py (--list / --one)
  → JSON issue list [{id, identifier, title, priority, state, ...}]
  → scheduler.py (dry-run mode)
    → 读取 issues → 按 priority/created_at 排序
    → 写 state/symphony/{claimed,running,retry,completed}/*.json
    → 写 logs/symphony-events.jsonl
    → 检查并发上限 → 对每个 claimed sprint:
      → workspace-manager.sh create <sprint-id>
      → runner.sh --dry-run --sprint-id <id>
      → runner 写 proof/run-request.md + proof/runner-env.json + logs/runner.log
    → 标记 completed
```

### Issue Adapter 数据结构

```python
{
  "id": "sprint-20260507-010946",
  "identifier": "SYM-010946",
  "title": "Solar-Harness x OpenAI Symphony Integration P0",
  "description": "...",
  "priority": "P0",
  "state": "active",
  "labels": ["integration", "symphony"],
  "blocked_by": [],
  "created_at": "2026-05-07T01:09:46Z",
  "updated_at": "2026-05-07T01:20:24Z",
  "contract_path": "~/.solar/harness/sprints/sprint-20260507-010946.contract.md"
}
```

### State Mapping (Solar status → Symphony state)

| Solar status | Symphony state |
|-------------|---------------|
| drafting | draft |
| approved / active | active |
| reviewing | in_review |
| passed | completed |
| failed_review | failed_review |
| planning | planning |
| interrupted | interrupted |

### Workspace 目录结构

```
${SOLAR_SYMPHONY_WORKSPACE_ROOT}/sprint-20260507-010946/
  .solar-sprint-id          # sprint ID 纯文本
  contract.md               # symlink 或 copy 自 sprints/
  WORKFLOW.md               # copy 自 templates/WORKFLOW.solar.md
  proof/                    # runner 产出
    run-request.md
    runner-env.json
  logs/                     # runner 日志
    runner.log
```

默认 root: `/Volumes/toshiba/SolarWorkspaces` (Toshiba 已挂载, 525G 可用)
Fallback: `~/.solar/workspaces`

### Scheduler 状态文件

```
~/.solar/harness/state/symphony/
  claimed/
    sprint-xxx.json    # {"sprint_id":"...","claimed_at":"...","priority":"P0"}
  running/
    sprint-xxx.json    # {"sprint_id":"...","started_at":"...","workspace":"..."}
  retry/
    sprint-xxx.json    # {"sprint_id":"...","retry_count":N,"next_retry_at":"...","backoff_ms":...}
  completed/
    sprint-xxx.json    # {"sprint_id":"...","completed_at":"...","result":"pass|fail"}
```

### Event Log Schema (JSONL)

```json
{"ts":"ISO8601","event":"claimed","sprint_id":"...","priority":"P0"}
{"ts":"ISO8601","event":"workspace_created","sprint_id":"...","path":"..."}
{"ts":"ISO8601","event":"runner_started","sprint_id":"...","mode":"dry-run"}
{"ts":"ISO8601","event":"runner_completed","sprint_id":"...","exit_code":0}
{"ts":"ISO8601","event":"completed","sprint_id":"...","result":"pass"}
```

### Runner 安全边界

```bash
runner.sh 启动前检查清单:
1. workspace 路径必须在 allowed roots 下 (Toshiba/SolarWorkspaces 或 ~/.solar/workspaces)
2. env 中无 CLAUDECODE / CLAUDE_CODE_* 变量
3. 当前 tmux pane 不是 solar-harness / solar-harness-lab
4. --unsafe-run-codex 未传 → 只写 proof artifacts, 不启动任何外部进程
```

### CLI 集成方案

在 `solar-harness.sh` 的主 `case` 分支中增加 `symphony)` 子命令:

```bash
symphony)
  shift
  case "${1:-status}" in
    status)   python3 "$HARNESS_DIR/lib/symphony/scheduler.py" --status ;;
    dry-run)  python3 "$HARNESS_DIR/lib/symphony/scheduler.py" --dry-run ;;
    workspace) shift; bash "$HARNESS_DIR/lib/symphony/workspace-manager.sh" show "$1" ;;
    *)        err "Unknown symphony subcommand: $1"; exit 1 ;;
  esac
  ;;
```

### Doctor 增强方案

在 `doctor_json()` 的 `result` dict 中追加:

```python
"symphony": {
    "installed": bool,         # scheduler.py 是否存在
    "workspace_root": str,     # 实际使用的 root
    "claimed": int,            # state/symphony/claimed/*.json 数量
    "running": int,            # state/symphony/running/*.json 数量
    "retry": int,              # state/symphony/retry/*.json 数量
    "repairs_available": []    # stale claimed / missing proof
}
```

### Python 依赖策略

P0 仅使用 stdlib: `json, os, glob, pathlib, datetime, argparse, sys, subprocess, re, shutil, fnmatch`。不引入第三方 pip 包。

## 风险点

| # | 风险 | 影响 | 缓解措施 |
|---|------|------|---------|
| R1 | Toshiba 挂载点路径 `/Volumes/toshiba` (小写) | workspace root 解析错误 | 代码中做 `-d` 检测确认, 已实测 df 显示 `/Volumes/toshiba` |
| R2 | symphony state 目录与 coordinator state 冲突 | 调度冲突/竞态 | symphony state 独立放 `state/symphony/`, 不碰现有 `sprints/` |
| R3 | runner.sh 误触 live tmux pane | 破坏正在运行的 agent | 三重保护: TMUX_PANE env 检查 + workspace 路径白名单 + dry-run 默认 |
| R4 | Python stdlib 不足以处理复杂调度 | 功能受限 | P0 只做 dry-run, 不需要复杂调度; 状态管理用 JSON 文件 + 文件锁 |
| R5 | `solar-harness.sh` patch 破坏现有功能 | 用户无法启动 harness | 只在末尾加新 `case` 分支, 不改任何现有代码路径; `bash -n` 验证 |
| R6 | workspace symlink 跨设备失败 | contract.md 不可读 | 先 `ln -sf`, 失败则 fallback `cp` |
| R7 | workspace key 含非法字符 (路径注入) | 安全漏洞 | 严格白名单 `[A-Za-z0-9._-]`, 不匹配则拒绝并报错 |
| R8 | JSONL log 文件无限增长 | 磁盘占用 | P0 不做轮转, 但文件大小可监控; P1 加 logrotate |
| R9 | 多个 dry-run 并发写同一 state 文件 | 数据损坏 | P0 dry-run 单进程顺序执行, 不存在并发; scheduler 强制串行 |
| R10 | doctor.sh patch 增加 Python 依赖 | 现有 doctor 功能受影响 | doctor 的 symphony section 独立在 Python heredoc 内, 不影响其他检查 |

## No-Live-Pane-Mutation Protection

1. **Builder MUST NOT** run `tmux send-keys`, `tmux respawn-pane`, `tmux kill-pane`, `tmux kill-session` targeting `solar-harness` or `solar-harness-lab` sessions.

3. **Runner guard** (D6): runner.sh checks `$TMUX_PANE` / `$TMUX` and refuses to run if inside a solar-harness pane.
4. **Workspace isolation**: all execution happens under `$SOLAR_SYMPHONY_WORKSPACE_ROOT`, never under `~/.solar/harness/` itself.
