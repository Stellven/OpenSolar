## 1. Objective

把 Solar-Harness 从“tmux pane 里坐着多个 Claude TUI 化身干活”升级为“Solar 负责产品/规划/审判/记忆，Symphony-style 后台执行器负责隔离 workspace、并发、重试和可观测执行”的 P0 最小闭环。

本 sprint 不要求完整接入 Linear，也不要求直接部署 openai/symphony 的 Elixir reference。目标是基于 Symphony SPEC 做 Solar 原生集成层:

```text
User / Codex / PM input
  -> Solar PM Gate / Product Brief
  -> Solar sprint contract
  -> solar-symphony adapter
  -> per-sprint isolated workspace
  -> agent runner (dry-run first, Codex app-server guarded)
  -> proof artifacts / telemetry / evaluator
```

最终用户体验目标:

> 用户不需要记命令、不需要盯 pane、不需要知道何时派发。Solar 在后台自动调度任务，只在需要决策或风险确认时提醒用户。

## 2. Background

当前 Solar-Harness 已有:

- `coordinator.sh` 状态机
- `pane-launcher.sh` persona 启动
- `.pane-assignments` pane 占用追踪
- `dispatch_to_pane` 派发和防吞键
- `PLANNER-INBOX.md` / `.planner-inbox.md`
- `codex-bridge/from-codex` / `to-codex`
- `sprints/*.status.json` / contract / plan / handoff / eval
- `cortex` memory / knowledge / artifacts
- tests / telemetry / self-evolve
- architect / research / mixture 拓扑
- PM pane + dual quadrant layout

但当前架构的关键问题:

1. live tmux pane 同时承担“用户界面”和“执行器”，容易被误杀、卡住或继承错误环境。
2. 多 agent 并发靠 pane 分配，不是任务级 workspace 隔离。
3. watchdog 主要是进程自愈，不是 issue/sprint 级 retry/backoff/reconciliation。
4. 用户仍然会被迫关注 pane 状态，而不是只管理工作目标。

Symphony SPEC 的关键启发:

- 长期运行 orchestrator 读取 work item。
- 每个 issue/work item 创建确定性的 isolated workspace。
- 由 repo-owned `WORKFLOW.md` 定义运行策略和 prompt。
- 维护 claimed/running/retry/completed 等单一权威调度状态。
- 通过 structured logs / runtime events 提供可观测性。
- agent runner 使用 Codex app-server protocol，并发、超时、stall、retry 都在调度层处理。

## 3. Target Architecture

### 3.1 不替换 Solar 的部分

Solar 继续负责:

- PM Gate: 解读用户意图、产出 product brief。
- Planner: 把 brief 变成 sprint contract / acceptance criteria。
- Evaluator: 审判 proof artifacts。
- Cortex/Memory: 汇总经验、状态、报告。
- tmux UI: 只作为 status surface 和人工交互面板，不再作为唯一执行器。

### 3.2 新增 Symphony-style 后台执行层

新增 `solar-symphony` 模块:

```text
~/.solar/harness/
  solar-symphony.sh
  lib/symphony/
    issue-adapter.py
    workflow-loader.py
    workspace-manager.sh
    scheduler.py
    runner.sh
  templates/
    WORKFLOW.solar.md
  docs/
    symphony-integration-adr.md
  tests/
    test-symphony-*.sh
```

职责:

- 从 `sprints/*.status.json` 读取候选 sprint，归一化为 Symphony Issue model。
- 为每个 sprint 创建 isolated workspace。
- 使用 `WORKFLOW.solar.md` 生成 agent prompt。
- 维护 `~/.solar/harness/symphony-state.json` 或 `state/symphony/*.json`。
- 先实现 dry-run runner，再 guarded 接入 Codex app-server。
- 输出 proof artifacts 给 evaluator，而不是直接改用户 live pane。

## 5. Forbidden Actions

- 禁止对 live sessions `solar-harness` / `solar-harness-lab` 执行 `tmux respawn-pane -k`、`tmux kill-pane`、`tmux kill-session` 做测试。
- 禁止把现有 tmux pane 当 Symphony worker 的唯一执行器。
- 禁止删除历史 sprint、日志、bridge processed 文件。
- 禁止 destructive git 操作，例如 `git reset --hard`、`git checkout --`。
- 禁止默认连接真实 Linear API；Linear 只允许作为后续 P1。
- 禁止直接把 openai/symphony Elixir reference 当生产依赖塞进 Solar。
- 禁止未隔离 workspace 就让多个 worker 改同一个项目目录。
- 禁止在没有 dry-run 证据前启用真实 Codex app-server runner。
- 禁止泄漏或硬编码 API token。

### D3: Sprint Issue Adapter

实现 `lib/symphony/issue-adapter.py`。

功能:

- 读取 `sprints/*.status.json`。
- 归一化为 Symphony Issue model:
  - `id`
  - `identifier`
  - `title`
  - `description`
  - `priority`
  - `state`
  - `labels`
  - `blocked_by`
  - `created_at`
  - `updated_at`
  - `contract_path`
- 支持 CLI:

```bash

```

### D4: Workspace Manager

实现 `lib/symphony/workspace-manager.sh`。

功能:

- 根据 sprint id 创建 deterministic workspace。
- 默认 root:

```text
${SOLAR_SYMPHONY_WORKSPACE_ROOT:-/Volumes/Toshiba/SolarWorkspaces}
```

若 Toshiba 不存在，fallback:

```text
~/.solar/workspaces
```

- workspace key 必须只含 `[A-Za-z0-9._-]`。
- workspace 内必须写入:
  - `.solar-sprint-id`
  - `contract.md` symlink 或 copy
  - `WORKFLOW.md` copy
  - `proof/`
  - `logs/`

### D5: Dry-run Scheduler

实现 `lib/symphony/scheduler.py` 或等价 shell/python 模块。

P0 只做 dry-run:

- 读取 issue-adapter 输出。
- 根据 priority / created_at / identifier 排序。
- 实现 `claimed`, `running`, `retry`, `completed` 状态文件。
- 支持并发上限，但 dry-run 不启动真实 agent。
- 支持 retry backoff 计算。
- 输出 structured JSONL logs:

```text
~/.solar/harness/logs/symphony-events.jsonl
```

### D6: Runner Contract / Guarded Codex App-server Stub

实现 `lib/symphony/runner.sh`。

P0 要求:

- 默认 `--dry-run`，只渲染 prompt，不启动 Codex。
- `--unsafe-run-codex` 才允许尝试 `codex app-server`。
- 启动真实 Codex 前必须检查:
  - 当前 workspace 在 allowed root 下。
  - 没有 `CLAUDECODE` / `CLAUDE_CODE_*` 污染。
  - 不在 `solar-harness` live pane 里运行。
- 输出:
  - `proof/run-request.md`
  - `proof/runner-env.json`
  - `logs/runner.log`
