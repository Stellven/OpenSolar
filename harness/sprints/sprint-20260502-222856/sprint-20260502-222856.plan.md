## 技术方案

### 数据结构

```
watchdog PERSONA_PANES:
  [0]="planner"  [1]="builder"  [2]="evaluator"  [3]="second-builder"

coordinator PANE 常量:
  PANE_PLANNER="solar-harness:0.0"
  PANE_BUILDER="solar-harness:0.1"
  PANE_EVALUATOR="solar-harness:0.2"
  PANE_BUILDER2="solar-harness:0.3"
```

### Dispatch 路由逻辑 (coordinator.sh)

```python
# handle_active / handle_approved 中选择目标 builder:
if pane_idle(PANE_BUILDER2):
    dispatch_to_pane(PANE_BUILDER2, ...)  # builder2 空闲优先
elif pane_idle(PANE_BUILDER):
    dispatch_to_pane(PANE_BUILDER, ...)   # builder1 空闲
else:
    # 两个都忙, 放入队列等下次轮询
    log "both builders busy, deferring"
```

### Persona 文件策略

`second-builder.md` = `builder.md` 的硬链接 (不是符号链接, 因为 `cat` 需要内容):
```bash
cp ~/.solar/harness/personas/builder.md ~/.solar/harness/personas/second-builder.md
```

## 风险点

1. **双 builder 并发写文件冲突**: 两个 builder 可能同时写同一个项目目录。缓解: second-builder 不使用 worktree (只用 `cd $HOME`), 实际 sprint 工作目录不同 (sprint 文件在 sprints/ 下, 按 sid 隔离)。但如果两个 builder 同时改 `coordinator.sh` 等共享文件就会冲突。
   - **处理**: 初始版本 second-builder 只做**调研/报告类 sprint** (不改代码文件), 后续再加代码隔离。

2. **codex-bridge 永久下线**: pane 3 改成 second-builder 后, `codex-bridge.sh` 不再有独立 pane 运行。但 `call_codex()` 在 coordinator.sh 中仍存在 (后台异步, 不依赖 pane)。
   - **处理**: codex-bridge.sh 改为纯后台守护进程 (不占 pane), 或暂时禁用。合约说用户说"没用", 所以禁用 OK。

3. **watchdog respawn worktree**: second-builder 不走 worktree, 但 watchdog respawn 的 `_respawn_workdir` 逻辑需确认 second-builder 用 `$HOME` 而非 builder 的 worktree 路径。
   - **处理**: watchdog respawn 逻辑中 `persona == "builder"` 的 worktree 判断不会匹配 `second-builder`, 自动用 `$HOME`。

4. **bash 3.2 兼容**: `declare -A PERSONA_PANES` 在 macOS 默认 bash 3.2 不支持。但 watchdog 当前已在用, 说明已走 `/opt/homebrew/bin/bash` (5.x)。
   - **确认**: watchdog shebang 是 `#!/bin/bash` 但通过 coordinator 用 homebrew bash 启动, 或脚本内有兼容处理。

5. **测试 kill pane 3**: 需要 tmux session `solar-harness` 存在且 pane 3 在运行。测试时不能 kill 自己 (builder pane 1)。
   - **处理**: 测试脚本只 kill pane 3, 不影响 builder pane 1。
