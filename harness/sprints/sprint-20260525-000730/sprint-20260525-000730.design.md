# Design — GEPA-Optimized Self-Evolving Requirement Compiler

> **Sprint**: `sprint-20260525-000730`
> **Slice**: `strategy` (P0/P1 — compiler profile + eval harness + GEPA adapter)
> **Knowledge Context: solar-harness context inject used**
> **Harness Modules Used: harness-knowledge, harness-graph, harness-skills (autoresearch)**

## 0. Slice Posture

本切片把 Requirement Compiler 从硬编码参数升级为 GEPA 离线可优化的自进化系统。核心原则：**GEPA 只优化 profile，不直接改 canonical artifacts；生产编译链路保持确定性。**

## 2. Architecture Decisions

### D1: Profile Storage — SQLite + JSON 文件双模式

**决断**: Profile 定义存 JSON 文件 (`profiles/*.json`)，运行时缓存到 SQLite (`compiler_profiles` 表)。

**理由**:
- JSON 文件可 git 版本化、可人工审阅、可 propose/review/promote
- SQLite 缓存加速运行时查询
- 与现有 `artifact_store.py` 的存储模式一致

### D2: ASI 权重 — 从 golden cases 校准，初始等权

**决断**: ASI 初始权重 = 等权 (每维 1/N)，P0 不做自动权重优化。

**理由**:
- 缺乏足够数据做统计校准
- 等权是最安全的起点
- 后续 GEPA 可自动调权重

### D3: CompileEvalHarness — 纯规则优先，LLM-as-judge 为可选增强

**决断**: P0/P1 只实现纯规则评估 (schema 检查、字段完整度、DAG 可执行性、acceptance 覆盖率)。LLM-as-judge 作为 P2 可选。

**理由**: 纯规则确定性高、可复现、不依赖模型调用
- Open Question Q4 降级为 P2

### D4: Golden Cases 来源 — accepted artifacts

**决断**: 从 `~/Knowledge/_raw/solar-harness/accepted/` 选取高质量 sprint artifacts 作为 golden cases。

**理由**: 已有 accepted 标记的数据最可靠；人工构造成本高

### D5: 与现有 PM → Planner → Builder 主链的集成方式

**决断**: 编译入口不变；只在编译完成后追加 eval 步骤。Profile 查询是编译器内部实现细节，不改外部接口。

**理由**: 最小侵入；现有编译管线不受影响

## 5. CompileEvalHarness

### 评估维度

| 维度 | 检查内容 | 权重 (初始) |
|---|---|---|
| IR schema compliance | RequirementIR 是否符合 schema | 1/7 |
| Contract completeness | 6 类 policy 是否齐全 + acceptance 映射 | 1/7 |
| DAG executability | 无 cycle + write_scope 有声明 + gate 有 acceptance | 1/7 |
| Acceptance coverage | 每个 acceptance 有 ≥1 validation step | 1/7 |
| PRD-contract-DAG alignment | 三份产物对齐 (无漂移) | 1/7 |
| Trace consistency | planner/builder/evaluator traces 互相一致 | 1/7 |
| Coverage score | requirement_coverage.py 的覆盖率 | 1/7 |

### Score 计算

```python
ASI = sum(dimension_score[i] * weight[i] for i in dimensions)

# hard fail: 任何 hard validator 失败 → ASI = 0, 不看分项
if any(hard_validator_failed):
    ASI = 0
```

### Hard Validators (D5 from PRD)

| Validator | 触发条件 | 结果 |
|---|---|---|
| HV1: IR schema invalid | RequirementIR 不符合 schema | ASI=0, hard fail |
| HV2: Contract mismatch | contract 字段与 IR 对不上 | ASI=0, hard fail |
| HV3: DAG cycle | task_graph 有环 | ASI=0, hard fail |
| HV4: Acceptance no validation | acceptance 无 validation 映射 | ASI=0, hard fail |
| HV5: Research no evidence | research 任务缺 evidence gate | ASI=0, hard fail |
| HV6: High-risk no approval | 高风险节点缺 approval gate | ASI=0, hard fail |

## 7. GEPA Adapter Interface

```python
class CompileGEPAAdapter:
    """Adapter connecting CompileEvalHarness to GEPA optimizer."""

    def __init__(self, gepa_adapter, eval_harness, profile_registry):
        self.gepa = gepa_adapter          # reuse integrations/gepa_optimizer/adapter.py
        self.eval = eval_harness           # new: lib/compile_eval/
        self.registry = profile_registry   # new: lib/compiler_profile/

    def fitness_function(self, candidate_profile, golden_cases):
        """GEPA calls this to evaluate a candidate profile."""
        scores = []
        for case in golden_cases:
            # Compile with candidate profile
            artifacts = compile_with_profile(case.input, candidate_profile)
            # Evaluate
            result = self.eval.evaluate(artifacts, case.expected)
            scores.append(result.asi_score)
        return statistics.mean(scores)  # GEPA maximizes this

    def propose(self, candidate_profile, score, notes):
        """Delegate to gepa_optimizer promote/propose."""
        return self.gepa.propose(candidate_profile, score, notes)

    def promote(self, profile_id):
        """Atomic promote via gepa_optimizer promote."""
        return promote(profile_id)  # reuse promote.py

    def rollback(self, profile_id):
        """Safe rollback via gepa_optimizer promote.rollback."""
        return rollback(profile_id)
```

**集成点**:
- `gepa_adapter.run()` — 复用，fitness_function 替换为 CompileEvalHarness
- `artifact_store` — 复用，存储 profile candidates + ASI scores
- `budgets` — 复用，控制优化预算
- `promote` — 复用，atomic profile promotion
- `cli._cmd_propose` — 复用，propose CLI

## 10. Open Questions Resolution

| Q | Answer | Rationale |
|---|---|---|
| Q1 | 等权 (1/7), 后续 GEPA 调 | 缺乏数据校准 |
| Q2 | JSON 文件 + SQLite 缓存 | 可 git 版本化 + 快速查询 |
| Q3 | evidence_policy schema 可统一 | 两个 sprint 用同一套 profile schema |
| Q4 | P0 纯规则, P2 LLM-as-judge | 确定性优先 |
| Q5 | accepted artifacts | 已有高质量数据 |

## 11. Constraints

- RequirementIR 是唯一事实源
- GEPA 离线 only, 不在线优化
- Hard validators 优先于 score
- 必须复用现有 GEPA optimizer
- 新目录在 `harness/lib/` 下
- 不破坏现有 PM → Planner → Builder 主链

## 12. Stop Rules

- 缺 task_graph.json → 不派 builder
- compiler_profile schema 未定义 → 不算 done
- CompileEvalHarness 不能对 golden case 产出 score → 不算 done
- GEPA adapter 未接入现有 run/propose/promote → 不算 done
- Hard validators 任一未实现 → 不算 done
- 新建 optimizer substrate (平行重造) → 违规
