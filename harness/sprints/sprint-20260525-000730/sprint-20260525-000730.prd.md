# PRD: GEPA-Optimized Self-Evolving Requirement Compiler

**Source**: codex-pm-router (requirement_ir `req-34c0c515a55a`)
**Priority**: P0
**Lane**: strategy
**Handoff To**: planner

Knowledge Context: solar-harness context inject used

## 背景 / Context

Solar-Harness 的 Requirement Compiler 已经能从原始需求编译出 RequirementIR → PRD → Contracts → TaskDAG → Handoff 完整产物链。但当前的编译参数 (intake_policy, requirement_ir_policy, contract_compiler_policy, dag_compiler_policy, evidence_policy, handoff_policy) 都是硬编码或手工调的，没有版本化、没有评估、没有自动优化。

现有基础设施：
- `integrations/gepa_optimizer/` — GEPA (Gene-Expression Programming Adapter) 离线优化器，已有 run/propose/review/promote/rollback 链路
- `artifact_store` — 产物存储
- `pareto` — Pareto frontier 分析
- `budget caps` — 优化预算控制
- `requirement coverage` / `accepted export` / `8765 status` — 现有评估和状态管线

**核心矛盾**: Requirement Compiler 的编译质量 (IR 对齐度、contract 完整性、DAG 可执行性、acceptance 覆盖率) 完全取决于 profile 参数，但目前没有系统化的方式评估和优化这些参数。GEPA 已经存在于 harness 中，但还没有接入 Requirement Compiler 的 profile 优化。

## 用户问题 / Problem

1. **Profile 参数不可优化**: intake_policy、dag_compiler_policy 等编译参数是固定的，无法根据历史编译结果自动调优
2. **无编译质量评估**: RequirementIR、Contracts、TaskDAG 的质量没有统一评估框架 → 无法量化 profile 变化的影响
3. **无 Pareto 专长**: 不同任务类型 (implementation/research/high-risk/full-prd) 需要不同 profile，但当前只有一套参数
4. **产物漂移**: PRD/contract/DAG 多份产物可能互相不一致，没有编译期对齐检查

## 用户目标 / Goals

1. **compiler_profile registry**: 把 6 类编译策略版本化为可注册、可切换的 profile，GEPA 只优化 profile，不直接改 canonical artifacts
2. **CompileEvalHarness**: 统一评估 RequirementIR/Contracts/TaskDAG/Handoff/Trace/Coverage/Acceptance Verdict 为 score + ASI (Aggregate Suitability Index)
3. **GEPA adapter**: 将 CompileEvalHarness 接入现有 `integrations/gepa_optimizer/` 的 run/propose/review/promote/rollback 链路，复用现有 substrate
4. **Generalization-ready**: 支持 single-task、multi-task、generalization 三种模式，保留 Pareto frontier

## 用户故事 / User Stories

- **As PM**, I want compiler_profile registry 能让我为不同任务类型选不同 profile, so that implementation 任务和 research 任务用不同的编译策略。
- **As Planner**, I want CompileEvalHarness 能量化评估编译产物的质量, so that 我能判断一个 profile 变化是否真的改善了输出。
- **As Builder**, I want GEPA 优化的 profile 不直接改生产编译链路, so that 我的执行环境是确定性的、可复现的。
- **As Evaluator**, I want acceptance coverage 和 requirement trace 能作为 GEPA 的 feedback 信号, so that 优化器能基于验证结果改进 profile。
- **As 平台维护者**, I want 复用现有 GEPA optimizer 而不是平行重造, so that 我不用维护两套优化 substrate。

## 功能需求 / Requirements

### Phase 拆分

| Phase | 内容 | 依赖 |
|---|---|---|
| **P0/P1** | compiler_profile schema + registry、CompileEvalHarness、golden cases/trainset/valset、ASI trace schema、GEPA adapter 接口打通 | 无 |
| **P2** | Profile selection / canary / rollback / Optimization View | P0/P1 |

### R1: compiler_profile Schema & Registry
- Profile 包含 6 类策略: intake_policy, requirement_ir_policy, contract_compiler_policy, dag_compiler_policy, evidence_policy, handoff_policy
- 每个 policy 有版本号 + 参数 + 适用范围
- Profile 注册到 registry，支持标签 (implementation/research/high-risk/full-prd)
- GEPA 只优化 profile 参数，不直接改 canonical artifacts (PRD/contract/DAG)
- Profile 变更走 propose → review → promote → rollback 链路

### R2: CompileEvalHarness
- 输入: 一组编译产物 (RequirementIR, Contracts, TaskDAG, Handoff) + planner/builder/evaluator traces
- 评估维度: IR schema 合规性、contract 完整性、DAG 可执行性、acceptance 覆盖率、trace 一致性
- 输出: 总分 score + ASI (Aggregate Suitability Index) + 分项 breakdown
- Golden cases / trainset / valset 用于校准评估基准

### R3: ASI Trace Schema
- 记录每次编译的 ASI score、使用的 profile 版本、任务类型、时间戳
- 支持按 profile 版本、任务类型、时间范围查询 ASI 趋势
- ASI trace 是 GEPA 优化 feedback 的核心数据源

### R4: GEPA Optimizer Adapter
- 复用现有 `integrations/gepa_optimizer/` 的 run/propose/review/promote/rollback
- Adapter 把 CompileEvalHarness 的 score 作为 GEPA fitness function
- 支持 Pareto frontier: 保留不同 profile 专长，不只看平均分
- Budget caps: 复用现有优化预算控制
- 安全 promotion: profile 变更必须经过 review 才能进入生产

### R5: Hard Validators
- 以下条件必须 hard fail，不受 GEPA score 影响:
  - IR schema invalid
  - Contract mismatch
  - DAG cycle detected
  - Acceptance 无 validation 映射
  - Research 任务无 evidence gate
  - 高风险节点无 approval gate

### R6: Integration Points
- 与 `requirement coverage` 集成: coverage 作为 CompileEvalHarness 的评估维度
- 与 `accepted export` 集成: accepted artifacts 作为 golden cases 来源
- 与 `8765 status` 集成: status 变更作为 trace 反馈

## 非目标 / Non-Goals

- **不做完整四区 PM pane 重构** — 本 sprint 只做编译底座
- **不绕过 planner 直接进入 builder** — 强制 handoff
- **不做 P2 profile selection UI** — P0/P1 只做后端接口
- **不做 online optimization** — GEPA 是离线/旁路，不允许边优化边派单
- **不做 GEPA 算法本身的改进** — 只做 adapter，不改 optimizer 核心

## 约束 / Constraints

- **RequirementIR 仍然是事实源**: PRD/Contracts/TaskDAG/Handoff 都由 deterministic compiler 从 profile 生成
- **GEPA 是离线 Optimization Plane**: 不允许直接在生产请求上边优化边派单
- **Hard validators 优先于 GEPA score**: schema invalid / contract mismatch / DAG cycle / acceptance 缺 validation 时必须 hard fail
- **必须复用现有 `integrations/gepa_optimizer/`**: 不平行重造 optimizer substrate
- **必须支持 Pareto frontier**: 不要只用平均分
- **新目录结构必须落在 harness/ 范围内**: `harness/lib/compiler_profile/`, `harness/lib/compile_eval/`
- **与现有 PM → Planner → Builder 主链兼容**: 不能破坏现有编译管线

## 风险 / Risks

| # | 风险 | 影响 | 缓解 |
|---|---|---|---|
| R1 | PRD/contract/DAG 多份产物漂移 | 多源不一致 | Requirement IR 做唯一事实源，所有视图从 IR 编译 |
| R2 | 原始需求直接派给 Builder | 执行发散 | 强制 product-brief / planner handoff |
| R3 | 验收标准没有映射到验证步骤 | gate 形同虚设 | 编译期做 acceptance coverage 检查 |
| R4 | GEPA 优化 profile 后生产编译结果改变 | 确定性丧失 | profile 变更走 propose/review/promote，不自动进入生产 |
| R5 | CompileEvalHarness 评估维度不全 | ASI 不反映真实质量 | P0/P1 先做最小集，用 golden cases 校准 |
| R6 | Golden cases 不够代表真实分布 | GEPA 过拟合 | trainset/valset 分离，支持后续扩展 |
