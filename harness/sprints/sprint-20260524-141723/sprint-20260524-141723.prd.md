## 用户问题 / Problem

如果不做 S02 架构设计而直接让 S03 builder 动手，会出现 5 个具体问题：

1. **命名冲突**: 105859 用 `*.extracted.md`，intake 要求 `*.semantic.md`，不统一会导致 grep 断裂和 QMD layer 不一致
2. **schema 缺字段**: intake 要求 latency/token_cost/model/provenance，但 105859 的 `extract_jobs`/`extract_outputs` 表缺这些列，没有 schema v2 计划
3. **状态机不完整**: intake 要求 `EXTRACT_FAILED_RETRYABLE` 和 `DONE_RAW_ONLY_WARN`，105859 的状态机没有这两个状态
4. **adapter 缺失**: intake 要求 8 类 source (YouTube/GitHub/PDF/accepted/Solar/Obsidian/ChatGPT/Web)，但 105859 只实现了 3 类
5. **query grounding 未定义**: O10 要求 extracted → raw/vault grounding，但接口契约不存在

## 用户目标 / Goals

1. 决断 5 个架构决策 (A1-A5)：文件命名、schema 扩展、状态机扩展、adapter 策略、query grounding 接口
2. 产出完整的 6 层系统分层架构 (Source Adapters → Dispatcher → Registry → Extract Pipeline → QMD Indexer → Query & Reporting → Observability)
3. 定义所有层间接口契约 (Dispatcher↔Registry, Adapter↔Dispatcher, Extractor↔Registry, Validator↔Registry, GroundingHook↔Registry)
4. 设计向后兼容和迁移方案，确保 105859 已交付产物不受影响
5. 为 S03 builder 产出可执行的 task_graph (S3-A through S3-F)

## 用户故事 / User Stories

- **As S03 Builder**, I want 一份带有 write_scope 的 task_graph, so that 我知道每个节点该改哪些文件、不改哪些文件。
- **As S04 Builder**, I want A5 的 GroundingHook 接口契约, so that 我知道 query grounding hook 的 input/output/fallback 策略。
- **As S05 Evaluator**, I want 完整的 outcome→decision 追踪矩阵, so that 我能按 outcome 逐项验收。
- **As 平台维护者**, I want 6 层架构的降级策略, so that 任何一层故障都不会阻塞 raw/vault 的可查询性。
- **As Planner (S02 本身)**, I want 明确的 stop_rules, so that 不在缺 decision 时声称 done。

## 功能需求 / Requirements

### R1: Architecture Decision A1 — File Naming
- `*.semantic.md` 作为正式名称
- `*.extracted.md` 保留为 symlink 别名，兼容期 2 sprint
- `watermarks` 表 `layer` 值从 `extracted` 迁移为 `semantic`
- `qmd_index_events.layer` 新写入用 `semantic`，旧数据不动

### R2: Architecture Decision A2 — Schema Extension
- `extract_jobs` 新增: `schema_version`, `endpoint`, `started_at`, `finished_at`
- `extract_outputs` 新增: `latency_ms`, `token_input_count`, `token_output_count`, `cost_estimate`, `model_fingerprint`
- 使用 `ALTER TABLE ADD COLUMN`（不用新表），idempotent migration
- `migration_log` 记录 version=2

### R3: Architecture Decision A3 — State Machine Extension
- 新增 `EXTRACT_FAILED_RETRYABLE` 状态（抽取阶段可重试失败）
- 新增 `DONE_RAW_ONLY_WARN` 终态（raw/vault 可查但无 semantic）
- 重试策略: max 3 次, exponential backoff
- Skip path: `EXTRACT_ELIGIBLE → DONE_RAW_ONLY_WARN`（circuit breaker 或 adapter 不支持）
- 新状态不影响现有 documents 行

### R4: Architecture Decision A4 — Adapter Strategy
- 统一 AdapterProtocol: `source_kind`, `discover(limit)`, `verify_spans(doc_id, db)`
- 8 类 source_kind: obsidian_vault, raw_chatgpt, raw_web, youtube_transcript, github_trends, pdf_manual, accepted_sprint, solar_artifact
- 5 类新 adapter 由 S03 实现: youtube, github, pdf, accepted, solar
- YouTube 迁移策略: 旁路并行，S05 验证后切换 cron

### R5: Architecture Decision A5 — Query Grounding Interface
- GroundingHook 只读接口: input(query, extracted_hits, registry) → output(grounded_context with evidence_spans, confidence, source_layer)
- Fallback policy: 无 span → confidence=0.5; span 不存在 → drop claim; raw/vault 同路径 → 优先用 raw/vault
- 超时 2s → fallback raw/vault only
- Consumer: context inject, Tech Hotspot report, status server

### R6: System Layering
- 6 层架构: Source Adapters → Dispatcher → Registry → Extract Pipeline → QMD Indexer → Query & Reporting → Observability
- 每层有明确的职责、依赖和降级策略

### R7: Compatibility & Migration
- 所有变更向后兼容，不影响 105859 已交付产物
- Migration 步骤明确，S03 builder 可按序执行
- 状态机新增转移用白名单，不允许 free-form state string

## 非目标 / Non-Goals

- **不写运行时代码** — S03 builder 依据本 design 实施
- **不替换 QMD embedding** — ThunderOMLX 是 L2 语义抽取 worker，不是 embedder
- **不覆盖所有实现细节** — 单个大 PRD 不是目标，足够 builder 执行即可
- **不直接派 builder** — PRD 必须先交给 planner
- **不用"已完成"替代可复现证据** — Definition of Done 铁律

## 约束 / Constraints

- 本切片是 **planning-only**，不写 runtime 代码
- 不动 105859 已交付的 9 个 lib 文件（S03 只 extend，不改原有行）
- `watermarks.layer` 的 `extracted` 行 migration 用 DELETE + INSERT（SQLite PRIMARY KEY 是 layer，不能 ALTER）
- 状态机新增转移必须用白名单
- GroundingHook 是只读操作
- 所有新 adapter 的 `discover()` 必须支持 `--limit`
- YouTube 迁移期间旧路径和新路径并存，旧 cron 不删直到 S05 验证
- 与并行 sprint `sprint-20260524-105859` 的交叉必须显式标注
