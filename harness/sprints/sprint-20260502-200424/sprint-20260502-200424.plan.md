## 根因分析 (D1)

### 事故复现链

Watchdog `check_panes()` 在检测到 builder pane 异常后执行 respawn:
```
tmux respawn-pane -k -t "$SESSION_NAME:0.$pidx" "bash ${_esc_h}/start-incarnation.sh $persona ${_esc_w}"
```

手动执行等价命令成功,说明命令本身没问题。**status 127 = command not found**。

### 排查假设 (4个)

#### H1: `set -eu` + worktree.sh 触发 exit 1 → start-incarnation.sh 整体退出 127

**验证方法**:
```bash
bash -x ~/.solar/harness/start-incarnation.sh builder $HOME 2>&1 | tail -50
```
看 `setup_builder_worktree` 是否触发 `set -e` 退出。

**证据来源**: `start-incarnation.sh` 第10行 `set -eu`,第42行 `source worktree.sh` + 调用 `setup_builder_worktree`。worktree.sh 中 `git worktree add` 如果失败,在 `set -e` 下会直接退出脚本。但 status 127 不是 exit 1...继续查。

**更新**: status 127 = shell 找不到命令。`respawn-pane` 的命令字符串中的 `bash` 路径问题? 检查 tmux 的 PATH。

#### H2: tmux respawn-pane 的 env/PATH 与手动终端不同 → 找不到 bash

**验证方法**:
```bash
# 查看 watchdog 运行时的 PATH
tmux show-environment -t solar-harness 2>/dev/null | grep PATH
# 对比手动 shell
echo $PATH
```

**关键点**: watchdog 后台进程 (run_watchdog) 由 `bash coordinator-watchdog.sh start` 启动,其 PATH 可能缺少 `/opt/homebrew/bin` 或 `/usr/local/bin`。但 `bash` 在 `/bin/bash` 或 `/opt/homebrew/bin/bash`,应该不会找不到...

**更新**: respawn-pane 命令用 `bash` 而非绝对路径。如果 tmux session 的 PATH 被清空或极简,可能找不到 bash。

#### H3: `printf '%q'` 转义问题 → 命令字符串被错误解析

**验证方法**:
```bash
_h=$(printf '%q' "$HOME/.solar/harness")
_w=$(printf '%q' "$HOME")
echo "tmux respawn-pane -k -t solar-harness:0.1 \"bash ${_h}/start-incarnation.sh builder ${_w}\""
```

**关键点**: `printf '%q'` 可能产生 `$'\XXX'` 格式转义,tmux respawn-pane 的命令解析可能不认这种格式。

**观察**: `_esc_h` 对 `$HOME/.solar/harness` 通常不需要转义 (无特殊字符)。但如果路径含空格或特殊字符,`%q` 会加反斜杠或单引号,tmux 可能不正确解析。

#### H4: respawn-pane 成功但 start-incarnation.sh 内部 127 → `claude` 命令找不到

**验证方法**:
```bash
# 查 pane-exit.jsonl 中 builder 的退出记录
grep builder ~/.solar/harness/logs/pane-exit.jsonl | tail -5
```

**关键点**: `start-incarnation.sh` 第87行 `CLAUDE_CMD="claude --dangerously-skip-permissions"`,如果 PATH 中没有 `claude`,会 exit 127。watchdog 重启 pane 后 tmux 不继承完整用户 env → `claude` 找不到。

**最可能真因**: **H4 + H2 组合**。Watchdog 后台进程 env 不完整,`start-incarnation.sh` 内部执行 `claude` 时 PATH 不含 claude 所在目录。

### 验证优先级

1. 先查 pane-exit.jsonl (H4 证据,最快)
2. 再查 watchdog 日志中的实际 respawn 命令 (H3 证据)
3. 最后用 `bash -x` 跑一次看全过程 (H1 证据)

## 变更文件

### 1. `~/.solar/harness/coordinator-watchdog.sh` — D2 修复

**改动 1: respawn-pane 命令用绝对路径 + 环境注入**

当前 (第284行):
```bash
tmux respawn-pane -k -t "$SESSION_NAME:0.$pidx" "bash ${_esc_h}/start-incarnation.sh $persona ${_esc_w}"
```

修改为:
```bash
# 用绝对路径 + 注入用户 PATH,避免 tmux env 不完整导致 bash/claude 找不到
local _user_path
_user_path=$(PATH="/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin:$HOME/.local/bin:$HOME/.npm-global/bin:${PATH}" echo "$PATH")
tmux respawn-pane -k -t "$SESSION_NAME:0.$pidx" "env PATH='${_user_path}' HOME='${HOME}' /opt/homebrew/bin/bash ${_esc_h}/start-incarnation.sh $persona ${_esc_w}"
```

核心思路: 给 respawn 命令显式注入完整 PATH + 用绝对路径调 bash,消除 tmux env 不一致。

**改动 2: respawn 后验证 pane 活性 (不靠重试,靠诊断)**

在 respawn 后 5 秒,检查 pane 是否又死了:
```bash
sleep 5
local check_cmd
check_cmd=$(tmux list-panes -t "$SESSION_NAME" -F '#{pane_index} #{pane_dead} #{pane_current_command}' 2>/dev/null | awk -v pi="$pidx" '$1==pi{print $2,$3}')
if [[ "$check_cmd" =~ ^1 ]]; then
  # pane 又死了,记录详细诊断到 watchdog.log
  err "Pane $pidx respawn 后立即死亡! diag: $check_cmd"
  # 写 pane-exit 诊断
  echo "{\"ts\":\"$(date -u +%Y-%m-%dT%H:%M:%SZ)\",\"event\":\"respawn_failed\",\"pane\":\"$pidx\",\"persona\":\"$persona\",\"diag\":\"$check_cmd\"}" >> "$HARNESS_DIR/logs/pane-exit.jsonl"
fi
```

这不是"重试绕过",而是**诊断+告警**,帮助定位根因。

**改动 3: start-incarnation.sh 调用改用绝对路径 bash**

当前 `respawn-pane` 命令中 `bash` 可能是 PATH 查找的结果。改为:
```bash
RESTART_BASH="/opt/homebrew/bin/bash"
[[ -x "$RESTART_BASH" ]] || RESTART_BASH="/bin/bash"
```

### 2. `~/.solar/harness/start-incarnation.sh` — D2 兼容性加固

**改动: 顶部加 PATH 保底**

在 `set -eu` 之后 (第10行后) 加:
```bash
# PATH 保底: watchdog respawn 后 tmux env 可能不完整
for _p in /opt/homebrew/bin /usr/local/bin "$HOME/.local/bin" "$HOME/.npm-global/bin"; do
  [[ -d "$_p" ]] && case ":${PATH}:" in *":$_p:"*) ;; *) export PATH="$_p:${PATH}" ;; esac
done
```

这确保 `claude`、`node`、`git` 等命令在任何启动环境下都能找到。

## 技术方案

### 核心修复思路

status 127 = command not found。最可能的链条:
```
watchdog 后台进程 (forked from `start`) 
  → PATH 不含 claude/node 路径
  → respawn-pane 启动 start-incarnation.sh
  → start-incarnation.sh 内部 `claude --dangerously-skip-permissions` 
  → bash 找不到 claude → exit 127
  → pane 死亡
```

**修复**: 两层防御:
1. **watchdog 层**: respawn 命令显式注入完整 PATH + 用绝对路径 bash
2. **start-incarnation.sh 层**: 顶部 PATH 保底,确保 claude/node/git 可被找到

### 测试设计

测试脚本不直接在 builder pane 跑 (自杀问题),而是:
- 脚本在**独立 tmux session** (`test-watchdog`) 中运行
- 通过 `tmux send-keys` 发命令到 `solar-harness:0.1` (builder pane)
- 监控 builder pane 的 claude PID 变化
- 5 次循环验证
