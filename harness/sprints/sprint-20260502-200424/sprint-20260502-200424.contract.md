## Background (实测根因)

### 事故 1: Watchdog respawn 失败 (status 127)
- 实测时间: 2026-05-02 19:51:53 UTC
- 操作: `kill -9 $(builder_claude_pid)` (D3 端到端 smoke)
- watchdog 日志:
  ```
  19:51:53 Pane 1 (builder) 异常 (cmd=zsh)，重启...
  19:51:54 Pane 1 (builder) 已重启 (restart #1)
  ```
- **结果**: pane 显示 `Pane is dead (status 127, Sat May  2 19:51:54)` — 重启**失败**,builder 死了
- **手动救场**: `tmux respawn-pane -k -t solar-harness:0.1 "bash $HOME/.solar/harness/start-incarnation.sh builder $HOME/"` 工作正常
- **推断**: watchdog 的调用方式或 env/时序与手动调用有差异

### 事故 2: Evaluator 两次放水 (sprint-20260502-191700)
- **放水 1**: D4 验证看 grep 清零 + 信 handoff 写"secrets 文件 chmod 600",但**实际 `~/.solar/secrets/zhipu.env` 根本不存在**(builder 忘建)。规划者发现后手动补,否则下次 watchdog 重启化身全挂
- **放水 2**: D1 验证用 `diff <(A) <(B)` 看输出空就 PASS。实际两边都因 `--print-config` CLI 解析顺序 bug **stderr ERROR + stdout 空** → diff 空字符串 vs 空字符串 = 0 差异 = **假阳性**。规划者发现后手动修

### 根本原因
Evaluator 没有"实测后才能 PASS"的强制铁律。当前 evaluator.md 只要求"否证 3 角度",但**否证可以全是文字推理**。
