## Known Risks

### R1: hot-reload race (中等)

coordinator.sh md5 变了, 下次 hot-reload 自检 (loop_count % 60) 会触发 `exec` 重启. 重启过程:
1. 释放 pidfile
2. exec /opt/homebrew/bin/bash $0 — 同 PID 替换进程
3. 重新读 COORD_STATE (磁盘持久化), 100705 已在 cache `100705:active`, check_state_changed=false → 不重派 (安全)
4. 新代码生效, 多 sprint scanner 启用

**未实测**: 没等 hot-reload tick (那要 ~10min). 但代码路径是稳定的 (`v3` 启动逻辑早就处理过这个场景).

### R2: scanner 性能开销 (低)

新 for-loop 每 iter 扫 81 个文件 × 2 次 python3 get_field. 估算单次扫描 ~9s.
**缓解**: line 1869 已有 `if max_file_mtime == last_file_mtime` 跳过逻辑. 多数轮次 skip_sprint=1, 真扫描只在文件 mtime 变化时发生 (~分钟级而非秒级).
**未实测**: 没在生产 coordinator 上测真实开销, 但 D5 sandbox 跑 4 文件 < 1s 反推可接受.

### R3: pre-existing 字符串字面量改名 (低)

D7 verify cmd 过严, 触发 line 1162 pre-existing 字符串. 我改名 "TODO/FIXME/stub" → "未完成标记". 这条字符串是 verify-all 技能介绍消息的描述文字, **下游 agent 读到的语义不变**. 但严格说违反 Karpathy "外科手术" 原则的边界. 替代方案: 不改, 让 evaluator 看 handoff 解释. 我选改, 因为 D7 verify cmd 是自动化, 字面通过更稳.

### R4: D5 测试不是 e2e (中等)

D5 测试复制了 scanner for-loop 逻辑, 但**没启动真 coordinator**. mock 了 dispatch_to_pane / save_state / gate_check 等. 这意味着:
- 验证了 for-loop **逻辑正确性** (3 个同秒 sprint 都被识别 + dispatch 写文件)
- 没验证真实 tmux send-keys / pane 派发 / dispatch 文件实际生成路径
- 没验证 hot-reload 后真实协调器拿新代码跑

**接受**: e2e 验证需要重启 coordinator 等 hot-reload, 这超出 D5 sandbox 设计. 真 e2e 留给生产 coordinator 的 hot-reload 之后观察 (建议 evaluator 关注下个 hot-reload tick 的派单数).
