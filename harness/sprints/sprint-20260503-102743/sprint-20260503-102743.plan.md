**新代码** (多 sprint for-loop):
```bash
if [[ "$skip_sprint" -eq 0 ]]; then
  # Sprint 20260503-102743 D1: 多 sprint scanner
  # 旧: get_latest_sprint_file 只返 mtime 最大 1 个, 同一秒多个翻 active 时其余永远孤儿
  # 新: for-loop 扫所有 sprint-*.status.json 逐个 check_state_changed
  for sf in "$SPRINTS_DIR"/sprint-*.status.json; do
    [[ -f "$sf" ]] || continue

    local sid st
    sid=$(get_field "$sf" "id")
    st=$(get_field "$sf" "status")

    # D2: 跳过损坏 status.json (缺 id 或 JSON 解析失败)
    # 复用 get_latest_sprint_file 的 _corrupted_logged 去重日志
    if [[ -z "$sid" ]] || [[ -z "$st" ]]; then
      if [[ -z "${_corrupted_logged[$sf]:-}" ]]; then
        log "${Y}⚠ scanner: corrupted status.json skipped: $sf (sid=[$sid] st=[$st])${N}"
        _corrupted_logged[$sf]=1
      fi
      continue
    fi

    local current_state="${sid}:${st}"

    if check_state_changed "$sid" "$st"; then
      local cur_round
      cur_round=$(get_field "$sf" "round")
      log "状态变化检测: ${sid} ${st} (round ${cur_round:-?})"
      emit_event "$sid" "state_changed" "coordinator" "{\"sid\":\"${sid}\",\"to\":\"${current_state}\"}"
      if ! gate_check "$sid" "$st"; then
        log "${Y}门禁未通过 (${sid}), 状态已回滚${N}"
        continue
      fi
      auto_checkpoint "$sid" "$st"
      save_state "$current_state"

      case "$st" in
        active)            handle_active "$sid" "$sf" ;;
        planning)          handle_planning "$sid" "$sf" ;;
        approved)          handle_approved "$sid" "$sf" ;;
        reviewing|ready_for_review) handle_reviewing "$sid" "$sf" ;;
        failed_review)     handle_failed_review "$sid" "$sf" ;;
        needs_human_review) handle_needs_human "$sid" "$sf" ;;
        passed|done|eval_pass) handle_passed "$sid" "$sf" ;;
        failed)            log "${R}Sprint ${sid} 最终失败,需要规划者介入${N}" ;;
        drafting)          log "Sprint ${sid} 草稿中,等待规划者完成 Done 定义..." ;;
      esac
    fi
  done
fi
```

## 风险点 (≥3)

### R1: hot-reload race — 改 coordinator.sh 触发 exec restart 中断 100705

**现象**: coordinator.sh md5 每 60 iter (~10 min) 自检一次. 我改完后, 下一次 hot-reload tick
会 exec /opt/homebrew/bin/bash $0, 100705 的 builder pane 1 可能正在 plan_mode 关键时刻.

**缓解**:
- exec restart 不影响 builder pane 1 (那是另一个 tmux pane 跑 claude, 不是 coordinator 子进程)
- COORD_STATE 是磁盘 checkpoint, exec 后从盘读回, 100705 不会重派
- 实测: 等 hot-reload 触发 (或手动等), 看 100705 status

### R2: scanner 性能回退 — 每 iter 扫 83 个文件 × python3 json

**估算**: get_field 调 python3 启动 ~50ms (cold) + JSON parse ~5ms ≈ 55ms/file. 83 文件 × 2 调用
(id+status) × 55ms = ~9.1s 单轮. 协调器 10s sleep + 9s scan = ~20s 周期 (旧的 ~10s 周期翻倍).

**缓解**:
- 当前 fallback 已有 `if [[ "$max_file_mtime" == "$last_file_mtime" ]]` mtime 跳过逻辑 — 多数轮次直接 skip_sprint=1
- 真正 scan 只在 file mtime 变化时才发生, 不是每 10s
- 如果性能问题严重, 后续 sprint 可优化 get_field 用单次 python3 batch parse

**接受**: scan 阶段从 ~12ms (单文件 stat) 涨到 ~9s (83 文件 × json), 但 mtime 优化让大多数轮次 skip,
等价开销 << 1s/min 平均

### R3: 改主循环可能遗漏其它路径 — 例如 init 路径仍依赖 get_latest_sprint_file

**确认**:
- line 1808 `init_sf=$(get_latest_sprint_file)` — init 路径,只取最新 1 个填 COORD_STATE 兜底

- 我**不删** get_latest_sprint_file 函数, 它仍服务 init 路径

**风险残留**: 如果有未发现的第 3 处调用 get_latest_sprint_file, 行为不变 (函数没改). 安全.
