## Background

当前 `solar-harness` 存在一组会相互放大的问题:

1. `wake` / tmux 重建链路调用需要人工按 Enter 的 `pane-launcher.sh` → "看起来恢复了 pane,实际 Claude 没起"
2. 启动链分叉:
   - `solar-harness.sh` 走 `pane-launcher.sh`
   - `coordinator-watchdog.sh` 走 `start-incarnation.sh`
   - 两条链对 planner/builder/evaluator 的模型/工具权限/代理行为不一致
3. `builder` 的 worktree 隔离只在初始启动生效,watchdog 重启后退回原目录,破坏隔离假设
4. Zhipu token 明文在脚本里 (实测 `model-config.sh` 含 `ZHIPU_AUTH_TOKEN="cee4edb0bb554b709623d2c2d63c4065.q0unUsiCytMqBo07"`),严重泄漏风险
5. 多处 `tmux send-keys` 直接拼接未转义路径,`work_dir` 含空格时失效
6. 测试 (`test-harness-resilience.sh` 等) 与实现已脱节,回归保护失真

## ⚠️ 自我影响警告 (规划者注释,builder 必读)

本 sprint **会影响正在运行的 harness**:
- D4 移除明文 token → 当前 builder/evaluator 化身依赖此 token,删除后下次 watchdog 重启会挂掉
- 实施时 builder **必须**:
  1. 先建立 secrets 文件 (如 `~/.solar/secrets/zhipu.env`,chmod 600,不入版本控制)
  2. 改 model-config.sh 从该文件 source 读取 token (脚本里仍可用 `$ZHIPU_AUTH_TOKEN`)
  3. **再**删除明文,确保切换无缝
- D1 启动链统一 → 改完后必须验证当前正在跑的化身可以**热**切换到新启动器 (无需手动 kill/重启) — 即 watchdog 下次重启时使用新启动器仍能起来

## 建议实现方向 (不强制具体写法,但效果必须达到)

1. 抽共享配置函数 `lib/persona-config.sh` 提供 `get_persona_config <persona>` 输出 KV
2. 提供两种入口:
   - **交互入口**: 用于手工 attach 时展示 banner (允许阻塞)
   - **无交互入口**: 用于 wake / watchdog / 后台恢复 (绝对不阻塞)
3. `solar-harness.sh start/wake` 与 `coordinator-watchdog.sh` 都调用同一**无交互**启动器
4. `--print-config <persona>` 调试接口内建,方便排障
5. `model-config.sh` 只保存"变量名和读取逻辑",不保存秘密值本身;真值来源:
   - `~/.solar/secrets/zhipu.env` (chmod 600)
   - 或环境变量 `$ZHIPU_AUTH_TOKEN` (caller 注入)
6. `tmux send-keys` 路径转义: 用 `printf '%q' "$path"` 或 bash 内置引用规则
