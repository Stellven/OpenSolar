# Execution Contract: Solar PM Pane + Dual Quadrant Expansion

日期: 2026-05-05
任务名: solar-pm-pane-dual-quadrants
执行方: solar-harness
优先级: P0
来源: Codex bridge / 用户夜间委托

## 1. Objective

把 Solar-Harness 从“单四分屏执行系统”升级为“PM 入口治理 + 两个四分屏并行车间”的最小可用版本。

必须实现两个用户命令:

1. `solar-harness` 启动第一个四分屏。
2. `solar-harness 扩展` 启动第二个四分屏。

同时新增一个专门的 PM pane/persona，用于阅读用户留言、解释需求、把控优先级、验收口径和 stop rules，再交给 planner/builder/evaluator 执行。

## 2. Background

当前 Solar-Harness 已有:

- `solar-harness.sh` 作为主入口。
- `pane-launcher.sh` 按 persona 启动 pane。
- `personas/planner.md`, `personas/builder.md`, `personas/evaluator.md`, `personas/second-builder.md`。
- `coordinator.sh` 状态机、`dispatch_to_pane`、`.pane-assignments` 防覆盖。
- `codex-bridge/from-codex` 自动消费 execution contract。
- `sprints/*.status.json`, `*.plan.md`, `*.handoff.md`, `*.eval.md` 生命周期。

当前慢的根因不只是 builder 数量少，而是用户意图进入系统前缺少 Product/Requirement Gate，导致 planner 过早工程化、builder 可能干偏、evaluator 只验 Done 不验真实目标。

本 sprint 只做最小闭环，不要大重写 coordinator。

## 3. Target Topology

第一个四分屏由 `solar-harness` 启动:

```text
window 0: Product Delivery
  pane 0.0  PM              阅读用户留言，产出 product brief
  pane 0.1  Planner         根据 product brief 生成 sprint contract
  pane 0.2  Builder         实现
  pane 0.3  Evaluator       按 acceptance criteria 验收
```

第二个四分屏由 `solar-harness 扩展` 启动:

```text
window 1: Strategy Lab
  pane 1.0  Architect       架构设计、技术路线、论文应用
  pane 1.1  Lab Builder     诊断、benchmark、实验实现
  pane 1.2  Lab Evaluator   实验验收、回归验证
  pane 1.3  Observer        日志监控、Codex bridge、卡死检测
```

## 5. Forbidden Actions

- 禁止删除或破坏现有 `solar-harness` 默认启动能力。
- 禁止把第二四分屏做成必须手动复制命令才能启动。
- 禁止让两个 builder 默认共享同一个 sprint 写同一 worktree。
- 禁止把 PM pane 当 builder 使用。
- 禁止为了快速通过只改文档、不改实际启动行为。
- 禁止在没有隔离策略前开启两个 builder 同时改同一个文件范围。
- 禁止删除现有 sprint、日志、bridge processed 文件。
- 禁止 destructive git 操作，例如 `git reset --hard`、`git checkout --`。

## 6. Required Work

1. 新增 PM persona。
   - PM 必须阅读用户留言/外部请求。
   - PM 必须产出 product brief。
   - PM 必须定义 priority、lane_hint、acceptance criteria、non_goals、stop_rules。
   - PM 不直接写代码，不直接改 sprint status 到 implementation。

2. 新增 Product Brief schema/template。
   - 最少字段: `title`, `source`, `intent`, `problem`, `priority`, `lane_hint`, `acceptance`, `non_goals`, `stop_rules`, `handoff_to`。
   - schema 验证必须可用。

3. 修改 `solar-harness.sh` 默认四分屏。
   - `solar-harness` 或 `solar-harness.sh` 默认启动 window 0。
   - window 0 必须是 PM / Planner / Builder / Evaluator。
   - 启动后 coordinator/watchdog 保持可用。

4. 实现 `solar-harness 扩展`。
   - 若 session 不存在，必须给出清晰提示: 先运行 `solar-harness`。
   - 若 window 1 已存在，不能重复创建，应该 attach 或提示已存在。
   - 若 window 1 不存在，创建第二个四分屏并启动 Architect / Lab Builder / Lab Evaluator / Observer。
   - 命令也应兼容 `solar-harness.sh 扩展`。

5. 新增 layout 配置。
   - 写 `farm-layout.json`，记录两个 window、八个角色、tmux pane target。
   - 后续 coordinator/lane scheduler 可以读取该文件。

6. 最小 lane hint 支持。
   - 不要求完整 lane scheduler。
   - 但 PM product brief 必须能够表达 `lane_hint: delivery|lab|strategy`。
   - 文档说明后续 coordinator 如何基于 lane_hint 派发。

7. Watchdog/persona-config 兼容。
   - 新 persona 能通过 `pane-launcher.sh <persona>` 启动。
   - watcher 不应把新 pane 误判为异常或错误角色。

8. 测试。
   - 至少提供一个 layout smoke test。
   - 至少提供一个 product brief schema test。
   - 必须跑 `bash -n` 覆盖改动 shell 文件。

## 11. Failure Rules

遇到以下情况必须停止并上报:

- `solar-harness` 默认入口无法兼容现有用户习惯。
- `solar-harness 扩展` 无法稳定检测已有 window。
- 新 pane 需要修改 Claude 权限或模型密钥配置。
- PM pane 无法通过 `pane-launcher.sh` 正常启动。
- 测试会破坏当前正在运行的用户 session。
- 需要大规模重写 coordinator 状态机。

## 12. Night Execution Instruction

用户准备离开，本合约必须由 Solar-Harness 自动执行。不要等待用户手动复制、手动触发或手动确认。

若需要保守处理，优先实现 dry-run/mock 测试和非破坏性启动扩展；不要中断当前正在跑的 sprint。
