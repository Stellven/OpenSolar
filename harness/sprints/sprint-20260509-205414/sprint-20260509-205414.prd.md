# PRD — Solar Experience Memory Layer

**Sprint**: sprint-20260509-205414
**Source**: 昊哥 intake — "借鉴 MIA，把成功/失败 sprint 轨迹压缩成可检索工作流记忆"
**Priority**: P0
**Lane**: control-plane
**Created**: 2026-05-10T00:58:00Z

## 1. Context

Solar harness 已经跑过几十个 sprint，事件、status.history、handoff/eval/dispatch 文件躺在 `~/.solar/harness/sprints/`。这些是真实的成功/失败轨迹，但 coordinator 派单前不会去看它们。结果是同一类故障反复出现:

- **C-u 风暴**: planner/builder pane 被反复 C-u 清屏 (短期内 5+ 次)
- **错派 pane**: bypass_pm 路由偶发把 builder 任务发到 planner
- **corrupt status**: 并发写 status.json 导致字段缺失/重置
- **终态 phase 噪音**: passed/closed 状态仍触发 wake/dispatch 循环
- **队列阻塞**: builder/evaluator pane 被旧任务占满，新 sprint 长时间排队

这些都已经在过去的 events.jsonl 里有过征兆，但 coordinator 没有"经验"。MIA (Memory Intelligence Agent) 思路: 把轨迹压缩成可检索记忆，决策前 retrieve 一遍。

## 2. Problem

P1 — **Coordinator 失忆**: 每次派单是无记忆决策，依赖硬编码规则；规则覆盖不到的边角必然踩坑。
P2 — **轨迹未结构化**: events.jsonl 是流水账，handoff/eval 是自然语言；同类问题难聚类、难检索。
P3 — **教训不复用**: 即使评估 PASS 也总结过 lesson，下次仍要重学；FAIL 的 root cause 散落在各 sprint 目录。
P4 — **状态机噪音**: 终态 sprint 还会被 autopilot 触发；这本身就是经验缺失的症状。
P5 — **无反馈闭环**: coordinator 派单 → 失败 → 不会自动产生避坑记忆。

## 3. Goals

G1. **Trajectory Extraction**: 从每个 sprint 的 events/status.history/handoff/eval 抽取结构化 trajectory record。
G2. **Experience Compression**: 把 trajectory 压成结构化 experience entry: trigger condition / failure pattern / mitigation / outcome / confidence。
G3. **Searchable Index**: sqlite + Tantivy/FTS 双索引，支持 by-trigger / by-pattern / by-pane / by-role 检索，p95 < 100ms。
G4. **Retrieval API**: `solar-harness experience query <signal>` 返回 top-K 相关经验 + 建议动作。
G5. **Coordinator Pre-Dispatch Hook**: 派单前 query experience，匹配到 anti-pattern 时变更行为 (delay/route/abort)。
G6. **Backfill**: 历史所有 sprint 全量灌入 experience layer。
G7. **End-to-End Proof**: 注入合成 anti-pattern (终态 phase 重派) → coordinator 检索到记忆 → 跳过派单。

## 4. User Stories

US-1 (Coordinator): 我准备 wake planner 派 sprint X，先 `experience query "wake planner sprint terminal_phase"` → 返回 "终态 phase 不应 wake (3次出现)" → 我 abort wake。
US-2 (Builder pane): coordinator 派单前查到 "builder pane 0.2 当前持有 lease"，改派到空闲 pane。
US-3 (Evaluator): 看到失败 sprint 时 `experience query "evaluator stall handoff_missing"` → 列出过去 5 次 stall 的 mitigation。
US-4 (昊哥): 周复盘时 `experience top --window 7d` 看本周高频 anti-pattern。
US-5 (Self-evolution): 失败 sprint 自动生成新 experience entry，下次派单立即生效。

## 5. Functional Requirements

FR-1 **Trajectory Schema** — 每个 sprint 产出一份 `experience/trajectory/<sid>.json`:
```
{
  "sid": "...",
  "outcome": "passed|failed|aborted",
  "duration_sec": 1234,
  "events_compressed": [{"ts","event","by","note"}, ...],
  "phase_path": ["spec","prd_ready","planning","active","reviewing","passed"],
  "anti_patterns": ["c_u_storm","mis_dispatch","status_corruption","terminal_phase_wake","queue_block"],
  "lessons": ["...","..."]
}
```

FR-2 **Experience Schema** — 压缩后的 `experience/entries/<id>.json`:
```
{
  "id": "exp-2026-05-10-001",
  "trigger": {"role":"coordinator","action":"wake_planner","context":"phase=passed"},
  "pattern": "terminal_phase_wake",
  "evidence_sids": ["sprint-...","sprint-..."],
  "frequency": 3,
  "first_seen": "2026-05-08T...",
  "last_seen": "2026-05-09T...",
  "mitigation": "skip wake when phase in {passed,closed,aborted}",
  "confidence": 0.85,
  "severity": "low|medium|high"
}
```

FR-3 **CLI** — `solar-harness experience extract|compress|index|query|backfill|stats`。
FR-4 **Storage** — `~/.solar/harness/experience/`:
  - `trajectory/<sid>.json` (源)
  - `entries/<id>.json` (压缩条目)
  - `index.db` (sqlite + FTS5)
FR-5 **Retrieval** — `experience query <signal>` 返回 top-K (默认 5) 按 (frequency × confidence × recency) 排序。
FR-6 **Coordinator Hook** — `lib/coordinator_hooks.py: pre_dispatch(sid, action) → Decision{allow, deny, route_to, delay_sec, reason}` 接入 coordinator 主循环。
FR-7 **Backfill** — 处理 ~/.solar/harness/sprints/ 现存所有 sprint，幂等。
FR-8 **Stats UI** — `experience stats --json` 返回 top patterns / hit rate / coverage。

## 6. Acceptance Criteria

A1 **Schema Valid**: 100% 历史 sprint 能被 extractor 处理，trajectory.json 通过 schema 校验。
A2 **Compression**: 至少识别 5 个 anti-pattern (c_u_storm, mis_dispatch, status_corruption, terminal_phase_wake, queue_block)，每个 ≥1 entry。
A3 **Retrieval p95 < 100ms**: 索引建立后，query top-5 在 p95 100ms 内返回。
A4 **Hook Integration**: coordinator pre_dispatch 钩子接入；hit 时写 events.jsonl `experience_hit` 事件。
A5 **End-to-End Proof**: 注入合成 sprint phase=passed 后触发 wake → coordinator 查到 terminal_phase_wake → 不派单 → 写 abort 原因。
A6 **Backfill 全量**: 当前 `~/.solar/harness/sprints/*.status.json` 全部进 trajectory；CLI `experience stats` 报告 coverage=100%。
A7 **No Regression**: 现有 coordinator/wake/autopilot 路径在 hook 关闭时行为不变；hook 默认开启但提供 `EXPERIENCE_HOOK=0` 关闭。
A8 **Audit**: 每次 hook 决策写 `experience/decisions.jsonl` (sid + decision + matched_entries + ts)。

## 7. Non-Goals

N1. 不做 LLM-based 总结：第一阶段只做规则化抽取与计数；LLM 总结后续 sprint 再做。
N2. 不替换 coordinator 主循环：只做 pre_dispatch 钩子接入。
N3. 不动 events.jsonl 写入路径：只读。
N4. 不引入新数据库引擎：用现有 sqlite。
N5. 不做跨 sprint 因果链推理：第一阶段只做单 sprint trajectory + 跨 sprint 频率统计。

## 8. Constraints

C1. 只读 `~/.solar/harness/sprints/`；写入限定 `~/.solar/harness/experience/`。
C2. 向后兼容: hook 失败 fail-open，coordinator 继续原行为。
C3. backfill 必须幂等; 重复跑不会复制 entry。
C4. 索引重建时间 < 30s (sprints<200 时)。
C5. Hook 决策延迟 p99 < 50ms (不能拖慢主循环)。
C6. 不允许 plaintext secret 进 trajectory/entry。
C7. 终态 sprint 一次抽取后 marked frozen，不再重复 extract。

## 9. Risks

R1 (中) anti-pattern 误识别 → 错误 deny 派单，阻塞工作。缓解: confidence < 0.7 仅 advisory，不 deny；hook 默认 advisory 模式，high-confidence 才 enforce。
R2 (中) backfill 把 secret 抓进 trajectory。缓解: extractor 跑 secret-strip filter；entry 写入前 schema 校验拒绝可疑字段。
R3 (低) sqlite FTS5 在大量 sprint 时慢。缓解: 索引分片 + 增量 update；监控 query p95。
R4 (低) coordinator hook 引入 race。缓解: hook 内部加 timeout (50ms)，超时 fail-open。
R5 (中) extractor 跟 events 模式漂移。缓解: extractor 按 schema_version 标记；版本不匹配回退到原始 fields。

## 10. Open Questions

OQ-1. anti-pattern 列表谁维护？建议: 内置 5 个种子，新模式由失败 sprint 自动 propose + 人工 confirm。
OQ-2. experience entry 是否对外可编辑？建议: 仅 CLI promote/demote/delete；不直接编辑 json。
OQ-3. retention 策略？建议: trajectory 永久保留，entry 半年内重新评分一次，confidence < 0.4 自动归档。
OQ-4. hook decision 是否可被 user override？建议: 提供 `experience override <sid> --reason ...` 写 audit 后允许通过。
OQ-5. 是否要把 entry 同步进 cortex_sources？建议: high-confidence (≥0.85) entry 自动同步，标 source=experience-layer。
