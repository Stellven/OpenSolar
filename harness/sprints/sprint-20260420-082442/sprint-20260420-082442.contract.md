## 需求

协调器启动三连锁修复: (1) 僵尸 pidfile 自愈 — 启动时用 kill -0 验活,死了清锁再启动; (2) pidfile 所有权统一 — 只由 coordinator 自己管(外部 solar-harness 启动脚本不准预写 pidfile); (3) last_state 启动逻辑修复 — 协调器启动时不把当前 active 保存为 last_state,保证第一次轮询就触发 handle_active 派发; 顺带修 (4) solar-harness update-contract 的 'local: can only be used in a function' bug; 全程不回退 sprint-20260417-213037 和 sprint-20260419-223020 的修复

## Done 定义

### D1 · 僵尸 pidfile 自愈（根因 #1）
- [ ] `coordinator.sh` 启动自检（当前 1131-1134 行附近）：若 `.coordinator.pid` 存在 → `kill -0` 验活 → **活**才退出，**死**则 `rm -f pidfile` 继续启动（不再盲目信任锁文件）
- [ ] 清锁时写一条日志 `stale pidfile detected (PID=X dead), removed and proceeding`，便于事后审计
- [ ] `solar-harness` 外部启动逻辑（当前 ps aux 看到的 `echo $! > pidfile` 两处）同步去掉"信任死 PID"的分支——统一以 `kill -0` 验活

### D2 · pidfile 所有权统一（根因 #2）
- [ ] **`solar-harness` 启动 coordinator 时禁止外部预写 pidfile**（即删掉 `echo $! > "$HARNESS_DIR/.coordinator.pid"` 两处），只做 `nohup ... &` 后台启动
- [ ] `coordinator.sh` 启动时**必写** `echo $$ > "$pidfile"`（已有，保留）+ `trap "rm -f '$pidfile'" EXIT`（已有，保留）
- [ ] 启动流程文档化：`DISPATCH-PROTOCOL.md` 追加《PID 锁所有权约定》一节，明确"pidfile 只由 coordinator 自己读/写/清"的不变量
- [ ] 提供 `solar-harness coord-status` 子命令返回 `{running:true/false, pid:N, uptime_s:N, stale_lock:bool}`，方便一键自诊

### D3 · last_state 启动逻辑修复（根因 #3）
- [ ] `coordinator.sh` 启动时**不得**把当前 sprint 的 `active/reviewing/failed_review` 等**中间态**直接保存为 last_state
- [ ] 启动初始化规则：`load_last_state` 返回空（`""`）或 `sprint-NONE:init`，使第一次主循环轮询必然进入 `状态变化检测` 分支并派发
- [ ] 仅 `drafting/passed/done/failed` 这类**终态/静态**允许在启动时保存为 last_state（避免启动即重复派发已完成的 Sprint）
- [ ] 启动日志必须打 `初始 last_state = <值> | 当前 sprint = <sid>:<status> | 首轮将触发派发? = yes/no`，便于调试

### D4 · update-contract `local` bug（根因 #4）
- [ ] 定位 `~/.solar/bin/solar-harness` 第 ~537 行的 `local: can only be used in a function` 错误
- [ ] 修复方法二选一：(a) 把相关代码块包进 `update_contract() { ... }` 函数；(b) 删除 `local` 改为普通变量
- [ ] 修复后 `~/.solar/bin/solar-harness update-contract <sid> done "- [ ] 条件X"` 能正确追加 Done 条件且不报错
