位置: `~/.solar/harness/codex-bridge.sh`

- [ ] stderr 全量重定向到 `~/.solar/harness/logs/codex-bridge.log`

### D3 · 调用分级策略（省额度的命门）
文档: `~/.solar/harness/docs/codex-usage-policy.md`
- [ ] S 级（**必调**，单次上限 4000 out-tokens）：审判官同一 Sprint 连续 FAIL ≥ 2 轮的根因梳理 / 架构方案 A vs B 对比决策 / 复杂算法设计（如并发协议、状态机）
- [ ] A 级（**可调**，上限 2000 out-tokens）：跨模块逻辑梳理 / 疑难 bug 根因（单轮排查未果）/ 规划者写大合约前的 Done 定义 sanity check
- [ ] B 级（**禁调**，策略层直接拒绝）：CRUD / 格式化 / 命令执行 / 单元测试生成 / 简单脚本 / 已有知识库答案的问题
- [ ] 每级给出 ≥ 3 条具体触发示例 + ≥ 3 条反例（会被误触发但不该调的场景）
- [ ] 调用前必须由 call_codex 做 tier 校验；tier=B 立即返回 `REJECTED_BY_POLICY` 不计入预算

### D4 · coordinator.sh 集成
- [ ] 新增 `call_codex(persona_from, tier, prompt, context_files...)` 函数
- [ ] 流程: 预算检查（D5）→ tier 校验（D3）→ 生成 `req_id="$(date +%Y%m%d-%H%M%S)-$$"` → 写 inbox/$req_id.req.md（含 YAML frontmatter: `tier / from / deadline_s / context_files`）→ 轮询 outbox/$req_id.res.md（默认 60s 超时，可由 deadline_s 覆盖）
- [ ] 超时或非 0 exit_code：记 ledger、熔断（同 persona_from + 同 tier 5 分钟冷却，冷却期内直接 `CIRCUIT_BREAKER_OPEN`）
- [ ] `handle_failed_review()` 在同一 Sprint 连续 FAIL ≥ 2 轮时**自动以 S 级**调用 codex 做根因梳理，返回结果以 `## Codex 根因分析` 段附加到下轮 handoff.md

### D5 · 预算熔断
- [ ] `~/.solar/harness/codex-budget.json`: `{daily_token_limit:20000, daily_call_limit:30, used_tokens_today:0, used_calls_today:0, reset_at:"YYYY-MM-DD", hard_stop:true}`
- [ ] `codex-budget.sh` 提供: `check` (返回 0/1)/ `consume <tokens>` / `reset` / `status`
- [ ] 每次 call_codex 前先 check；成功返回后按 ledger 的 tokens_out consume
- [ ] 日界重置：bridge 启动时比对 `reset_at` 日期，不同则重置计数
- [ ] 超限时：call_codex 立即返回 `BUDGET_EXCEEDED`（不进 inbox）+ 触发 `osascript-notify.sh` 发 macOS 通知给昊哥（声音 Purr）
- [ ] bridge 启动 banner 必须显示当前余量（`used_calls/daily_call_limit · used_tokens/daily_token_limit`）

## 约束

- **C1 · 不破坏现有流水线**: test-dispatch.sh 13 项必须全绿；Sprint 20260417-213037 的修复（send-keys 两步 / mtime 文件级 / pane 忙检测 / stderr 捕获）不得回退

- **C3 · B 级禁调铁律**: 任何 CRUD / 跑命令 / 简单代码生成若被误派给 codex，必须在策略层（D3）拒绝；审判官的"常规代码质量评估"仍由 GLM-5.1 做，不占 codex 额度

- **C5 · 沙箱可读**: codex 默认 `--sandbox read-only`，禁止 codex 直接修改仓库文件；任何代码落地仍由建设者执行
