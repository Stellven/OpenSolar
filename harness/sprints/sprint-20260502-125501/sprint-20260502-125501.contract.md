## Requirements

协调器 round-N+1 dispatch 漏洞修复 (B4): 当 sprint round-N FAIL → planner 修订合约 → status 转 active 时, 协调器只改 status.json 没给 builder pane 注入 round-N+1 dispatch prompt 和 send-keys 操作, 导致 builder 卡在 plan_mode 死等用户按键 (实测 sprint-20260502-083522 round-2 builder 卡 1h29min 烧 6.1k tokens 没动). 必须修 ~/.solar/harness/coordinator.sh 加 round-N+1 切换的 dispatch 路径: (1) status active 后立即重写 dispatch.md 注入 round-N+1 prompt; (2) tmux send-keys 给 builder pane 把 plan mode 切到 edit mode (shift+tab); (3) 给 builder 发明确指令'继续 round-N+1 按合约修订项重做'. 验证方式: 故意制造一个 round-1 FAIL → 观察建设者 pane 是否 5 分钟内自动收到 round-2 prompt 并继续工作, 不再死等用户按键.

## Definition of Done

- [ ] D1: 复现死锁 — 故意创建一个最小 sprint, builder round-1 交错 handoff, evaluator FAIL, planner contract_revised, 观察并记录现场 builder pane 是否真卡在 plan_mode (要看到 plan mode on 提示 + tmux capture 末尾无新输出 ≥ 2 分钟). 死锁现场 capture 写入报告.
- [ ] D2: coordinator.sh 加 round-N+1 dispatch 路径 — 在 status 由 reviewing → active 转换的 handler 里 (handle_contract_revised 或类似), 必须做三件事: (a) 重写 dispatch.md 注入 round-N+1 prompt (含合约修订项摘要 + builder 角色重申); (b) tmux send-keys 给 builder pane 一个 'Escape' 退出 plan mode + '/clear' 释放 token + 然后注入新 prompt; (c) 写一行 round_dispatched 事件到 events.jsonl.
- [ ] D3: 模拟测试 D1 复现的死锁场景 — 用修后的 coordinator.sh 重跑 D1, 验证: builder pane 5 分钟内自动收到 round-2 prompt + 退出 plan mode + 继续工作. 不再需要监护人手按 shift+tab.
- [ ] D4: 边界情况处理 — (a) builder pane 已经在 edit mode 不该再 send Escape; (b) builder 已经在跑 round-N+1 不该重复 dispatch; (c) builder pane 不存在或 tmux session 死了的优雅 fallback.
- [ ] D5: 测试覆盖 — 至少 3 个 bash test case 覆盖 D2/D3/D4, 加到 ~/.solar/harness/tests/ 或类似目录, 跑测试全绿.
- [ ] D6: 文档更新 — 在 ~/.solar/harness/README.md 或 docs/coordinator.md 加一段 'round-N+1 dispatch flow', 说明事件链 + send-keys 时机 + 兜底.
- [ ] D7: 报告归档 — ~/.solar/reports/2026-05-02-coordinator-dispatch-fix.md, 不放 /tmp.
